### 2-1. 단위테스트 목적 및 범위

#### 목적

REQ-ODD-001에서 신규 구현·수정된 7개 GAP 해소 모듈(`covimail` 신규 7개 파일 + 수정 4개 파일)의 개별 클래스·메서드 단위 기능 정확성을 JUnit 5 + Mockito 기반 격리 환경에서 검증한다. 본 결과서는 UT-ODD-001-01~28 시나리오별 수행 결과, 결함 내용, 조치 이력, AI 진단 의견을 PM에게 공식 보고하는 최종 산출물이다. 특히 Exchange/LDAP Exponential Backoff, MSAL4J OAuth2 토큰 Cache-Aside, Redis 분산 락 3종 임계 구역, 비동기 조직도 4상태 전환의 핵심 경계 케이스를 격리 검증한 결과를 상세히 기록한다.

#### 수행 범위

| 범위 구분 | 내용 |
|---------|------|
| **대상 모듈** | `covimail` — 신규 7개 파일 + 수정 4개 파일 (총 11개) |
| **재활용 모듈 (제외)** | `smart4j_msapi`, `smarts4j_coviexchange` — 변경 없음 |
| **테스트 프레임워크** | JUnit 5 (Jupiter) + Mockito 4.x + WireMock(포트:9091) + Embedded Redis |
| **수행 UT 수** | UT-ODD-001-01 ~ UT-ODD-001-28 (총 28건) |
| **관련 GAP** | G-01(세션 재연결), G-02(OAuth2 인증), G-03(배포그룹 CRUD), G-04(주소록 동기화), G-05(Redis 분산 락), G-06(비동기 처리), G-07(배포그룹 폴더) |

### 2-5. AI 의견

> **산출 근거**: 설계 상세 결과서(`300AI_D_040`), 단위테스트 계획서(`400AI_E_100`), 수행 보고서(`400AI_E_110`) 전 문서 재검증 기반 AI 기술 진단 의견

#### 전체 구조적 의견

1. **Exponential Backoff 타이밍 검증(G-01) — CI 환경 의존도 주의**: `Thread.sleep()` 기반 지연 검증은 CI 서버 부하에 따라 1,000ms/2,000ms/4,000ms 기준이 불안정할 수 있다. `Clock.fixed()` 또는 `FakeTicker` 패턴을 UT-02, 03 수행 전에 선택해야 한다. 결정 지연 시 빌드 불안정이 반복될 위험이 높다.

2. **OAuth2 Cache-Aside 동시성(G-02) — 이중 호출 방지 검증 필수**: TS-ODD-S02 멀티 스레드 시나리오에서 동일 `domainId` 2개 스레드가 동시에 캐시 미스를 만나는 경우 MSAL4J 이중 호출이 발생할 수 있다. `synchronized` 블록 또는 Redis `SETNX` 분산 락 적용 여부를 UT-06 수행 전 코드 리뷰로 반드시 확인해야 한다.

3. **CoviMSAPI 체이닝 Mock 복잡도(G-03) — 구조 단순화 권고**: `CoviMSAPI.org()` → `OrgSvcProxy` → `addGroup()` 체이닝 Mock 구성은 Mockito 일반 패턴으로 처리가 복잡하다. 인터페이스 추상화 레벨을 `CoviMSAPI.addGroup(groupDTO)` 수준으로 단일화하거나, Light Stub 클래스를 직접 작성하는 것이 테스트 유지보수성 측면에서 유리하다.

4. **4폴더 XML 완전성(G-04, G-07) — null 데이터 처리 정책 명시 필요**: `selectContactList()` 또는 `selectContactGroup()` 결과가 빈 목록(`[]`)인 경우 해당 `<GroupName>` 폴더를 XML에 포함할지 제외할지 설계 문서에 명시되어 있지 않다. UT-15 수행 전 비즈니스 담당자와 협의하여 코드 주석 및 assert에 정책을 반영해야 한다.

5. **Redis 분산 락 `finally` 해제 보장(G-05) — 보안·안정성 핵심**: `lock:outlook:distgroup:{groupCode}` Lock이 `try { ... } finally { delete(lockKey); }` 구조로 항상 해제되는지 UT-11~12, 16 수행 전 코드 리뷰 필수. `finally` 블록 누락 시 TTL 만료 전까지 해당 그룹에 대한 모든 요청이 차단되는 운영 장애로 이어질 수 있다.

6. **`@Async` 격리 전략 사전 확정 필요(G-06)**: UT-22~24는 `getOrgChartAsync()` 메서드가 `outlookAsyncExecutor` 스레드에서 실행되므로 JUnit main 스레드 assert 타이밍이 불일치한다. `@TestPropertySource("spring.async.enabled=false")` 비활성화 방식과 `CompletableFuture.get()` 동기 대기 방식 중 하나를 계획서 확정 전에 결정해야 한다.

7. **MSAL4J ↔ EWS 라이브러리 클래스패스 충돌(R-03) — 빌드 우선 검증**: `msal4j` + `ews-java-api-3.1.5.jar` 동시 의존성은 클래스패스 충돌 위험이 있다. 모든 UT 수행 전 `mvn dependency:tree | grep -E "msal|ews"` 명령으로 충돌 여부를 먼저 확인하고, 충돌 발생 시 Shade Plugin 또는 `<exclusion>` 처리 후 진행해야 한다.
