### 2-5. AI 의견

> **산출 근거**: 설계 상세 결과서(`300AI_D_040`), 단위테스트 계획서(`400AI_E_100`), 수행 보고서(`400AI_E_110`) 전 문서 재검증 기반 AI 기술 진단 의견

#### 전체 구조적 의견

1. **OAuth2 인증 전환(GAP-02) — 가장 높은 기술 리스크**: `TokenRefreshService` + `EWSCommon` 연동은 `msAccessToken` Redis Cache-Aside 패턴의 동시성 안전성(TC-02 멀티 스레드 캐시 미스 시 MSAL4J 중복 호출 방지)이 핵심 검증 포인트다. `synchronized` 블록 또는 Redis `SETNX`(Set If Not Exists) 분산 락 적용 여부를 TC-01~04 수행 전에 코드 리뷰로 반드시 확인해야 한다.

2. **청크 업로드 경계값(GAP-05) — Content-Range 계산 오류 주의**: `BigAttachHelper.uploadChunk()`의 Content-Range 마지막 청크 계산 시 `fileSize % chunkSize == 0`인 정확히 나누어 떨어지는 경계 케이스와 나머지가 있는 경우 모두 TC-05에서 검증해야 한다. 경계값 누락 시 Graph API 415/416 오류로 이어질 수 있다.

3. **Redis SCAN 커맨드 전환(GAP-09) — 운영 위험 선제 차단**: `SubscriptionRenewalScheduler`의 `keys("ewsSubscription:*")` 패턴은 Redis 단일 스레드 블로킹 위험이 있다. TC-07, 08 수행 전 `SCAN` 커맨드 대체 여부를 코드 리뷰로 선행 확인하고, Embedded Redis 환경에서는 소규모 데이터(10건 내외)로 검증하는 것을 권장한다.

4. **SvcImplFactory evict() 동시성(GAP-10) — Race Condition 잠재**: `evict()` 직후 동시에 `getMailService()`가 호출되는 Race Condition 시나리오(TS-UNIT-S05-02 멀티 스레드 검증)는 단위 테스트 단계에서는 단일 스레드로 진행하되, 통합 테스트 단계에서 반드시 멀티 스레드 시나리오를 추가 검증해야 한다.

5. **isHidden null 처리 정책(GAP-04) — 명시적 정책 정의 필요**: `GWFolderDTO.isHidden=null`인 폴더의 처리 방침(포함 vs 제외)이 설계 문서에 명시되어 있지 않다. TC-11 수행 전 비즈니스 담당자와 협의하여 null 처리 정책을 코드 주석 및 테스트 assert에 명시적으로 반영할 것을 권고한다.

6. **GAP-03 보류 리스크**: `CalendarSvcImpl` 반복 일정 변경 분기 로직의 원본 분석서(`200AI_C_050`) 미확보 상태가 지속될 경우, 본 GAP은 단위 테스트뿐 아니라 인수 테스트 AC-WML-006까지 전체 블로킹된다. PM 차원의 조속한 원본 분석서 확보 요청이 필요하다.
