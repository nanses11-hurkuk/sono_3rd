### 2-7. 단위테스트 데이터 전략

#### 전략 개요

단위 테스트는 실 외부 서비스(Azure AD, Exchange, Graph API, Redis) 연결 없이 **완전 격리된 Mock 환경**에서 실행한다. 테스트 데이터는 코드 내 `@BeforeEach` 또는 별도 Fixture 클래스로 관리한다.

#### 데이터 준비 상세

| 데이터 유형 | 준비 방법 | 적용 TC |
|----------|---------|---------|
| **AccessToken (유효/만료)** | `MSAccessTokenDTO` Mock 객체 생성 (`accessToken="mock-token-xxx"`, TTL=3500초) | TC-01, 02, 03, 04 |
| **Redis 캐시 히트 데이터** | Embedded Redis에 `msAccessToken:{domainId}` 키 사전 주입 | TC-01 |
| **Redis 캐시 미스 상태** | Embedded Redis 초기 상태 유지 (키 없음) | TC-02, 03 |
| **MSAL4J 응답 스텁** | WireMock: `POST /oauth2/v2.0/token` → `{"access_token":"...", "expires_in":3600}` | TC-02 |
| **MSAL4J 타임아웃 스텁** | WireMock: 응답 지연 11,000ms 설정 → 10초 타임아웃 유발 | TC-03 |
| **EWS Bearer 헤더 검증 데이터** | `ExchangeConfigDTO.authType = "OAuth2"` Mock 주입 | TC-04 |
| **청크 업로드 세션 데이터** | `GWUploadSessionDTO` (draftMessageId, fileName, fileSize=30MB, contentType) Mock | TC-05, 06 |
| **Graph API 업로드 응답 스텁** | WireMock: `PUT /uploadSession/{id}` → `206 Partial Content`, `500 Internal Server Error` | TC-05, 06 |
| **EwsSubscriptionInfo (만료 임박)** | `expirationDateTime = now + 30분` (만료 임박 기준 60분 이내) | TC-07 |
| **EwsSubscriptionInfo (만료 여유)** | `expirationDateTime = now + 120분` (만료 여유 기준 초과) | TC-08 |

#### 민감 데이터 처리 원칙

- 실제 `oauthClientSecret`, `ewsPwd`, `adPwd` 등 민감 정보는 테스트 코드에 절대 하드코딩 금지
- Mock 객체에는 `"mock-secret-***"` 형태의 대체값 사용
- 테스트 완료 후 Embedded Redis 데이터 자동 초기화 (`@AfterEach` 활용)

### 2-10. AI 기반 단위테스트 실행 전략 계획

#### 시나리오 합성 (Scenario Synthesis)

- **AI 도구 활용**: Claude 기반으로 각 메서드의 입력/출력 스펙을 분석하여 Happy Path, Edge Case, Negative Case를 자동 합성
- **합성 대상**: TC-WML-UNIT-01~12 전 케이스에 대해 Given-When-Then 형식으로 시나리오 초안 작성
- **검토 프로세스**: AI 생성 시나리오 → 이동준(담당자) 검토 → PM 최종 확인

#### 스크립트 자율 생성 (Script Auto-Generation)

- JUnit 5 기반 테스트 클래스 껍데기(Scaffold) AI 자동 생성 후 개발자 수작업 보완
- Mockito `@Mock`, `@InjectMocks`, `@Spy` 어노테이션 패턴 AI 제안
- 생성 파일 위치: `src/test/java/egovframework/covimail/` (covimail 모듈), `src/test/java/egovframework/coviexchange/` (coviexchange 모듈)

#### 자가 치유 및 유지보수 (Self-Healing)

- 소스 코드 변경 시 영향받는 TC 목록 AI 자동 탐지 계획
- `@Tag("wml-unit")` 태그 부여로 전체 단위 테스트 선택적 재실행 지원
- Redis TTL, 타임아웃 수치 등 비즈니스 상수 변경 시 테스트 파라미터 자동 갱신 검토

| 6 | **문서 추적성**: 본 계획서는 단위테스트 계획 단계 산출물로, 실행 결과는 `400AI_E_110.REQ-WML-001_단위테스트 결과서`에 기록 예정. |
