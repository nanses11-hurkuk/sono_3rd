### 2-1. 단위테스트 목적 및 범위

#### 목적

REQ-ODD-001에서 신규 구현·수정된 7개 GAP 해소 모듈의 개별 클래스·메서드 단위 기능 정확성을 격리된 Mock 환경에서 검증하여, 통합 테스트 진입 전 결함을 조기 탐지하고 회귀(Regression) 방지 기반을 확립한다. 특히 Exchange/LDAP 세션 재연결 Exponential Backoff, MSAL4J OAuth2 토큰 캐시, Redis 분산 락, 비동기 조직도 결과 폴링의 핵심 경계 케이스를 격리 검증한다.

#### 범위

| 범위 구분 | 내용 |
|---------|------|
| **대상 모듈** | `covimail` 모듈 — 신규 7개 파일 + 수정 4개 파일 (총 11개 대상) |
| **재활용 모듈 (제외)** | `smart4j_msapi`, `smarts4j_coviexchange` — 변경 없음, 단위테스트 불필요 |

| **인수 테스트** | 단위테스트 완료 후 별도 수행 (AC-ODD-001-01~12, Staging 환경). 본 계획서 범위 외 |

### 2-7. 단위테스트 데이터 전략

#### 전략 개요

단위 테스트는 실 외부 서비스(Azure AD, Exchange, LDAP, CoviMSAPI, Redis) 연결 없이 **완전 격리된 Mock 환경**에서 실행한다. 테스트 데이터는 `OddTestFixture.java` Fixture 클래스 및 `@BeforeEach`로 중앙 관리한다.

| 데이터 유형 | 준비 방법 | 적용 UT |

| MSAL4J 타임아웃 스텁 | WireMock 응답 지연 11,000ms | UT-07 |
| Redis 캐시 히트 (`outlookToken:{domainId}:{userId}`) | Embedded Redis 사전 주입 | UT-05 |
