### 2-7. 단위테스트 데이터 전략

#### 전략 개요

소스코드 수정 불가로 Mock 데이터 주입이 불가능하다. 대신 **개발 서버 DB에 테스트 전용 Vendor 계정 데이터를 직접 삽입**하고, API 블랙박스 호출로 검증한다. 테스트 종료 후 삽입 데이터는 `DELETE` 처리하여 원상 복구한다.

#### 데이터 준비 상세

| 데이터 유형 | 준비 방법 | 적용 TC |
|----------|---------|---------|
| 정상 Vendor 계정 | `INSERT INTO EUM_VENDOR_USER_MAP` — `VENDOR_ID='TST001'`, `USE_YN='Y'`, `EXT_USER_ID='test_user_01'` | TC-01, 05, 06, 08 |
| 비활성화 Vendor 계정 | `INSERT` 후 `UPDATE USE_YN='N'` | TC-04, 06 |
| 미등록 Vendor ID | 데이터 미삽입 (존재하지 않는 `VENDOR_ID='NONE'` 요청) | TC-03 |
| Redis ACL 키 | 정상 인증 후 자동 생성된 `eum:acl:vendor:TST001:test_user_01` 확인 | TC-05, 06 |
| JWT 미포함 요청 | `Authorization` 헤더 누락 Postman 요청 | TC-09 |
| 코드 리뷰 체크리스트 | 개발 상세 결과서 Code Diff 기준 12개 항목 점검표 | TC-02, 07, 12 |

#### 민감 데이터 처리 원칙

- 테스트용 `secret` 값은 실제 운영 Vendor 시크릿 절대 사용 금지 → 별도 테스트 Vendor 계정 신청
- 테스트 완료 후 삽입 데이터 즉시 `DELETE` + Redis 테스트 키 `redis-cli DEL` 처리
