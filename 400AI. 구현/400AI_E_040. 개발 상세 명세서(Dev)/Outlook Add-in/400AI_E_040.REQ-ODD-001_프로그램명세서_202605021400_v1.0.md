#### 2-1-4. 캐시 설계 (Redis)

설계 상세 결과서 2-2절에서 확정된 Redis 캐시 키 전체 명세는 아래와 같다. 본 요구사항 구현으로 신규 등록되는 키 패턴이며, 기존 Redis 키와 네임스페이스 충돌 여부를 배포 전 반드시 검토해야 한다.

| 키 패턴 | 자료형 | 값 | TTL | 용도 | 연관 GAP |
|---------|------|---|-----|------|---------|
| `lock:outlook:session:{mailAddress}` | String | "1" | 10초 | `createTempSession()` Distributed Lock. 10초 내 처리 완료 기대. Lock 해제 시 즉시 삭제(finally 블록) | G-05 |
| `lock:outlook:contacts:{userId}` | String | "1" | 30초 | `getContacts()` 동기화 Distributed Lock. EWS 조회 소요 시간 + 여유분 설정 | G-05 |
| `lock:outlook:distgroup:{groupCode}` | String | "1" | 30초 | `manageDistGroup()` 수정 Distributed Lock. LDAP + Exchange 이중 처리 소요 시간 고려 | G-05 |
| `outlookAsync:{taskId}` | String | XML 문자열 또는 "ERROR:{message}" | 300초 (5분) | 비동기 조직도 조회 결과 임시 저장. 대용량 XML 저장 가능. 메모리 용량 사전 점검 필요 | G-06 |
| `outlookToken:{domainId}:{userId}` | String | Bearer accessToken 값 | 3,500초 | OAuth2 AccessToken 캐시. Azure AD 토큰 유효 시간(3,600초) 이내로 TTL 설정 | G-02 |
| `outlook:session:{mailAddress}` | String | 세션 상태 정보 | 60초 | Exchange 세션 상태 정보 캐시. `ExchangeSessionManager` 헬스체크 주기와 일치 권장 | G-01 |
| `bridge:user:v1:{v1ContactId}` | String | 연락처 JSON | 60분 | 연락처 캐시 (Outlook ↔ 그룹웨어 매핑). 연락처 변경 시 캐시 무효화 전략 추후 검토 | G-04 |

**Redis 캐시 운영 상세 주의사항**

`outlookAsync:{taskId}` 키에 저장되는 조직도 XML은 조직 규모에 따라 수십 KB에서 수백 KB에 달할 수 있다. 동시에 다수의 조직도 조회 요청이 발생하면 Redis 메모리 사용량이 급격히 증가할 수 있으므로, 운영 Redis의 `maxmemory-policy`를 `allkeys-lru` 또는 `volatile-lru`로 설정하여 TTL 만료 전 메모리 부족 상황에 대비해야 한다. 또한, `outlookToken:{domainId}:{userId}` 키에 저장되는 OAuth2 AccessToken은 보안 민감 정보이므로, Redis 인스턴스에 대한 네트워크 접근 제어(VPC 내부 통신으로 제한)가 필수이다.

### 2-4. 핵심 로직 구현 상세 (Logic Details)

> **요구사항 ID**: REQ-ODD-001
> **산출 근거**: `300AI_D_040.REQ-ODD-001_설계 상세 결과서_202604151530_v1.0` 2-4절(구현 가이드), 2-8절(Service 메서드 시그니처)

#### 2-4-1. Service Layer 메서드 상세

**[G-01] ExchangeSessionManager.java 전체 구현 명세**

```java
/**
 * ExchangeSessionManager.java
 * 패키지: egovframework.covimail.common.service.impl
 * 역할: LDAP/EWS 세션 획득 시 최대 3회 Exponential Backoff 재시도 래퍼
 * 요구사항 ID: REQ-ODD-001
 * 산출 근거: 300AI_D_040.REQ-ODD-001_설계 상세 결과서 2-4절
 */
@Component
public class ExchangeSessionManager {

    private static final int    MAX_RETRY  = 3;
    private static final long[] BACKOFF_MS = { 1_000L, 2_000L, 4_000L };

    @Autowired
    private ResourceManager resourceManager;  // smarts4j_coviexchange 재활용

    /**
     * EWS ExchangeService 획득 (재시도 포함)
     * @param domainId    도메인 식별자
     * @param mailAddress 메일박스 소유자 메일 주소
     * @return ExchangeService 인스턴스
     * @throws Exception 3회 재시도 후 최종 실패 시
     */
    public ExchangeService getEwsSession(String domainId, String mailAddress)
            throws Exception {
        return executeWithRetry(() -> {
            ExchangeConfigDTO config =
                resourceManager.get(domainId).getEWSConfigDTO();
            ExchangeService svc =
                EWSCommon.getExchangeService(config, mailAddress);
            svc.getServerInfo();  // Ping: 연결 유효성 검증
            return svc;
        });
    }

    /**
     * LDAP 초기화 (재시도 포함)
     * LdapManager.initLdapConfig()의 매 요청 재생성 구조를 래핑하여 안정성 확보
     * @param ldapManager LdapManager 인스턴스
     * @param domainId    도메인 식별자
     * @throws Exception 3회 재시도 후 최종 실패 시
     */
    public void initLdapWithRetry(LdapManager ldapManager, String domainId)
            throws Exception {
        executeWithRetry(() -> {
            ExchangeConfigDTO config =
                resourceManager.get(domainId).getEWSConfigDTO();
            ldapManager.initLdapConfig(config);
            return null;
        });
    }

    /**
     * 재시도 공통 템플릿 메서드
     * Exponential Backoff: 1초 → 2초 → 4초
     * @param action 실행할 작업 (CheckedSupplier 람다)
     * @return action 실행 결과
     * @throws Exception 최대 재시도 횟수 초과 시 마지막 Exception 전파
     */
    private <T> T executeWithRetry(CheckedSupplier<T> action) throws Exception {
        int attempt = 0;
        while (attempt < MAX_RETRY) {
            try {
                return action.get();
            } catch (Exception e) {
                attempt++;
                if (attempt >= MAX_RETRY) throw e;
                // INFO 레벨 로그: 재시도 횟수, 대기 시간 기록
                Thread.sleep(BACKOFF_MS[attempt - 1]);
            }
        }
```

```java
/**
 * OAuth2AuthStrategy.java — Azure AD OAuth2 구현체 (신규)
 * MSAL4J ConfidentialClientApplication 기반
 * Redis 토큰 캐시: outlookToken:{domainId}:{mailAddress}, TTL 3,500초
 */
@Component("oauth2AuthStrategy")
public class OAuth2AuthStrategy implements OutlookAuthStrategy {

    @Override
    public String authenticate(String domainId, String mailAddress) throws Exception {
        String cacheKey = "outlookToken:" + domainId + ":" + mailAddress;

        // 1. Redis 캐시 조회 (캐시 히트 시 MSAL4J 호출 없이 반환)
        String cachedToken = RedisDataUtil.get(cacheKey);

        // 3. MSAL4J ConfidentialClientApplication 빌드 및 토큰 획득
        ConfidentialClientApplication app = ConfidentialClientApplication
            .builder(clientId,
                     ClientCredentialFactory.createFromSecret(clientSecret))
            .authority("https://login.microsoftonline.com/" + tenantId)
            .build();

        IAuthenticationResult result = app.acquireToken(
            ClientCredentialParameters.builder(
                Collections.singleton("https://outlook.office365.com/.default"))
            .build()
        ).get(10, TimeUnit.SECONDS);

        // 4. 토큰 Redis 캐시 저장 (TTL 3,500초 — Azure AD 기본 만료 3,600초 이내)
        String token = result.accessToken();

        RedisDataUtil.setEx(cacheKey, token, 3500);
        return token;
    }

    public String getUserType() { return "EXTERNAL_OAUTH2"; }
```

#### 2-4-2. 트랜잭션 관리

본 요구사항은 Spring `@Transactional`과 Exchange PowerShell/EWS/LDAP 호출 간 분산 트랜잭션이 지원되지 않는 구조이다. 트랜잭션 경계 및 보상 전략은 아래와 같다.

| 처리 레이어 | 트랜잭션 범위 | 롤백 가능 여부 | 보상 전략 |
|----------|------------|------------|---------|
| DB (MyBatis SQL 매퍼 조회) | `@Transactional(readOnly=true)` 적용 권장 | 가능 | — |
| LDAP 그룹 생성 → Exchange 메일 활성화 | 분산 트랜잭션 불가 | Exchange 측 롤백 불가 | 오류 발생 시 운영자 수동 보정 또는 이력 로그 기반 재처리 |
| Redis Distributed Lock | 트랜잭션 미적용 | — | finally 블록 Lock 해제로 자동 복구. TTL 만료 이중 보호 |
| `getOrgChartAsync()` 비동기 실행 | 트랜잭션 미적용 | — | 오류 시 Redis에 ERROR 상태 기록, 클라이언트에 재시도 안내 |

#### 2-4-3. 비동기 처리 상세

비동기 처리는 Spring `@EnableAsync` + `@Async("outlookAsyncExecutor")` 조합으로 구현한다. `outlookAsyncExecutor` Bean의 Thread Pool 설정값(corePoolSize: 3, maxPoolSize: 10, queueCapacity: 50)은 서버 CPU 코어 수 및 예상 동시 조직도 조회 요청 수를 기반으로 운영 환경에 맞게 재검토해야 한다. Thread Pool 포화(Queue Full) 발생 시 `TaskRejectedException`이 발생하므로, `OutlookAddinCon`에서 해당 예외를 처리하여 클라이언트에 적절한 오류 응답을 반환해야 한다. 비동기 결과 폴링 주기는 클라이언트 팀과 협의하여 결정하며, 과도한 폴링으로 인한 서버 부하를 방지하기 위해 최소 1초 이상의 간격을 권장한다.

#### 2-8-3. 회귀 영향 분석

`OutlookAddinCon.java`의 `switch(type)` 분기 수정으로 인해, 기존에 정상 동작하던 GETCONFIG, GETBIGATTACHINFO, FILEUPLOAD, GETBIGATTACH, GETORGMAP 케이스에 영향이 없어야 한다. GETORGCHART는 동기에서 비동기로 전환되므로, 기존 동기 응답을 기대하는 Outlook 클라이언트가 변경된 응답 구조(taskId 반환)에 대응하지 못하면 클라이언트 오류가 발생한다. 이를 방지하기 위해 서버 측 배포 전 클라이언트 팀의 폴링 로직 배포 완료 여부를 반드시 확인해야 한다.

`OutlookAddinSvcImpl.java`의 `getContacts()` 변경으로 기존 응답 XML 구조가 4단계 폴더로 확장되므로, Outlook 클라이언트 주소록 파싱 로직이 4개 `<GroupName>` 노드를 모두 처리할 수 있어야 한다.
