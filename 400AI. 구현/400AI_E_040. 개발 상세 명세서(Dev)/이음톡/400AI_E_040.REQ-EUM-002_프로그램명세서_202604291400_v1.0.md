#### Service Layer 메서드 상세

**EumMessageRouter.route()**:
```java
@Component
public class EumMessageRouter {

    private final EumMessageRepository    messageRepository;
    private final EumRedisIndexService    redisIndexService;
    private final RedisTemplate<String,String> redisTemplate;
    private final EumTenantRoutingFilter  routingFilter;
    private final ObjectMapper            objectMapper;

    public void route(EumMessage msg) {
        if (!isCrossTenant(msg)) {
            // 단일 테넌트: 기존 이음톡 채팅 처리 흐름으로 위임 (변경 없음)
            return;
        }
        // 크로스 테넌트 처리 시작
        if (!routingFilter.isAllowed(msg.getTenantId(), extractTargetTenantId(msg))) {
            throw new CrossTenantRoutingException("미허용 테넌트 라우팅: " + msg.getTenantId());
        }

        // Step 1: MongoDB 메시지 원본 저장 (유실 방지 최우선)
        EumMessage saved = messageRepository.save(msg);

        // Step 2: Redis 인덱스 갱신
        redisIndexService.addIndex(msg.getRoomId(), saved.getId(),
                                   msg.getTimestamp().getTime());

        // Step 3: Redis Pub/Sub 발행
        String topic = buildTopicName(msg.getTenantId(), msg.getRoomId());
        String payload = objectMapper.writeValueAsString(
                EumMessagePayload.from(msg));
        redisTemplate.convertAndSend(topic, payload);
    }

    private boolean isCrossTenant(EumMessage msg) {
        // sender와 target의 tenantId가 다르면 크로스 테넌트
        return !msg.getTenantId().equals(extractTargetTenantId(msg));
    }

    private String buildTopicName(String tenantId, String roomId) {
        return String.format("eum.routing.%s.%s", tenantId, roomId);
    }
}
```

**EumRedisIndexService.addIndex() / getLatestMessages() / acquireLock()**:
```java
@Service
public class EumRedisIndexService {

    private final RedisTemplate<String,String> redisTemplate;

    // 메시지 인덱스 추가 (Sorted Set: Score = 타임스탬프 ms)
    public void addIndex(String roomId, String messageId, long timestampMs) {
        String key = "eum:msg:idx:" + roomId;
        redisTemplate.opsForZSet().add(key, messageId, timestampMs);
    }

    // 최신 메시지 N건 조회 (타임스탬프 내림차순)
    public Set<String> getLatestMessages(String roomId, int count) {
        String key = "eum:msg:idx:" + roomId;
        return redisTemplate.opsForZSet().reverseRange(key, 0, count - 1);
    }

    // 분산 락 획득 (SETNX, TTL 5초)
    public boolean acquireLock(String roomId) {
        String key = "eum:lock:" + roomId;
        Boolean acquired = redisTemplate.opsForValue()
                .setIfAbsent(key, "1", 5, TimeUnit.SECONDS);
        return Boolean.TRUE.equals(acquired);
    }

    // 분산 락 해제
    public void releaseLock(String roomId) {
        redisTemplate.delete("eum:lock:" + roomId);
    }
}
```

#### 트랜잭션 관리

- RDBMS 채팅방 테이블 컬럼 갱신(`ROOM_TYPE`, `TENANT_SCOPE` 업데이트)은 Spring `@Transactional`(REQUIRED 전파)로 원자성을 보장한다.
- MongoDB 저장은 단일 문서 Write이므로 MongoDB 자체 원자성을 보장한다.
- MongoDB 저장 성공 후 Redis 인덱스 갱신 → Redis Pub/Sub 발행 순서로 처리한다. MongoDB 저장 실패 시 이후 Redis 처리를 전면 중단하여 메시지 유실을 방지한다.
- Redis 인덱스 갱신 실패 시 `@Retryable(maxAttempts = 3, backoff = @Backoff(delay = 300))` 재시도를 적용한다. 3회 재시도 후에도 실패 시 오류 로그를 기록하고 Pub/Sub 발행은 계속 진행한다(인덱스 누락은 차후 복구 가능).
- 강한 일관성이 필요한 채팅방 삭제 작업에는 `acquireLock(roomId)`로 분산 락을 획득한 후 처리하고, 완료 후 `releaseLock(roomId)`를 실행한다.

#### 비동기 처리

- `EumMessageRouter.route()` 내 MongoDB 저장, Redis 인덱스 갱신, Pub/Sub 발행은 채팅 응답 Latency를 최소화하기 위해 `@Async` 메서드로 비동기 처리한다.
- `@Async` 스레드 풀 설정:
  ```java
  @Bean("eumRoutingExecutor")
  public TaskExecutor eumRoutingExecutor() {
      ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
      executor.setCorePoolSize(5);
      executor.setMaxPoolSize(20);
      executor.setQueueCapacity(100);
      executor.setThreadNamePrefix("eum-routing-");
      executor.initialize();
      return executor;
  }
  ```
- Redis Pub/Sub 구독 리스너(`EumMessageBrokerConfig`)는 별도 스레드에서 비동기로 동작한다. 리스너 내에서 V2 Client WebSocket 전달과 Push 알림 발송을 처리한다.

#### MyBatis Mapper 쿼리 명세 (RDBMS 채팅방 유형 처리)

| Mapper 메서드 | SQL | 비고 |
|-------------|-----|------|
| `updateRoomType` | `UPDATE EUM_CHAT_ROOM SET ROOM_TYPE=#{roomType}, TENANT_SCOPE=#{tenantScope} WHERE ROOM_ID=#{roomId}` | 사간 채팅방 생성 시 호출 |
| `findByRoomId` | `SELECT ROOM_ID, ROOM_TYPE, TENANT_SCOPE FROM EUM_CHAT_ROOM WHERE ROOM_ID=#{roomId}` | 크로스 테넌트 여부 판별 시 활용 가능 |

#### 데이터 접근 권한

- V1 사용자는 V1 소속 채팅방 데이터에만 직접 접근 가능하며, V2 소속 채팅방 데이터는 Pub/Sub를 통해 전달받은 메시지만 표시한다.
- MongoDB `eum_messages` 컬렉션은 `tenantId` 필드를 기준으로 발신 계열사를 구분하며, V2 서버는 V2가 수신 대상인 메시지만 조회하도록 쿼리 조건을 제한한다.
- 크로스 테넌트 메시지 발행 전 `EumTenantRoutingFilter.isAllowed()`로 발신·수신 테넌트 조합의 허용 여부를 반드시 검증한다. 미등록 테넌트 조합은 라우팅을 거부한다.
