### ■ 배포 실행 요약 (Executive Summary)

**요구사항 개요**: 이음톡(EumTalk) / [이음톡] 멀티 테넌트 메시지 라우팅 시스템 구현

**요구사항 상세 설명**: 계열사 간(V1 ↔ V2) 원활한 채팅을 위한 Redis Pub/Sub 기반 메시지 브로커 및 라우팅 레이어를 구축한다. `EumMessageRouter`(크로스 테넌트 판별 → MongoDB 저장 → Redis Pub/Sub 발행), `EumRedisIndexService`(Sorted Set 인덱스 + 분산 락), `EumTenantRoutingFilter`(허용 테넌트 조합 관리)를 구현한다.

| 항목 | 내용 |
|------|------|
| **수정 파일** | `EUM_CHAT_ROOM_ALTER.sql` (컬럼 추가), `EumMessage.java` (신규), `EumMessageRepository.java` (신규), `EumMessageBrokerConfig.java` (신규), `EumMessageRouter.java` (신규), `EumRedisIndexService.java` (신규), `EumTenantRoutingFilter.java` (신규), `EumMessagePayload.java` (신규), `application.yml` (설정 추가) — 총 8개 파일 변경(1개 기존 DDL 추가, 7개 신규, 1개 설정 추가) |
| **DB 반영** | Oracle: `EUM_CHAT_ROOM` 테이블에 `ROOM_TYPE`, `TENANT_SCOPE` 컬럼 추가 (기존 데이터 `DEFAULT 'INTERNAL'` 자동 적용). MongoDB: `eum_messages` 컬렉션 신규 생성, `roomId+timestamp` / `tenantId+timestamp` 복합 인덱스 생성. |
| **테스트 통과율** | 자료없음 (단위 테스트 10개 TC 수행 후 결과 갱신 필요. 목표: 핵심 3개 클래스 90% 이상 커버리지) |
| **요구사항 충족** | 수용. 요구사항 기준(`REQ-EUM-002` — "V1, V2 간 채팅기능 구현 요청 고려")으로 개발 계획 / 상세 명세 / 소스 변경 요청서 추적키 정합성 확인 완료. 단, PM 데이터 격리 의사결정 선행 조건 이행 후 최종 충족 여부 확인. |

### ■ 기술적 변경 상세 (Technical Review)

**소스 코드 정합성 (Code Diff)**

| 구분 | Before | After |
|------|--------|-------|
| `EumMessageRouter.java` | 존재하지 않음 | 신규 생성. Smarts4j Extension Point 구현. `isCrossTenant()` 판별 → MongoDB 저장(`Step 1`) → Redis 인덱스 갱신(`Step 2`) → Pub/Sub 발행(`Step 3`) 순서 보장. MongoDB 저장 실패 시 Pub/Sub 발행 전면 중단. `@Async("eumRoutingExecutor")` 비동기 처리. |
| `EumMessageBrokerConfig.java` | 존재하지 않음 | 신규 생성. Redis Pub/Sub `PatternTopic("eum.routing.*")` 패턴 구독. `ThreadPoolTaskExecutor`(coreSize 5, maxSize 20, queue 100) 설정. |
| `EumRedisIndexService.java` | 존재하지 않음 | 신규 생성. Sorted Set `ZASET` 인덱스 (`eum:msg:idx:{roomId}`), SETNX 분산 락 (`eum:lock:{roomId}`, TTL 5초). `@Retryable(maxAttempts=3, delay=300ms)` 재시도 적용. |
| `EumTenantRoutingFilter.java` | 존재하지 않음 | 신규 생성. `application.yml` `eum.routing.allowed-routes` 설정 기반 허용 테넌트 조합 판별. 신규 계열사 추가 시 설정 변경만으로 확장 가능. |
| `EumMessage.java` | 존재하지 않음 | 신규 생성. MongoDB `@Document(collection="eum_messages")` 도메인 객체. `msg` 필드 로그 출력 금지 처리 포함. |
| `EUM_CHAT_ROOM` (기존 테이블) | `ROOM_TYPE`, `TENANT_SCOPE` 컬럼 없음 | `ALTER TABLE`로 컬럼 추가. 기존 레코드 `ROOM_TYPE DEFAULT 'INTERNAL'` 자동 적용. 기존 채팅 기능 영향 없음. |
| 기존 단일 테넌트 채팅 처리 | 기존 이음톡 채팅 처리 흐름 | 변경 없음. `isCrossTenant() == false` 시 즉시 반환하여 기존 흐름으로 위임. |

**DB 인프라 검증**

| 항목 | 내용 |
|------|------|
| DDL/엔티티 변경 | `EUM_CHAT_ROOM` 컬럼 추가 (`ROOM_TYPE VARCHAR2(20) DEFAULT 'INTERNAL'`, `TENANT_SCOPE VARCHAR2(200)`). 기존 데이터 정합성 유지 확인 필요. |
| MongoDB 컬렉션 | `eum_messages` 컬렉션 신규 생성. `roomId+timestamp` / `tenantId+timestamp` 복합 인덱스 적용 확인 필요. |
| 마이그레이션 검증 | 스테이징 DB에서 DDL 실행 후 기존 채팅방 데이터 `ROOM_TYPE = 'INTERNAL'` 자동 적용 확인 필요. |
| Redis 인프라 | `eum.routing.*` Pub/Sub 채널, `eum:msg:idx:*` Sorted Set, `eum:lock:*` 분산 락 키 기존 키 충돌 여부 확인. V1·V2 공유 Redis 연결 설정 확인 필요. |
| 롤백 스크립트 | `ALTER TABLE EUM_CHAT_ROOM DROP COLUMN ROOM_TYPE; ALTER TABLE EUM_CHAT_ROOM DROP COLUMN TENANT_SCOPE;` 별도 보관. MongoDB·Redis 데이터는 이전 버전 미참조이므로 서비스 영향 없음. |
