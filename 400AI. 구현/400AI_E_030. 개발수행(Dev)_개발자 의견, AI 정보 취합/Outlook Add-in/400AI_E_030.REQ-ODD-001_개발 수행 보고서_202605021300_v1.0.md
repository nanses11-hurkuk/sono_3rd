### 2-1. AI 의견

> **산출 근거**: `300AI_D_040.REQ-ODD-001_설계 상세 결과서_202604151530_v1.0` 소스코드 최종 재검증 결과, `200AI_C_030.REQ-ODD-001_분석 결과보고서_202604061908_v1.0` GAP 분석 결과, `400AI_E_020.REQ-ODD-001_개발 계획서_202605021200_v1.0`

본 요구사항(REQ-ODD-001)은 Outlook Add-in의 세션 안정화, 인증 다변화, 배포 그룹 관리 연동, 연락처 동기화 완성, Redis 분산 락, 비동기 처리라는 6가지 독립된 기술 과제를 하나의 요구사항으로 묶어 구현하는 복합적 성격의 작업이다. AI 관점에서 가장 중요한 판단 사항은 아래와 같다.

첫째, **설계 단계 소스코드 재검증을 통해 개발 범위가 분석 단계 예측 대비 유리하게 축소**되었다. `OrgSvc` CRUD 4개 메서드와 `RestGroupCon`(LDAP + Exchange PowerShell 이중 처리)이 이미 완비된 것이 확인되었으며, `ContactSvcImpl.selectContactList()` 및 `selectContactGroup()`도 EWS 완전 구현 상태임이 검증되었다. 따라서 G-03(배포 그룹)과 G-04/G-07(연락처)의 핵심 비즈니스 로직은 신규 개발이 아닌 연동 레이어 추가로 처리 가능하다.

둘째, **G-01(세션 재연결)과 G-02(OAuth2 인증)는 아키텍처 전략 관점에서 신중하게 접근해야 한다**. `LdapManager.initLdapConfig()`가 매 요청마다 `LdapTemplate`을 재생성하는 구조는 세션 풀이 없는 상태이므로, `ExchangeSessionManager`의 Exponential Backoff 재시도 래퍼가 없으면 베트남 현지 네트워크 레이턴시 환경에서 연속 장애가 발생할 수 있다. OAuth2 인증 전략은 `GraphServiceContext.accessToken` 필드 구조를 재활용하여 MSAL4J 기반으로 구현하되, `domainId` 기반 런타임 전략 선택(Kerberos ↔ OAuth2)이 올바르게 동작하는지 반드시 통합 테스트로 검증해야 한다.

셋째, **Redis Distributed Lock(G-05)은 모든 임계 구역에 일관되게 적용해야 하며, finally 블록 Lock 해제를 빠뜨리면 전체 서비스 잠금이 발생**할 수 있다. Lock TTL을 임계 구역 작업 최대 소요 시간보다 충분히 크게 설정하되, 무한 대기를 방지하기 위한 상한선 관리가 필요하다.

넷째, **비동기 처리(G-06)에서 taskId를 클라이언트에 즉시 반환한 후 Redis 저장이 실패하는 엣지 케이스**를 반드시 처리해야 한다. `outlookAsync:{taskId}` 키가 존재하지 않을 때 클라이언트가 무한 PENDING 상태로 빠지지 않도록, 폴링 API에서 일정 횟수 이상 PENDING이 지속될 경우의 처리 방침을 개발자와 PM이 사전에 합의해야 한다.

> **산출 근거**: `200AI_C_030.REQ-ODD-001_분석 결과보고서_202604061908_v1.0` (2)항 비고, `400AI_E_020.REQ-ODD-001_개발 계획서_202605021200_v1.0`

개발자(이동준) 의견은 분석 단계 수행보고서의 개발팀 확인 요청 내용 및 개발 계획서의 이슈/리스크 항목을 기반으로 아래와 같이 정리한다.

OAuth2 인증 지원 여부는 Outlook 클라이언트 버전별로 다르므로, 고객사 운영 환경의 Outlook 버전 목록을 확보한 후 `KerberosAuthStrategy`와 `OAuth2AuthStrategy` 중 기본 전략을 결정해야 한다. `CoviMSAPI` 호출 시 `baseUrl`(coviexchange 서버 URL)을 개발 환경과 운영 환경에서 분리하여 설정파일(`properties` 또는 Redis `getBaseConfig`)로 관리해야 하며, 배포 전 환경별 설정값 검증이 필요하다.

#### Domain Model

본 요구사항의 도메인 모델은 별도의 JPA Entity를 사용하지 않으며, Exchange Server와 Active Directory를 외부 도메인 소스로 사용하고 Redis를 캐시/락/비동기 저장소로 활용한다. 핵심 도메인 객체는 아래와 같다.

- **배포 그룹(Distribution Group)**: LDAP의 `groupCode`, `displayName`, `mail` 속성과 Exchange의 `primaryMail`로 구성. `GWGroupDTO`로 전달되며, LDAP 그룹(`LdapGroupSvcImpl`)과 Exchange 메일 그룹(`ExchGroupSvcImpl`) 두 레이어에 동시 존재.
- **연락처(Contact)**: Exchange EWS 기반의 `MSContactDTO`(개인 연락처)와 `MSContactGroupDTO`(그룹 연락처/배포그룹)로 구성. `selectContactList()` 반환 데이터는 이메일/전화/주소/확장속성을 완전 파싱한 상태.
- **비동기 작업(AsyncTask)**: `taskId`(UUID), `status`(PENDING/DONE/ERROR), `xmlResult`, `errorMessage`, `createdAt`, `completedAt`으로 구성. Redis에만 존재하며 DB에 저장되지 않음(`OutlookAsyncResultVO`로 클라이언트에 반환).
- **OAuth2 토큰(Token)**: `domainId`, `mailAddress`, `accessToken`, `expiresAt`으로 구성. Redis 키 `outlookToken:{domainId}:{mailAddress}`에 TTL 3,500초로 캐싱됨(`OutlookTokenVO`).

요구사항 REQ-ODD-001의 핵심 처리 흐름은 아래 4가지이다.

**흐름 1 — 세션 재연결 (G-01)**
Outlook 클라이언트에서 `POST /outlook/service.do` 요청이 들어오면 `OutlookAddinSvcImpl`에서 `ExchangeSessionManager.getEwsSession()` 또는 `initLdapWithRetry()`를 호출한다. `ExchangeSessionManager`는 내부적으로 `executeWithRetry()`를 통해 최대 3회(1초 → 2초 → 4초 Backoff) 재시도하며, 3회 초과 시 `Exception`을 던져 상위 호출자에게 전파한다.

**흐름 2 — 배포 그룹 CRUD (G-03)**
`MANAGEDISTGROUP` 요청이 들어오면 `manageDistGroup()`에서 Redis Lock(`lock:outlook:distgroup:{groupCode}`)을 획득한다. Lock 획득 성공 시 opType(CREATE/MODIFY/DELETE/ADD_MEMBER/REMOVE_MEMBER)에 따라 `CoviMSAPI.org()` 메서드를 분기 호출한다. `CoviMSAPI.org()`는 내부적으로 `OrgSvc` → `RestGroupCon`(LDAP + Exchange PowerShell 이중 처리)을 순차 호출한다. 처리 완료 후 finally 블록에서 Lock을 반드시 해제하고 XML 응답을 반환한다.

**흐름 3 — 연락처 동기화 (G-04, G-07)**
`GETCONTACTS` 요청이 들어오면 `getContacts()`에서 Redis Lock(`lock:outlook:contacts:{userId}`)을 획득한다. 이후 `CoviMSAPI.contact().selectContactList()`로 개인 연락처(EWS), `selectOrgChartDeptList` SQL 매퍼로 부서 연락처(DB), `selectOrgChartUserList` SQL 매퍼로 회사 연락처(DB), `CoviMSAPI.contact().selectContactGroup()`으로 배포 그룹 연락처(EWS)를 순차 조회한다. 4개 결과를 `<CPAddressBook><GroupName name="개인">...</GroupName>...</CPAddressBook>` XML 구조로 빌드하여 반환한다.

**흐름 4 — 비동기 조직도 조회 (G-06)**
`GETORGCHART` 요청이 들어오면 `OutlookAddinCon`에서 `getOrgChartAsync()`를 호출한다. 메서드는 `@Async("outlookAsyncExecutor")`로 즉시 백그라운드 실행되며, taskId(UUID)를 즉시 클라이언트에 반환한다. 백그라운드에서 조직도 XML 빌드가 완료되면 Redis 키 `outlookAsync:{taskId}`에 TTL 300초로 저장한다. 클라이언트는 `GET /outlook/asyncResult.do?taskId={UUID}`로 폴링하여 status(PENDING/DONE/ERROR) 및 result를 확인한다.

#### Security

인증 및 보안 처리는 아래와 같이 설계되어 있다.

- **CAUTK 토큰 기반 인증**: 기존 `POST /outlook/service.do`의 CAUTK 파라미터(`{mailAddress}|{timestamp}|{type}|{authKey}`)는 기존 방식을 그대로 유지한다. `authKey` 검증 로직은 기존 `OutlookAddinCon`의 인증 처리 로직을 변경하지 않는다.
- **OAuth2 토큰 보안**: MSAL4J로 획득한 Azure AD AccessToken은 Redis에 암호화 없이 저장되므로, Redis 인스턴스 자체에 대한 네트워크 접근 제어(방화벽) 및 인증이 필수이다. `clientSecret`은 소스코드에 하드코딩하지 않고 반드시 Redis `getBaseConfig` 또는 외부 비밀 관리 시스템을 통해 참조해야 한다.
- **Redis Distributed Lock 보안**: `RedisDataUtil.setnx()` 기반 Lock 키는 `lock:outlook:` 네임스페이스 접두사로 관리하여 다른 도메인의 Lock 키와 충돌하지 않도록 설계한다.
- **PowerShell 실행 권한**: WAS 서버에서 PowerShell Core(pwsh)를 실행하는 서비스 계정은 최소 권한 원칙(Principle of Least Privilege)을 적용하여 Exchange 관리 권한 외 불필요한 시스템 권한을 부여하지 않아야 한다.

#### 인수 테스트 시나리오

**시나리오 1 — 세션 재연결 안정성 (G-01)**

전제 조건: Exchange 서버 또는 LDAP 서버가 일시적으로 응답하지 않는 환경을 Mock으로 구성한다. 실행 순서: Outlook Add-in에서 임의의 요청(예: GETORGCHART)을 전송한다. Exchange 연결 1회 실패 후 1초 대기 → 2회 실패 후 2초 대기 → 3회 성공 시나리오를 수행한다. 기대 결과: 최종 정상 응답이 반환되고, 클라이언트에 오류가 노출되지 않아야 한다.

**시나리오 2 — OAuth2 인증 전략 전환 (G-02)**

전제 조건: Redis `OutlookAuthType` 설정값을 "OAuth2"로 설정하고 Azure AD 앱이 구성된 테스트 환경에서 수행한다. 실행 순서: `POST /outlook/service.do` 요청을 전송한다. `OutlookAuthStrategySelector`가 `OAuth2AuthStrategy`를 선택하고, MSAL4J로 Azure AD 토큰을 획득하여 Redis에 캐싱하는 과정을 로그로 확인한다. 동일 요청을 재전송하여 Redis 캐시 히트로 MSAL4J를 재호출하지 않는지 확인한다. 기대 결과: 첫 요청에서 MSAL4J 호출이 1회 발생하고, 두 번째 요청에서는 Redis 캐시가 사용되어야 한다.

**시나리오 3 — 배포 그룹 CRUD (G-03)**

전제 조건: coviexchange 서버 및 LDAP/Exchange 연동이 정상 동작하는 테스트 환경. 실행 순서: opType=CREATE로 신규 배포 그룹 생성 요청을 전송하고 XML 성공 응답 확인. opType=MODIFY로 해당 그룹의 displayName 변경 후 확인. opType=ADD_MEMBER로 구성원 추가 후 LDAP 그룹 구성원 목록 확인. opType=REMOVE_MEMBER(oldGroup 포함)로 구성원 제거 후 확인. opType=DELETE로 그룹 삭제 후 LDAP 그룹 미존재 확인. 기대 결과: 각 opType별 XML 성공 응답 반환 및 LDAP/Exchange 상태 반영 확인.

**시나리오 4 — 연락처 동기화 4단계 폴더 (G-04, G-07)**

전제 조건: Exchange EWS에 개인 연락처 및 그룹 연락처(배포 그룹)가 등록된 테스트 계정. 실행 순서: `GETCONTACTS` 요청을 전송하고, 응답 XML을 파싱한다. `<GroupName name="개인">`, `<GroupName name="부서">`, `<GroupName name="회사">`, `<GroupName name="배포그룹">` 4개 노드가 모두 존재하는지 확인한다. 각 폴더의 데이터가 EWS 및 DB 데이터와 일치하는지 검증한다. 기대 결과: 4개 폴더가 포함된 `<CPAddressBook>` XML 반환.

**시나리오 5 — Redis Distributed Lock 동시성 (G-05)**

전제 조건: Redis 서버가 정상 동작하는 테스트 환경. 실행 순서: 동일한 `groupCode`에 대해 `MANAGEDISTGROUP` 요청을 2개 스레드로 동시 전송한다. 기대 결과: 첫 번째 요청이 Lock을 획득하고 처리를 완료하며, 두 번째 요청은 Lock 획득 실패로 즉시 오류 응답(`IllegalStateException`)을 반환해야 한다.

**시나리오 6 — 비동기 조직도 조회 (G-06)**

전제 조건: `outlookAsyncExecutor` Bean이 등록된 서버 환경. 실행 순서: `GETORGCHART` 요청을 전송하고 taskId가 포함된 즉시 응답을 확인한다. `GET /outlook/asyncResult.do?taskId={taskId}`로 폴링하여 status=PENDING을 확인한다. 일정 시간 후 재폴링하여 status=DONE 및 result(XML)가 반환되는지 확인한다. 기대 결과: 비동기 처리 완료 후 status=DONE, result=조직도 XML 반환.

Outlook Add-in 클라이언트(COM Addin / Web Addin) 측에서는 아래 두 가지 변경 사항에 대한 클라이언트 연동 검토가 필요하다. 첫째, `GETORGCHART` 응답 방식이 기존 동기(XML 직접 반환)에서 비동기(taskId 반환 → 폴링)로 변경되므로, 클라이언트에서 폴링 로직을 신규 구현해야 한다. 둘째, `GETCONTACTS` 응답 XML이 4단계 폴더 구조(`개인/부서/회사/배포그룹`)로 확장되므로, 클라이언트의 주소록 트리 파싱 로직이 4개 폴더를 모두 처리할 수 있어야 한다. 이 두 가지는 서버 측 개발 완료 후 클라이언트 팀과의 별도 연동 테스트 일정을 수립해야 한다.

### 2-12. 트랜잭션 및 예외 처리 (Transaction & Exception) (개발+AI 의견 취합)

> **요구사항 ID**: REQ-ODD-001

#### Transaction (트랜잭션 전략)

본 요구사항은 Java(Spring)와 Exchange PowerShell/LDAP/EWS 간 분산 트랜잭션이 기본적으로 지원되지 않는 구조이다. 분석 결과보고서 3-6절에서 도출된 대응 전략은 아래와 같다.

| 시나리오 | 문제점 | 대응 방안 |
|--------|--------|---------|
| LDAP 그룹 생성 성공 → Exchange 메일 활성화(`enableGroup()`) 실패 | LDAP에는 그룹 존재, Exchange에는 메일 그룹 없음 | `ExchGroupSvcImpl.enableGroup()` 실패 시 로그 기록 후 운영자 수동 보정 또는 재시도 정책 수립 |
| `getContacts()` Redis Lock 획득 → EWS 조회 도중 Exception 발생 | Lock이 미해제 상태로 잔류 가능 | finally 블록에서 `RedisDataUtil.delete(lockKey)` 반드시 실행. TTL을 통한 자동 해제 이중 보호 |
| `manageDistGroup()` Redis Lock 획득 → `CoviMSAPI.org()` 호출 실패 | LDAP/Exchange 상태 변경 없음, Lock 잔류 | finally 블록 Lock 해제 + `CoviMSAPI.org()` Exception 상위 전파 후 XML 오류 응답 반환 |
| `getOrgChartAsync()` 비동기 실행 중 Exception 발생 | taskId는 이미 클라이언트에 반환됨 | catch 블록에서 Redis 키에 "ERROR:{errorMessage}" 저장. 폴링 API에서 status=ERROR 반환 |
| OAuth2 토큰 획득 성공 → Redis 저장 실패 | 다음 요청에서 캐시 미스로 MSAL4J 재호출 | Redis 저장 실패 시 로그만 기록하고 획득한 토큰을 그대로 사용. 기능 장애로 이어지지 않음 |

본 요구사항의 Spring `@Transactional` 적용 범위는 기존 SQL 매퍼 호출 부분에만 한정되며, Exchange PowerShell/EWS/LDAP 호출 결과는 Spring 트랜잭션 경계 밖에 위치한다. Exchange 계통 오류 발생 시 DB Rollback은 수행되나 Exchange 상태는 롤백되지 않으므로, 중요 배포 그룹 관리 작업의 경우 이력 로그 테이블 저장을 추후 검토하도록 PM과 협의를 권장한다.

#### Custom Exception (커스텀 예외 정의)

설계 상세 결과서에서 별도의 커스텀 예외 클래스가 명시되지 않았으므로, 아래는 AI 의견으로 제안하는 예외 처리 가이드이다. 개발자가 검토 후 채택 여부를 결정해야 한다.

| 예외 발생 위치 | 권장 처리 방식 | 클라이언트 응답 |
|------------|------------|------------|
| `ExchangeSessionManager.executeWithRetry()` — 3회 초과 실패 | `Exception` 상위 전파 | XML 오류 응답 (기존 오류 응답 포맷 준수) |
| `manageDistGroup()` — Redis Lock 획득 실패 | `IllegalStateException` 발생 | XML 오류 응답 ("배포 그룹 수정 중. 잠시 후 재시도." 메시지 포함) |
| `getContacts()` — Redis Lock 획득 실패 | XML 오류 응답 반환 | "getContacts: 동기화 진행 중입니다." 메시지 포함 |
| `manageDistGroup()` — 알 수 없는 opType | `IllegalArgumentException` 발생 | XML 오류 응답 ("Unknown opType: {opType}") |
| `getOrgChartAsync()` — 조직도 조회 실패 | catch 블록에서 Redis에 "ERROR:{message}" 저장 | 폴링 시 status=ERROR 반환 |

**2단계 — 운영 환경 배포 전 최종 점검**

배포 사전 체크리스트 8개 항목 전체 확인 완료 후, PM 최종 승인을 득한다. 배포는 서비스 영향을 최소화하기 위해 업무 시간 외(야간 또는 주말)에 수행하는 것을 권장한다.

**3단계 — 운영 배포 실행**

기존 운영 WAR/JAR 파일을 반드시 백업한다(`{파일명}_backup_{날짜시간}` 형식). 신규 빌드된 WAR/JAR를 배포하고 서버를 재기동한다. 기동 후 로그에서 `outlookAsyncExecutor` Bean 등록 메시지 및 `ExchangeSessionManager` 초기화 로그를 확인한다.

**4단계 — 배포 후 즉시 검증**

기존 기능(GETCONFIG, GETBIGATTACHINFO, GETORGMAP 등) 정상 동작 확인 (회귀 테스트). 신규 기능(MANAGEDISTGROUP — CREATE 1건, GETCONTACTS, GETORGCHART 비동기 폴링) 동작 확인.

#### 롤백 절차

배포 후 아래 조건 중 하나라도 발생하면 즉시 롤백을 실행한다.

- 기존 기능(GETCONFIG, GETORGMAP 등)에서 오류 응답이 발생하는 경우.
- `ExchangeSessionManager` 재시도가 모든 요청에서 3회 연속 실패하는 경우.
- Redis Lock 미해제로 전체 서비스가 잠기는 경우.
- `outlookAsyncExecutor` Thread Pool 포화(Queue Full)로 인한 요청 거부가 발생하는 경우.

롤백 실행 시 배포 전 백업해 둔 WAR/JAR 파일로 즉시 교체하고 서버를 재기동한다. Redis에 생성된 신규 키 패턴(`lock:outlook:*`, `outlookAsync:*`, `outlookToken:*`)은 모두 TTL이 설정되어 있으므로, 롤백 후 자동 만료된다. 단, `bridge:user:v1:{v1ContactId}` 키(TTL 60분)는 필요 시 `SCAN` 명령으로 삭제할 수 있다. 롤백 후 장애 원인을 분석하고 재배포 계획을 PM에게 보고한다.
