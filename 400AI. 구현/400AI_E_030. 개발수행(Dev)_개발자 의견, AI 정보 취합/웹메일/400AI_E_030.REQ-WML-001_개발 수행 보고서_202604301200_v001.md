#### Logic Flow

**[GAP-WML-001-02] OAuth2 AccessToken 갱신 흐름**

1. `MsMailSvcImpl` 또는 `EWSCommon`에서 Exchange 세션 생성 시 `ExchangeConfigDTO.authType`을 확인한다.
2. `authType`이 `"OAuth2"`인 경우, `TokenRefreshService.getOrRefreshToken(domainId)`를 호출한다.
3. `TokenRefreshService`는 Redis `msAccessToken:{domainId}` 키를 조회하여 유효한 토큰이 있으면 반환한다.
4. Redis에 토큰이 없거나 만료된 경우, MSAL4J `ConfidentialClientApplication.acquireToken(ClientCredentialParameters)`를 호출하여 신규 토큰을 발급한다. 최대 10초 대기한다.
5. 발급된 토큰은 Redis에 TTL 3,500초로 저장한 후 반환한다.
6. `EWSCommon`은 반환된 토큰을 `exchangeService.getHttpHeaders().put("Authorization", "Bearer " + accessToken)`으로 설정한다.
7. `authType`이 `"WebCredentials"`인 경우 기존 `WebCredentials(ewsUser, ewsPwd, ewsDomain)` 방식을 그대로 사용한다.

**[GAP-WML-001-05] 대용량 첨부 청크 업로드 흐름**

1. 클라이언트가 `POST /v1/mail/{mailBoxKey}/uploadSession`을 호출하여 업로드 세션을 생성한다.
2. `BigAttachHelper.createUploadSession()`이 Graph API `POST /v1.0/users/{mailBoxKey}/messages/{draftMessageId}/attachments/createUploadSession`을 호출하여 `uploadUrl`을 획득한다.
3. 세션 정보를 Redis `uploadSession:{sessionId}` 키로 TTL 600초 저장하고, `sessionId`를 클라이언트에 반환한다.
4. 클라이언트는 `PUT /v1/mail/{mailBoxKey}/uploadSession/{sessionId}`를 반복 호출하여 5MB 단위 청크를 전송한다.
5. `BigAttachHelper.uploadChunk()`가 `Content-Range: bytes {start}-{end}/{total}` 헤더와 함께 실제 `uploadUrl`에 PUT 요청을 전송한다.
6. 응답 코드 200·201·202 이외의 경우 예외를 발생시킨다. 202는 진행 중, 201은 완료를 의미한다.

**[GAP-WML-001-10] SvcImplFactory 런타임 전환 흐름**

1. `MailConAjax.java`는 HTTP 요청 헤더에서 `domainId`를 추출한다.
2. `SvcImplFactory.getMailService(domainId)`를 호출한다.
3. `SvcImplFactory`는 `RedisDataUtil.getBaseConfig("MailType", domainId)`를 조회하여 `"Exchange"` 또는 `"IMAP"` 값을 확인한다.
4. `"Exchange"`인 경우 `@Qualifier("exchangeMailService")` 구현체를, 그 외에는 `@Qualifier("imapMailService")` 구현체를 반환한다.
5. 컨트롤러는 반환된 `MailSvc` 인터페이스를 통해 메일 서비스를 호출한다.

#### Security

**인증 방식 이중화** (`300AI_D_040.REQ-WML-001` 2-4절 기준)

- 기존 방식(WebCredentials): EWS 사용자 계정/비밀번호 기반 NTLM/Kerberos 인증. 베트남 온프레미스 환경에서 사용.
- 신규 방식(OAuth2/ClientCredentials): Azure AD `ConfidentialClientApplication` + ClientCredentials Flow. M365 하이브리드 환경에서 사용. 토큰은 Redis에 암호화 없이 저장되므로, Redis 접근 통제(방화벽 등)를 인프라팀에서 보장해야 한다.

**보안 고려 사항**

- `oauthClientSecret`은 `ExchangeConfigDTO`에 포함되어 Redis `exchangeConfig:{DomainId}` 키로 저장된다. Redis 접근 권한이 있는 모든 프로세스가 이 값을 읽을 수 있으므로, Redis 인증(AUTH 명령) 및 TLS 설정이 필수이다.
- `TokenRefreshService`가 반환하는 토큰은 EWS 전체 접근 권한(`EWS.AccessAsApp`)을 포함하므로, 토큰 유출 시 전사 메일 접근이 가능해진다. 로그 출력 시 토큰 값이 노출되지 않도록 마스킹 처리가 필요하다.
- `setSession.ps1` OAuth2 전환 시 인증서 방식(`pwshCertPath`, `pwshCertPassword`)이 사용될 경우, 인증서 파일 접근 권한을 서비스 계정으로 제한해야 한다.

#### Caching (Redis)

본 요구사항의 데이터 저장은 RDB가 아닌 Redis에 집중된다. 신규 추가되는 Redis 키 네임스페이스는 다음과 같다.

| 키 패턴 | 용도 | TTL | 저장 값 타입 | GAP ID |
|---------|------|-----|------------|--------|
| `msAccessToken:{DomainId}` | MS Graph/EWS AccessToken 캐시 | 3,500초 | String (Bearer 토큰 원문) | GAP-WML-001-02 |
| `ewsSubscription:{DomainId}:{userId}` | EWS 구독 ID 및 만료 정보 | 구독 만료 시각(최대 253,800초) | JSON (EwsSubscriptionInfo 직렬화) | GAP-WML-001-09 |
| `uploadSession:{sessionId}` | Graph API 업로드 세션 URL | 600초 | String (uploadUrl 원문) | GAP-WML-001-05 |
| `exchangeConfig:{DomainId}` | 도메인별 Exchange 설정 전체 | 무기한 (수동 갱신) | JSON (ExchangeConfigDTO 직렬화) | GAP-WML-001-07 |

**캐싱 전략 설명**

- `msAccessToken:{DomainId}`: MSAL4J가 발급하는 토큰의 유효 시간은 3,600초(1시간)이며, 만료 100초 전(3,500초)에 Redis TTL이 만료되도록 설정하여 자동 재발급이 유도된다. 캐시 미스(miss) 시 MSAL4J를 통해 즉시 재발급하므로 사용자 체감 지연은 최초 요청 시에만 발생한다.
- `ewsSubscription:{DomainId}:{userId}`: EWS 구독의 최대 유효 시간은 4,230분(253,800초)이다. `SubscriptionRenewalScheduler`가 60분 주기로 만료 60분 이내 항목을 자동 갱신하며, 갱신 성공 시 Redis TTL도 새 만료 시각으로 업데이트한다.
- `uploadSession:{sessionId}`: Graph API 업로드 세션의 유효 시간은 10분이다. 클라이언트가 세션 만료 후 청크를 전송하면 404 또는 410 오류가 발생하며, 이 경우 신규 세션 생성을 유도한다.
- `exchangeConfig:{DomainId}`: 도메인별 Exchange 서버 설정은 변경 빈도가 매우 낮으므로 TTL을 무기한으로 설정한다. 설정 변경 시 `ResourceManager.updateConfig(domainId, config)` 메서드를 통해 Redis와 메모리 캐시를 동시에 갱신한다.

#### Migration

본 요구사항은 RDB 스키마 변경이 없으므로 마이그레이션 스크립트가 불필요하다. 단, `exchangeConfig:{DomainId}` Redis 키에 `ExchangeConfigDTO`의 신규 필드(`authType`, `oauthClientId`, `oauthClientSecret`, `oauthTenantId`, `imapConnectionTimeout`, `imapReadTimeout`)가 추가되므로, 배포 후 기존 Redis에 저장된 `exchangeConfig` 값을 신규 필드 포함 형태로 재저장하는 초기화 작업이 필요하다. 이는 각 도메인 관리자 화면에서 설정 저장 버튼을 다시 실행하거나, 초기화 배치 스크립트를 통해 처리한다.

### 2-12. 트랜잭션 및 예외 처리 (Transaction & Exception)

> **참조 요구사항 ID**: REQ-WML-001
> **참조 개발검토서 ID**: 400AI_E_020.REQ-WML-001_개발 계획서_202604301200_v001.md
> **산출 근거**: `300AI_D_040.REQ-WML-001_설계 상세 결과서_202604151530_v1.0` — 2-4절 구현 가이드, `200AI_C_020.REQ-ODD-001_분석 계획서_202604061700_v1.0` — 3-6절 데이터 정합성 및 트랜잭션 전략

#### Transaction

웹메일 모듈은 Exchange/Graph API와 Redis를 중심으로 동작하며, 전통적인 RDB 트랜잭션(`@Transactional`)이 적용되지 않는다. 대신 다음과 같은 분산 트랜잭션 전략을 적용한다.

**[TokenRefreshService] 토큰 발급·저장 원자성**

MSAL4J 토큰 발급과 Redis 저장은 원자적으로 처리되지 않는다. MSAL4J 발급 성공 후 Redis 저장 실패 시, 다음 요청에서 MSAL4J를 다시 호출하므로 기능적으로 문제는 없으나 Azure AD 호출 횟수가 증가한다. 이 상황은 허용 가능한 수준이며 별도 보상 트랜잭션이 불필요하다.

**[BigAttachHelper] 청크 업로드 멱등성**

Graph API 업로드 세션은 이미 업로드된 범위(`nextExpectedRanges`)를 추적한다. 동일 청크 재전송 시 Graph API가 중복 처리를 방지하므로 별도 멱등성 처리가 불필요하다. 단, 업로드 세션 만료(10분) 후 재시도 시에는 신규 세션 생성이 필요하다.

**[SubscriptionRenewalScheduler] 구독 갱신 원자성**

구독 갱신(`createSubscription`) 성공 후 Redis TTL 업데이트 실패 시, 다음 스케줄러 실행에서 만료 임박 항목으로 재처리된다. 이 경우 중복 갱신이 발생할 수 있으나, EWS 구독 갱신은 멱등적으로 처리되므로 기능 이상은 없다.

**[ResourceManager] 설정 동기화**

`updateConfig(domainId, config)` 호출 시 Redis 저장과 `ConcurrentHashMap` 업데이트가 순차 처리된다. Redis 저장 실패 시 예외를 발생시키고 `ConcurrentHashMap` 업데이트를 롤백하여 일관성을 유지한다.

#### Custom Exception

| 예외 클래스 | 발생 조건 | 처리 방법 |
|-----------|---------|---------|
| `TokenRefreshException` (신규 권고) | MSAL4J 토큰 발급 실패, Azure AD 연결 오류, 10초 타임아웃 초과 | 상위 호출자에게 전파, HTTP 503 또는 인증 오류로 변환하여 클라이언트에 반환 |
| `UploadSessionExpiredException` (신규 권고) | Redis `uploadSession:{sessionId}` 키 미존재(만료) | HTTP 410(Gone) 응답으로 클라이언트에 신규 세션 생성 안내 |
| `UploadChunkException` (신규 권고) | `uploadChunk()` 실행 중 HTTP 200·201·202 이외 응답 수신 | `IOException` 래핑하여 상위 전파, 클라이언트에 재시도 또는 세션 초기화 안내 |
| `ExchangeConfigNotFoundException` (신규 권고) | `ResourceManager`에서 `domainId`에 해당하는 설정 미존재 | HTTP 400(Bad Request) 또는 500으로 변환. Redis 설정 초기화 안내 |
| 기존 `Exception` | EWS 호출 실패, PowerShell 실행 오류 등 | 기존 예외 처리 패턴 유지 (입력 파일에서 구체적 구현 확인 필요) |

> **AI 의견**: 현재 설계서 의사코드에서는 `throws Exception`으로 일괄 처리되어 있다. 운영 안정성을 위해 위 커스텀 예외 클래스를 신규 정의하고, 예외 유형별로 HTTP 응답 코드와 메시지를 명확히 구분하는 것을 권장한다. 다만 기존 코드베이스의 예외 처리 패턴을 먼저 확인하고, 프로젝트 표준에 맞게 적용한다.

#### 배포 순서

1. **`smarts4j_coviexchange` 먼저 배포**: 신규 API 엔드포인트, `EWSCommon` 인증 분기, `ResourceManager` Redis 동기화 변경이 포함된다. `covimail` 배포 전에 먼저 배포하여 API 엔드포인트 가용성을 확보한다.
2. **`covimail` 배포**: `TokenRefreshService`, `SubscriptionRenewalScheduler`, `SvcImplFactory`, `BigAttachHelper` 변경이 포함된다. `smarts4j_coviexchange` 배포 완료 후 배포한다.
3. **초기화 작업 수행**: 각 도메인의 `exchangeConfig:{DomainId}` Redis 키에 신규 OAuth2 필드를 포함하여 재저장한다.
4. **`setSession.ps1` 수동 적용**: PowerShell 스크립트는 서버 파일시스템에 직접 배치하므로 별도 배포 절차로 처리한다.

#### 롤백 가이드

**롤백 조건**: 배포 후 Exchange 연동 전체 장애(메일 목록 조회 불가, 발송 불가) 또는 로그인/인증 오류가 발생하는 경우.

**`smarts4j_coviexchange` 롤백**

1. 이전 버전 JAR/WAR 파일로 재배포한다.
2. `EWSCommon.java`의 `authType` 분기가 제거되므로, `ExchangeConfigDTO`에 `authType` 필드가 없어도 기존 `WebCredentials` 방식으로 자동 동작한다.
3. Redis `exchangeConfig:{DomainId}` 키의 신규 필드는 롤백 후 무시되므로 별도 삭제가 불필요하다.

**`covimail` 롤백**

1. 이전 버전 JAR/WAR 파일로 재배포한다.
2. `SvcImplFactory` 롤백 시 `MailConAjax.java`가 이전 방식으로 구현체를 직접 참조하게 된다.
3. `SubscriptionRenewalScheduler`가 제거되므로 기존 구독은 만료 시점에 자연 종료된다. 실시간 알림이 중단될 수 있음을 운영자에게 사전 고지한다.
4. Redis `msAccessToken:*`, `uploadSession:*` 키는 롤백 후 자연 만료되므로 별도 정리가 불필요하다.

**`setSession.ps1` 롤백**

1. 백업된 이전 버전 `setSession.ps1`(Kerberos 인증 방식)으로 교체한다.
2. Exchange 서버 모던 인증이 이미 활성화된 경우, Kerberos 방식 복구가 불가할 수 있으므로 인프라팀과 사전 협의한다.
