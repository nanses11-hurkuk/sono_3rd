### [AI 의견]

분석 수행보고서(`200AI_C_030.REQ-EUM-002`) 및 설계 상세 결과서(`300AI_D_040.REQ-EUM-002`) 기준으로 본 요구사항에 대한 AI 검토 의견은 다음과 같다.

세 가지 EUM 요구사항 중 REQ-EUM-002가 기술 복잡도와 위험도가 가장 높다. 분석 단계에서 기술적 위험도를 PM 제시 2/5에서 개발팀 재평가 기준 3/5로 상향한 근거가 충분하다. Redis Pub/Sub 기반의 실시간 메시지 라우팅은 구현 자체는 가능하나, 다음 세 가지 전제 조건이 개발 착수 전에 반드시 해소되어야 한다.

첫째, V1·V2 데이터 격리 정책과 사간(社間) 메시지 공유 범위에 대한 PM의 명확한 의사결정이 선행되어야 한다. 이 결정이 없으면 메시지 저장 구조(어느 DB에 무엇을 저장할 것인지)와 MongoDB 클러스터 구성을 확정할 수 없다.

둘째, Redis Pub/Sub는 구독자가 오프라인 상태일 때 메시지를 유실하는 특성이 있다. 본 설계에서는 MongoDB에 메시지를 먼저 저장한 후 Redis Pub/Sub로 발행하는 방식으로 유실을 방지하도록 설계하였으나, V2 서버가 Redis 구독을 받지 못한 상황에서의 재전달 메커니즘(Re-delivery)을 명확히 구현해야 한다.

셋째, V1·V2가 공유하는 Redis 인스턴스의 물리적 위치와 베트남 현지 네트워크 Latency가 실시간 채팅 품질에 직접 영향을 미친다. 개발 착수 전 인프라 팀과 공유 Redis 위치 및 예상 RTT(Round Trip Time)를 협의해야 한다.

### [개발자 의견]

Smarts4j Extension Point를 활용하여 기존 이음톡 채팅 서버의 코어 수정 없이 `EumMessageRouter` 컴포넌트를 추가한다. 메시지 수신 시 크로스 테넌트 여부를 판별하고(`isCrossTenant()`), 크로스 테넌트 메시지인 경우에만 Redis Pub/Sub를 통해 발행하는 방식으로 구현한다.

메시지 원본은 Redis 발행 전에 반드시 MongoDB에 먼저 저장하여, 네트워크 장애 시 메시지 유실이 발생하지 않도록 한다. Redis 인덱스는 메시지 검색과 채팅방 상태 관리에만 사용하고, 원본 데이터의 참조 포인터 역할로 한정한다.

토픽 명명 규칙 `eum.routing.[TenantID].[RoomID]`를 모든 라우팅 처리에 일관되게 적용하여, 향후 추가 계열사 연동 시 신규 TenantID를 등록하는 것만으로 확장 가능한 구조를 유지한다. 분산 환경에서의 채팅방 상태 일관성은 최종 일관성(Eventual Consistency) 모델을 적용하고, 채팅방 삭제와 같이 강한 일관성이 필요한 작업에만 Redis SETNX 기반 분산 락을 사용한다.

- **SaaS 테넌트 격리 정책**: 크로스 테넌트 메시지 처리를 위해 테넌트 경계를 선택적으로 허용하는 예외 처리 로직이 필요하다. 이는 기존 SaaS 테넌트 격리 정책의 예외 케이스이므로, 관련 설정 변경 시 전체 테넌트 격리 정책에 의도치 않은 영향을 주지 않도록 주의한다.

#### Domain Model

멀티 테넌트 메시지 라우팅 도메인은 다음 핵심 객체로 구성된다.

- `EumMessage`: 메시지 도메인 객체. `sender`(발신자, 계열사 포함 식별자), `target`(수신자), `msg`(내용), `timestamp`, `tenantId`(발신 계열사), `roomId`(채팅방 ID) 필드로 구성된다.
- `EumMessagePayload`: Redis Pub/Sub 발행 및 수신에 사용하는 Payload DTO. JSON 직렬화 대상.
- `EumTenantRoute`: 크로스 테넌트 라우팅 설정 정보. 발신 TenantID, 수신 TenantID, 허용 여부, 토픽 명명 패턴을 포함한다.
- `EumMessageIndex`: Redis에 저장하는 메시지 인덱스 객체. MongoDB 문서 ID, 채팅방 ID, 발신 시각을 포함하며 검색 시 MongoDB 직접 조회를 대체한다.

#### API Spec

사간 채팅방 생성 및 메시지 라우팅에 사용하는 주요 인터페이스는 다음과 같다.

**Redis Pub/Sub 토픽 명명 규칙**:
- 형식: `eum.routing.[TenantID].[RoomID]`
- 예시: `eum.routing.V1.ROOM_20260429_001`

**크로스 테넌트 메시지 Payload**:
```json
{
  "sender": "V1_User",
  "target": "V2_User",
  "msg": "안녕하세요",
  "timestamp": "2026-04-29T13:00:00.000Z",
  "tenantId": "V1",
  "roomId": "ROOM_20260429_001"
}
```

**계열사 간 사용자 검색 API (간접 호출)**:
- `GET /api/eum/users/search?tenantId={tenantId}&keyword={keyword}`
- 상대 계열사 인스턴스의 조직도 API를 서버 사이드에서 간접 호출하여 결과를 반환.

#### Logic Flow

멀티 테넌트 메시지 라우팅 흐름은 다음 순서로 동작한다.

1. V1 사용자가 사간 채팅방에서 메시지를 송신한다.
2. V1 이음톡 서버의 `EumMessageRouter`가 메시지를 수신한다.
3. `isCrossTenant()` 메서드로 수신자가 다른 테넌트(V2) 소속인지 판별한다.
4. 크로스 테넌트 메시지인 경우, MongoDB에 메시지 원본을 저장한다(유실 방지).
5. Redis 인덱스를 갱신하여 검색 가능 상태로 만든다.
6. `eum.routing.V1.{RoomID}` 토픽으로 Redis Pub/Sub를 통해 메시지를 발행한다.
7. V2 이음톡 서버의 구독 리스너가 해당 토픽에서 메시지를 수신한다.
8. V2 서버는 V2 소속 수신자 세션에 메시지를 전달하고, Push 알림을 발송한다.
9. 메시지가 단일 테넌트 내 대화인 경우, Redis 발행 없이 기존 이음톡 채팅 처리 흐름으로 전달한다.

#### Security

- 크로스 테넌트 메시지 발행은 V1·V2 서버 간 사전 등록된 공유 Redis 인스턴스를 통해서만 허용한다. 외부에서 직접 Redis 토픽에 메시지를 주입하는 것을 차단하기 위해 Redis 접근을 내부 네트워크로 제한한다.
- 크로스 테넌트 메시지에 포함된 사용자 식별 정보(sender, target)는 서버 사이드에서 서명·검증하여 위변조를 방지한다.
- MongoDB에 저장되는 메시지 원본의 암호화 적용 여부는 법적 보존 요건 확인 후 결정한다.
- 계열사 간 사용자 검색 API는 반드시 인증된 서버 사이드 호출로만 허용하며, 클라이언트에서 직접 상대 계열사 API를 호출하는 구조를 금지한다.

#### Caching

Redis 사용 구조는 다음과 같다.

- **Pub/Sub 채널 키**: `eum.routing.{TenantID}.{RoomID}` — 크로스 테넌트 메시지 실시간 전달에 사용.
- **메시지 인덱스 키**: `eum:msg:idx:{roomId}` — Sorted Set(ZASET) 구조로 타임스탬프를 Score로 사용하여 시간 순 조회를 지원.
- **분산 락**: `eum:lock:{roomId}` — 채팅방 삭제 등 강한 일관성이 필요한 작업에만 Redis SETNX 기반 분산 락 적용. TTL 5초.
- **키 네임스페이스**: 모든 라우팅 관련 키는 `eum.routing.*` 또는 `eum:msg:*` 네임스페이스로 관리하여 기존 Redis 데이터와 충돌을 방지한다.

#### Migration

RDBMS 채팅방 테이블 컬럼 추가는 기존 데이터에 `ROOM_TYPE DEFAULT 'INTERNAL'` 값이 자동 적용되므로, 기존 채팅방 데이터의 정합성에 영향을 주지 않는다. MongoDB 및 Redis 신규 구성은 기존 시스템과 독립적으로 설치·구성한다. 마이그레이션 스크립트를 별도 파일로 관리하고, 스테이징 환경 적용 검증 후 운영에 반영한다.

**시나리오 1 — V1 ↔ V2 실시간 메시지 전달**

5. V2 사용자 화면에 V1 사용자의 메시지가 정상 표시되는 것을 확인한다.

**시나리오 2 — 메시지 유실 없는 재전달 처리**

1. V2 서버를 일시 중단한 상태에서 V1 사용자가 메시지를 송신한다.
2. MongoDB에 메시지 원본이 저장된 것을 DB에서 직접 확인한다.
3. V2 서버를 재기동하고, 미전달 메시지가 재전달되는 것을 확인한다.

**시나리오 3 — 단일 테넌트 채팅 영향 없음 검증**

1. V1 사용자 간 내부 채팅방에서 메시지를 주고받는다.
2. `EumMessageRouter`의 `isCrossTenant()` 판별 결과가 false이고, Redis Pub/Sub 발행이 발생하지 않는 것을 로그에서 확인한다.

#### 디자인 패턴

- **Observer / Pub-Sub 패턴**: Redis Pub/Sub를 통해 V1·V2 서버 간 메시지를 비동기적으로 전달. 발신자(V1 서버)와 수신자(V2 서버)가 직접 연결되지 않는 느슨한 결합 구조.
- **Template Method 패턴**: `EumMessageRouter.route()` 메서드 내에서 단일 테넌트 처리와 크로스 테넌트 처리를 분기하여, 향후 새로운 라우팅 전략 추가 시 확장 용이.
- **Facade 패턴**: `EumMessageRouter`가 MongoDB 저장, Redis 인덱스 갱신, Pub/Sub 발행의 세 단계를 단일 인터페이스로 제공.

#### Controller

사간 채팅 메시지 전달은 WebSocket을 통해 처리하므로 별도 REST Controller가 주 처리 경로는 아니다. 계열사 간 사용자 검색 API(`EumUserSearchController`)를 통해 V2 조직도 조회를 지원한다.

#### Service

`EumMessageRouter`가 핵심 서비스 역할을 수행한다. `route()` 진입 시 크로스 테넌트 여부를 판별하고, 크로스 테넌트인 경우 MongoDB 저장 → Redis 인덱스 갱신 → Pub/Sub 발행의 순서로 처리한다. 각 단계는 `@Async`로 비동기 처리하여 채팅 응답 Latency를 최소화한다.

#### Repository

`EumMessageRepository`는 Spring Data MongoDB를 사용하여 `eum_messages` 컬렉션에 메시지 원본을 저장·조회한다. `roomId + timestamp` 기준 페이지 조회를 지원한다.

#### DTO

- `EumMessagePayload`: Redis Pub/Sub 발행 Payload. `sender`, `target`, `msg`, `timestamp`, `tenantId`, `roomId` 필드. JSON 직렬화.
- `EumMessageSearchResult`: MongoDB 조회 결과 반환 DTO. 페이지 정보 포함.

### [트랜잭션 및 예외 처리 (Transaction & Exception)]

**요구사항ID**: REQ-EUM-002 | **참조 문서**: `200AI_C_030.REQ-EUM-002_분석 수행보고서`, `300AI_D_040.REQ-EUM-002_설계 상세 결과서`

#### Transaction

- RDBMS 채팅방 유형 컬럼 갱신은 Spring `@Transactional`로 원자성을 보장한다.
- MongoDB 메시지 저장은 MongoDB 단일 문서 Write 작업으로 원자성이 보장된다. MongoDB와 Redis의 교차 트랜잭션은 지원되지 않으므로, MongoDB 저장 성공 후 Redis 인덱스 갱신을 순차적으로 처리한다. Redis 갱신 실패 시 재시도(`@Retryable`, 최대 3회)를 적용한다.
- 분산 환경에서의 채팅방 상태 일관성은 최종 일관성(Eventual Consistency) 모델을 적용한다. 채팅방 삭제와 같이 강한 일관성이 필요한 작업에는 Redis SETNX 기반 분산 락(TTL 5초)을 사용한다.

#### Custom Exception

| 예외 클래스 | 발생 조건 | HTTP 상태 코드 | 오류 코드 |
|------------|----------|----------------|----------|
| `CrossTenantRoutingException` | 미허용 테넌트 간 라우팅 시도 | 403 | EUM_ROUTING_DENIED |
| `MessageStorageException` | MongoDB 메시지 저장 실패 | 500 | EUM_MSG_STORE_FAIL |
| `MessagePublishException` | Redis Pub/Sub 발행 실패 | 500 | EUM_MSG_PUBLISH_FAIL |
| `RedisIndexUpdateException` | Redis 인덱스 갱신 실패 | 500 | EUM_IDX_UPDATE_FAIL |
| `TenantLockAcquisitionException` | 분산 락 획득 실패 (채팅방 삭제 등) | 409 | EUM_LOCK_FAIL |

`MessageStorageException` 발생 시 Redis Pub/Sub 발행을 수행하지 않아 메시지 유실을 방지한다.

### [테스트 및 성능]

#### 테스트 전략

단위 테스트(Unit Test), 통합 테스트(Integration Test), 인수 테스트(Acceptance Test), 성능 테스트(Load Test) 4단계로 수행한다.

단위 테스트는 JUnit 5 + Mockito를 사용하여 MongoDB와 Redis를 Mock으로 대체한 후 라우터 로직을 독립적으로 검증한다. 통합 테스트는 Testcontainers를 활용하여 MongoDB와 Redis 컨테이너를 실행한 상태에서 메시지 저장, 인덱스 갱신, Pub/Sub 발행·수신 전체 흐름을 검증한다.

성능 기준: 메시지 라우팅(MongoDB 저장 + Redis 발행) 처리 시간은 단일 건 기준 200ms 이내를 목표로 한다. 동시 100명 사간 채팅 환경에서 메시지 전달 Latency가 500ms 이하를 유지해야 한다.

메시지 순서 보장: Redis Pub/Sub 발행 시 메시지 타임스탬프를 포함하고, 수신 측에서 타임스탬프 기준으로 정렬하여 출력하는 방식으로 논리적 순서를 보장한다.

#### 모니터링 / 로깅

- **로그 레벨**: `EumMessageRouter`의 크로스 테넌트 판별 결과, MongoDB 저장 완료, Redis 발행 완료를 INFO 레벨로 기록. MongoDB 저장 실패, Redis 발행 실패는 ERROR 레벨로 기록하고 알림을 발송한다.

- **모니터링**: Redis Pub/Sub 발행 건수, MongoDB 저장 건수, 라우팅 오류 건수, 처리 소요 시간 분포를 모니터링 지표로 수집한다. APM 도구를 활용하여 V1→V2 메시지 전달 End-to-End 소요 시간을 추적한다.

3. RDBMS 채팅방 테이블 컬럼 추가 스크립트를 스테이징 DB에서 실행하고 정상 적용을 검증한다.
4. 스테이징 환경에 애플리케이션을 배포하고 인수 테스트 시나리오 3개를 모두 수행한다.
5. 성능 테스트를 스테이징 환경에서 수행하여 목표치(메시지 라우팅 200ms 이내) 달성을 확인한다.
6. 운영 환경 MongoDB 클러스터 및 공유 Redis 인스턴스를 구성한다.
7. 운영 RDBMS에 채팅방 테이블 컬럼 추가 스크립트를 실행한다.
8. V1 서버, V2 서버 순서로 애플리케이션을 배포한다.
9. 배포 후 V1↔V2 메시지 전달 동작을 확인한다.

#### 롤백 가이드

배포 후 이상 발생 시 다음 순서로 롤백을 수행한다.

1. V2 서버, V1 서버 순서로 이전 버전 애플리케이션으로 즉시 재배포한다.
2. RDBMS 채팅방 테이블에 추가한 컬럼은 기존 서비스에 영향을 주지 않으므로 롤백 시 즉시 제거하지 않아도 된다. 단, `ROOM_TYPE` 컬럼을 참조하는 코드가 이전 버전에 없으므로 무시된다.
3. MongoDB와 Redis의 신규 데이터는 롤백 후에도 잔류하지만, 이전 버전 코드에서는 해당 데이터를 참조하지 않으므로 서비스 영향이 없다. 데이터 정리는 별도 스케줄로 처리한다.
4. 롤백 완료 후 장애 원인 분석 및 수정 후 재배포 일정을 수립한다.
