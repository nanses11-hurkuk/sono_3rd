### [AI 의견]

분석 수행보고서(`200AI_C_030.REQ-EUM-003`) 및 설계 상세 결과서(`300AI_D_040.REQ-EUM-003`) 기준으로 본 요구사항에 대한 AI 검토 의견은 다음과 같다.

세 가지 EUM 요구사항 중 REQ-EUM-003은 구현 난이도와 기술적 위험도가 가장 낮다(2/5). Smarts4j Core가 Spring 표준 `MessageSource` 기반 i18n을 사용하고 이음톡 모듈이 이를 준수한다면, 베트남어 언어 리소스 파일(`messages_vi.properties` 또는 JSON) 추가와 번역 데이터 입력이 핵심 작업이다.

그러나 소스코드 직접 분석이 이루어지지 않은 상태이므로 다음 두 가지 사항을 개발 착수 직후 즉시 확인해야 한다.

첫째, 이음톡 소스코드 내 하드코딩된 한국어 텍스트의 규모를 조기에 파악해야 한다. 하드코딩 텍스트가 많을수록 i18n 키 교체 작업이 늘어나 당초 PM 제시 공수(4 M/D)를 초과할 가능성이 있다. 소스코드 확보 직후 정적 분석 도구(grep, SonarQube 등)로 하드코딩 텍스트 현황을 전수 조사하는 것을 권고한다.

둘째, 베트남 현지 네트워크 환경에서 CDN 경유 웹 폰트 로딩 성능을 실측해야 한다. 초기 로딩 시간이 기준치를 초과하는 경우 폰트 서브셋(Subset) 적용 또는 로컬 폰트 배포 방식으로 전환할 준비를 해야 한다.

베트남어 번역 데이터는 현지 원어민 검수 없이 기계 번역 결과를 그대로 사용하면 사용자 경험에 부정적 영향을 준다. 검수 일정을 개발 일정에 반드시 포함해야 한다.

### [개발자 의견]

Smarts4j `ResourceBundle` 시스템에 `vi` 로케일을 추가하여 베트남어 언어 팩을 연동한다. 브라우저의 Accept-Language 헤더와 사용자 세션에 저장된 로케일 설정을 우선순위에 따라 감지하는 `EumLocaleResolver`를 구현하여, 사용자가 설정한 언어가 항상 우선 적용되도록 한다.

`GET /api/common/i18n/messages?lang=vi` 엔드포인트를 통해 프론트엔드 클라이언트가 언어 리소스를 동적으로 로드할 수 있도록 제공한다. 프론트엔드에서는 `langPack['vi'][key]` 방식으로 키를 조회하고, 키가 없는 경우 키 문자열을 그대로 반환하여 번역 누락 상황에서도 서비스가 중단되지 않도록 처리한다.

Noto Sans VI 웹 폰트는 CDN 경로를 CSS에 설정하되, 성능 측정 결과에 따라 서브셋 폰트(자주 사용하는 성조 문자 중심)를 별도 제작하여 로컬 배포하는 방안을 검토한다. `SYS_MULTILANGUAGE` 테이블 베트남어 컬럼 확장 및 데이터 전수 입력은 번역 검수 완료 후 DML 스크립트로 일괄 처리한다.

- **프론트엔드 i18n 처리**: 베트남어 언어 리소스 파일을 추가하는 것은 기존 언어(한국어, 영어)의 동작에 영향을 주지 않는다. 단, 하드코딩 텍스트를 i18n 키로 교체하는 과정에서 기존 화면에 텍스트 변경이 발생할 수 있으므로 교체 전후 화면을 비교 검증한다.

- **폰트**: 신규 웹 폰트(Noto Sans VI) 추가는 기존 폰트 스택을 교체하지 않고 베트남어 문자에만 선택적으로 적용하도록 CSS 폰트 스택을 구성한다. 기존 한국어 UI에 폰트 변경이 발생하지 않도록 CSS 적용 범위를 명확히 제한한다.
- **알림 메시지**: 서버에서 Push 알림 또는 시스템 알림 메시지를 생성하는 경우, 수신자 언어 설정에 따라 베트남어 메시지를 발송하도록 알림 처리 로직을 수정해야 한다. REQ-EUM-001과 REQ-EUM-002의 오류 메시지도 동일하게 베트남어 지원 연계가 필요하다.

### [아키텍처]

**요구사항ID**: REQ-EUM-003 | **참조 문서**: `300AI_D_040.REQ-EUM-003_설계 상세 결과서`, `400AI_E_020.REQ-EUM-003_개발 계획서`

#### Domain Model

베트남어 다국어 대응 도메인은 다음 핵심 객체로 구성된다.

- `I18nMessage`: 다국어 메시지 도메인 객체. `key`(i18n 키), `lang`(로케일 코드, 예: `vi`), `value`(번역된 텍스트) 필드로 구성된다.
- `UserLocale`: 사용자 언어 설정 값 객체. `userId`, `locale`(vi, ko, en 등) 필드로 구성된다.
- `LocaleContext`: 현재 요청의 로케일 정보를 담는 객체. 브라우저 헤더, 사용자 세션, 시스템 기본값 우선순위로 결정된 최종 로케일을 보유한다.

#### API Spec

**엔드포인트**: `GET /api/common/i18n/messages?lang=vi`

**Response Body (성공, HTTP 200)**:
```json
{
  "lang": "vi",
  "messages": {
    "chat.send": "Gửi",
    "chat.receive": "Nhận",
    "chat.room.create": "Tạo phòng chat",
    "chat.notification.newMessage": "Bạn có tin nhắn mới",
    "error.auth.fail": "Xác thực thất bại. Vui lòng thử lại.",
    "...": "..."
  }
}
```

**사용자 언어 설정 저장 엔드포인트**: `PUT /api/common/user/locale`

**Request Body**:
```json
{
  "locale": "vi"
}
```

**Response Body (성공, HTTP 200)**:
```json
{
  "result": "SUCCESS",
  "locale": "vi"
}
```

#### Logic Flow

베트남어 로케일 감지 및 리소스 반환 흐름은 다음 순서로 동작한다.

1. 클라이언트에서 HTTP 요청이 들어올 때 `EumLocaleResolver`가 로케일을 결정한다.
2. 로케일 결정 우선순위: ① 사용자 세션에 저장된 로케일 설정 → ② 브라우저 Accept-Language 헤더 → ③ 시스템 기본 로케일(`ko`).
3. 결정된 로케일이 `vi`인 경우, Smarts4j `ResourceBundle`에서 `messages_vi.properties` (또는 `SYS_MULTILANGUAGE` 테이블의 vi 컬럼)에서 키-밸류 쌍을 로드하여 반환한다.
4. 프론트엔드 클라이언트는 초기 로딩 시 `GET /api/common/i18n/messages?lang=vi`를 호출하여 전체 언어 리소스를 메모리에 캐싱한다.
5. UI 텍스트 표시 시 `getLabel(key)` 함수를 통해 캐싱된 언어 리소스에서 텍스트를 조회한다. 키가 없는 경우 키 문자열을 그대로 표시한다.
6. 사용자가 언어를 변경할 때 `PUT /api/common/user/locale`로 설정을 저장하고, 클라이언트가 새 언어 리소스를 재로드한다.

#### Security

- `GET /api/common/i18n/messages` 엔드포인트는 인증 없이 공개 접근을 허용한다. 언어 리소스에는 보안상 민감한 정보가 포함되지 않으므로 인증 불필요.
- `PUT /api/common/user/locale` 엔드포인트는 인증된 사용자만 호출 가능하도록 Spring Security에서 접근을 제한한다.
- `SYS_MULTILANGUAGE` 테이블의 번역 데이터는 관리자 권한을 가진 사용자만 수정할 수 있도록 관리자 인터페이스를 통해서만 접근 허용한다.

#### Schema

`SYS_MULTILANGUAGE` 테이블에 베트남어 컬럼을 추가한다. (기존 테이블의 전체 컬럼 구조는 소스코드 확보 후 확인하여 아래 스크립트를 확정한다.)

```sql
ALTER TABLE SYS_MULTILANGUAGE ADD (
    MSG_VI   VARCHAR2(4000)    -- 베트남어 번역 텍스트
);
COMMENT ON COLUMN SYS_MULTILANGUAGE.MSG_VI IS '베트남어 메시지';
```

인덱스는 기존 `LANG_CD` 또는 키 컬럼 기준 인덱스를 그대로 활용한다. 베트남어 데이터 입력은 번역 검수 완료 후 DML 스크립트(`SYS_MULTILANGUAGE_VI_DATA.sql`)로 일괄 처리한다. 데이터 입력 시 전체 이음톡 UI 텍스트 키(String Catalog)에 대응하는 베트남어 값을 빠짐없이 입력한다.

사용자 언어 설정 컬럼이 사용자 프로필 테이블에 없는 경우 아래와 같이 추가한다.

```sql
ALTER TABLE EUM_USER_PROFILE ADD (
    LOCALE    VARCHAR2(10) DEFAULT 'ko'    -- 사용자 언어 설정 (vi, ko, en 등)
);
COMMENT ON COLUMN EUM_USER_PROFILE.LOCALE IS '사용자 언어 설정 코드';
```

#### Caching

Redis 언어 리소스 캐싱은 다음 구조로 저장한다.

- **키**: `eum:i18n:{lang}` (예: `eum:i18n:vi`)
- **값**: JSON 문자열. 전체 키-밸류 언어 리소스 맵.
- **TTL**: 별도 설정. 언어 리소스는 배포 시점에 업데이트되므로 TTL 무제한(또는 장기)으로 설정하고, 배포 시 캐시 무효화(`DEL eum:i18n:vi`)를 자동 수행한다.
- **캐시 무효화**: 번역 데이터 업데이트(DML 실행) 또는 애플리케이션 배포 시, 관리자 캐시 초기화 API(`DELETE /api/admin/i18n/cache?lang=vi`) 또는 배포 스크립트에서 Redis 키 삭제를 수행한다.

#### Migration

`SYS_MULTILANGUAGE` 테이블 컬럼 추가는 기존 레코드에 `NULL` 허용으로 추가되므로 기존 데이터에 영향이 없다. `EUM_USER_PROFILE` 테이블 컬럼 추가도 기본값(`DEFAULT 'ko'`)이 적용되어 기존 데이터의 정합성이 유지된다. 마이그레이션 스크립트는 DDL과 DML을 분리하여 관리한다. DDL은 배포 전 적용하고, 번역 DML은 검수 완료 후 별도 적용한다.

#### 클래스 다이어그램 (텍스트 표현)

```
<<interface>>
LocaleResolver (Spring)
        ↑ implements
EumLocaleResolver
  + resolveLocale(HttpServletRequest) : Locale
```

#### 디자인 패턴

- **Strategy 패턴**: `EumLocaleResolver`에서 로케일 결정 전략(세션 → 헤더 → 기본값)을 체계적으로 적용. 향후 로케일 결정 기준 변경 시 구현체만 수정 가능.
- **Cache-Aside 패턴**: `EumI18nService`에서 언어 리소스 조회 시 Redis 캐시를 먼저 확인하고, 캐시 미스(Miss) 시 DB에서 로드하여 Redis에 저장하는 방식.

#### Controller

`EumI18nController`는 `GET /api/common/i18n/messages?lang={lang}` 요청을 처리하여 해당 언어의 키-밸류 맵을 반환하고, `PUT /api/common/user/locale` 요청을 처리하여 사용자 언어 설정을 저장한다.

#### Service

`EumI18nService`는 Redis 캐시 조회 → 캐시 미스 시 DB(`SYS_MULTILANGUAGE`) 조회 → Redis 캐시 저장 순서로 언어 리소스를 로드한다. 배포 시 캐시 무효화(`invalidateCache()`)를 자동 호출하도록 `@EventListener(ApplicationReadyEvent.class)`에 등록하는 방안을 검토한다.

#### Repository

`EumUserLocaleMapper`는 MyBatis 기반 Mapper 인터페이스로, `EUM_USER_PROFILE.LOCALE` 컬럼 조회 및 업데이트를 처리한다.

#### DTO

- `UserLocaleDto`: `PUT /api/common/user/locale` 요청 DTO. `locale` 필드. 지원 로케일 코드 허용 목록 유효성 검사(`@Pattern` 또는 `@Enum`) 적용.
- `I18nMessagesResponseDto`: 응답 DTO. `lang`, `messages` (Map<String, String>) 필드.

### [트랜잭션 및 예외 처리 (Transaction & Exception)]

**요구사항ID**: REQ-EUM-003 | **참조 문서**: `200AI_C_030.REQ-EUM-003_분석 수행보고서`

#### Transaction

- 사용자 언어 설정 저장(`EumUserLocaleMapper.updateLocale()`)은 단일 레코드 업데이트이므로 Spring `@Transactional`로 원자성을 보장한다.
- Redis 언어 리소스 캐싱은 트랜잭션 대상 외부로 처리한다. DB 조회 성공 후 Redis 저장이 실패하는 경우 다음 요청 시 DB에서 재조회하는 방식으로 자동 복구한다. Redis 저장 실패 자체는 서비스를 중단시키지 않는다.

#### Custom Exception

| 예외 클래스 | 발생 조건 | HTTP 상태 코드 | 오류 코드 |
|------------|----------|----------------|----------|
| `UnsupportedLocaleException` | 지원하지 않는 로케일 코드 요청 | 400 | EUM_UNSUPPORTED_LOCALE |
| `I18nResourceNotFoundException` | 언어 리소스 DB 및 캐시 모두 조회 실패 | 500 | EUM_I18N_LOAD_FAIL |
| `LocaleUpdateException` | 사용자 언어 설정 저장 실패 | 500 | EUM_LOCALE_UPDATE_FAIL |

`I18nResourceNotFoundException` 발생 시 빈 맵(`{}`)을 반환하여 클라이언트가 키 문자열을 그대로 표시하는 Fallback을 유지하도록 처리한다.

#### 롤백 가이드

배포 후 이상 발생 시 다음 순서로 롤백을 수행한다.

1. 이전 버전 애플리케이션 WAR/JAR로 즉시 재배포한다.
2. `SYS_MULTILANGUAGE` 테이블에 추가한 `MSG_VI` 컬럼은 기존 서비스에 영향을 주지 않으므로 롤백 시 즉시 삭제하지 않아도 된다. 이전 버전 코드에서는 해당 컬럼을 참조하지 않으므로 무시된다.
3. `EUM_USER_PROFILE.LOCALE` 컬럼도 동일하게 이전 버전에서 무시된다.
4. Redis `eum:i18n:vi` 캐시 키는 필요 시 `DEL eum:i18n:vi`로 삭제한다.
5. 롤백 완료 후 장애 원인 분석 및 수정 후 재배포 일정을 수립한다.
