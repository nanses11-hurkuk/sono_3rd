### [AI 의견]

분석 수행보고서(`200AI_C_030.REQ-EUM-001`) 및 설계 상세 결과서(`300AI_D_040.REQ-EUM-001`) 기준으로 본 요구사항에 대한 AI 검토 의견은 다음과 같다.

현행 이음톡은 내부 임직원 전용 인증 체계로만 운영되고 있으며, 외부 Vendor 사용자 인증을 위한 별도 플러그인, 계정 저장소, 세션 격리 로직이 전혀 존재하지 않는다. 이를 Smarts4j의 Extension Point(`CustomAuthenticator` 인터페이스)를 활용하여 코어 수정 없이 플러그인 방식으로 추가하는 설계 방향은 기술적으로 타당하다. 다만 다음 사항을 구현 착수 전에 반드시 검토해야 한다.

첫째, 외부 Vendor API의 응답 Timeout 정책이 정의되지 않으면 Vendor API 장애 시 이음톡 전체 로그인 흐름이 Block되는 위험이 있다. `VendorAuthPlugin` 내부에 Timeout(예: 3초), 재시도(Retry) 횟수, 장애 시 Fallback 처리를 명시적으로 구현해야 한다.

둘째, WebSocket 세션 토큰을 JWT로 발행하는 경우 토큰 서명 키 관리 방식과 만료 시간을 설계에서 명시하고, 반드시 WSS(TLS) 위에서만 동작하도록 인프라 구성을 확인해야 한다.

셋째, Redis TTL 2시간 세션 ACL 만료 후 자동 갱신(Refresh) 처리 여부를 개발 착수 전에 정책으로 결정해야 한다.

소스코드가 제공되지 않은 상태에서 작성한 분석 보고서의 한계를 감안하여, 실제 코드 수준의 구현 시 인증 흐름 통합 테스트를 필수로 수행할 것을 권고한다.

### [개발자 의견]

Smarts4j `CustomAuthenticator` Extension Point를 활용하여 기존 내부 인증 로직과 완전히 분리된 Vendor 전용 인증 플러그인(`VendorAuthPlugin`)을 구현한다. 내부 사용자 로그인 API와 Vendor 로그인 API는 엔드포인트부터 분리(`POST /api/eum/auth/external`)하여 역할을 명확히 구분한다.

`EUM_VENDOR_USER_MAP` 테이블을 신설하여 Vendor 계정 정보를 독립적으로 관리한다. Redis 기반 세션 ACL은 TTL 2시간으로 설정하며, 계정 비활성화 이벤트 발생 시 해당 키를 즉시 삭제하는 방식으로 세션 강제 종료를 처리한다. 채팅 채널 권한 필터는 WebSocket 핸드셰이크 단계에서 동작시켜, 권한 없는 채널 접근 시도를 연결 수립 전에 차단한다.

외부 Vendor API Timeout 정책은 3초로 설정하고, 초과 시 인증 실패 처리 후 오류 메시지를 반환한다. WSS 적용은 인프라 담당자와 사전 협의하여 배포 환경 기준으로 확인한다.

### [아키텍처]

**요구사항ID**: REQ-EUM-001 | **참조 문서**: `300AI_D_040.REQ-EUM-001_설계 상세 결과서`, `400AI_E_020.REQ-EUM-001_개발 계획서`

#### Domain Model

Vendor 사용자 인증 도메인은 다음 핵심 객체로 구성된다.

- `VendorCredential`: 외부 로그인 요청 정보를 담는 값 객체. `vendorId`, `extUserId`, `secret` 필드로 구성된다.
- `VendorAuthResult`: 인증 결과를 담는 값 객체. JWT Access Token, WebSocket Endpoint URL, 만료 시각 정보를 포함한다.
- `VendorUserMap`: `EUM_VENDOR_USER_MAP` 테이블과 매핑되는 도메인 엔티티. 계정 ID, 비밀번호(해시), 소속 Vendor사 코드, 연락처, 접속 권한 범위, 사용 여부, 등록일시, 만료일시 필드로 구성된다.
- `SessionAcl`: Redis에 저장되는 세션 권한 정보. 접근 가능한 채팅 채널 목록, TTL 정보를 포함한다.

#### API Spec

**엔드포인트**: `POST /api/eum/auth/external`

**Request Body**:
```json
{
  "vendorId": "V_001",
  "extUserId": "user_a",
  "secret": "암호화된_비밀값"
}
```

**Response Body (성공, HTTP 200)**:
```json
{
  "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "tokenType": "Bearer",
  "expiresIn": 7200,
  "websocketEndpointUrl": "wss://eum.example.com/ws/chat"
}
```

**Response Body (실패, HTTP 401)**:
```json
{
  "code": "EUM_AUTH_FAIL",
  "message": "외부 사용자 인증에 실패하였습니다."
}
```

**Response Body (Vendor API 타임아웃, HTTP 503)**:
```json
{
  "code": "VENDOR_API_TIMEOUT",
  "message": "외부 Vendor API 응답 시간이 초과되었습니다. 잠시 후 다시 시도해 주세요."
}
```

#### Logic Flow

외부 사용자 인증 흐름은 다음 순서로 동작한다.

1. 클라이언트가 `POST /api/eum/auth/external`로 `vendorId`, `extUserId`, `secret`을 전송한다.
2. `EumAuthController`가 요청을 수신하여 `VendorAuthPlugin`에 인증을 위임한다.
3. `VendorAuthPlugin`은 외부 Vendor API를 호출하여 사용자 유효성을 검증한다. Timeout(3초) 초과 시 503 응답을 반환하고 처리를 종료한다.
4. 외부 API 검증 성공 시, `EUM_VENDOR_USER_MAP` 테이블에서 내부 매핑 정보를 조회한다.
5. 내부 보안 정책(채팅 접근 권한) 체크를 수행하고, 접근이 허용된 채널 목록을 확정한다.
6. Smarts4j 세션 Context에 외부 사용자 정보(vendorId, extUserId, 권한 범위)를 주입한다.
7. JWT Access Token을 생성하고, Redis에 세션 ACL을 TTL 2시간으로 저장한다.
8. 클라이언트에 JWT Access Token 및 WebSocket Endpoint URL을 반환한다.
9. 클라이언트는 수신한 토큰을 WebSocket 핸드셰이크 헤더에 포함하여 채팅 서버에 연결 요청을 보낸다.
10. `EumChatChannelFilter`가 핸드셰이크 단계에서 토큰을 검증하고, Redis 세션 ACL에서 채널 접근 권한을 확인한다. 권한이 없으면 연결을 거부한다.

#### Security

- JWT 서명 알고리즘은 HS256(HMAC-SHA256)을 사용하며, 서명 키는 애플리케이션 설정 파일에서 환경 변수로 주입한다. 소스코드에 하드코딩 금지.
- JWT 만료 시간은 Redis 세션 ACL TTL(2시간)과 동일하게 설정하여 일관성을 유지한다.
- 모든 통신은 WSS(WebSocket over TLS) 위에서만 허용한다. HTTP(WS) 연결 시도는 거부 처리한다.
- 외부 Vendor API 호출 시 `secret` 값은 전송 구간 암호화(TLS) 외에 추가로 HMAC 또는 AES 암호화를 적용하는 것을 권고한다. (구체적 방식은 Vendor사와 협의하여 확정)
- Redis 세션 ACL 키 네임스페이스는 `eum:acl:vendor:{vendorId}:{extUserId}`로 설정하여 내부 사용자 세션 키와 충돌을 방지한다.

#### Schema

`EUM_VENDOR_USER_MAP` 테이블 DDL은 다음과 같이 설계한다.

```sql
CREATE TABLE EUM_VENDOR_USER_MAP (
    ID              VARCHAR2(36)   NOT NULL,          -- PK, UUID
    VENDOR_ID       VARCHAR2(50)   NOT NULL,          -- 소속 Vendor사 코드 (예: V_001)
    EXT_USER_ID     VARCHAR2(100)  NOT NULL,          -- 외부 사용자 ID
    PWD_HASH        VARCHAR2(256)  NOT NULL,          -- 비밀번호 해시(BCrypt)
    CONTACT         VARCHAR2(200),                    -- 연락처
    ACCESS_SCOPE    VARCHAR2(1000),                   -- 접속 권한 범위 (JSON 문자열)
    USE_YN          CHAR(1)        DEFAULT 'Y',       -- 사용 여부 (Y/N)
    REG_DT          TIMESTAMP      DEFAULT SYSTIMESTAMP NOT NULL, -- 등록일시
    EXPIRE_DT       TIMESTAMP,                        -- 만료일시
    CONSTRAINT PK_EUM_VENDOR_USER_MAP PRIMARY KEY (ID),
    CONSTRAINT UQ_EUM_VENDOR_USER UNIQUE (VENDOR_ID, EXT_USER_ID)
);
```

인덱스는 `VENDOR_ID`, `EXT_USER_ID` 복합 조건 조회 빈도가 높으므로 복합 인덱스를 추가한다.

```sql
CREATE INDEX IDX_EUM_VENDOR_USER ON EUM_VENDOR_USER_MAP (VENDOR_ID, EXT_USER_ID);
```

#### Caching

Redis 세션 ACL은 다음 구조로 저장한다.

- **키**: `eum:acl:vendor:{vendorId}:{extUserId}` (예: `eum:acl:vendor:V_001:user_a`)
- **값**: JSON 문자열. 접근 가능한 채팅 채널 ID 목록, 사용자 정보(vendorId, extUserId), ACL 생성 시각 포함.
- **TTL**: 7200초 (2시간). 세션 활동 시 자동 갱신 여부는 정책 결정 후 구현.
- **계정 비활성화 처리**: `USE_YN = 'N'` 업데이트 이벤트 발생 시 해당 Redis 키를 즉시 `DEL`하여 세션을 강제 종료한다.

#### Migration

`EUM_VENDOR_USER_MAP` 테이블은 기존 테이블에 영향을 주지 않는 신규 생성이므로 롤백 위험이 낮다. 마이그레이션 스크립트(`EUM_VENDOR_USER_MAP_DDL.sql`)를 별도 파일로 관리하고, 운영 환경 적용 전 개발·스테이징 환경에서 실행 검증을 완료한다. 롤백 시에는 `DROP TABLE EUM_VENDOR_USER_MAP;` 스크립트를 실행한다.

#### 인수 테스트 시나리오

**시나리오 1 — Vendor 사용자 정상 로그인 및 채팅 접속**

1. Vendor 사용자가 `POST /api/eum/auth/external`로 유효한 자격 증명을 전송한다.
2. JWT Access Token 및 WebSocket Endpoint URL을 정상 수신한다.
3. 수신한 토큰을 헤더에 포함하여 허용된 채팅 채널에 WebSocket 연결을 시도한다.

5. 2시간 경과 후 세션이 만료되고 채팅 연결이 끊어지는 것을 확인한다.

**시나리오 2 — 비허용 채널 접근 차단**

1. Vendor 사용자가 정상 로그인하여 JWT 토큰을 수신한다.
2. 해당 토큰으로 접근 권한이 없는 내부 전용 채팅 채널에 WebSocket 연결을 시도한다.
3. `EumChatChannelFilter`가 연결 요청을 거부하고 403 응답을 반환하는 것을 확인한다.

**시나리오 3 — 계정 비활성화 시 즉시 세션 종료**

1. Vendor 사용자가 정상 로그인하여 채팅 채널에 연결 중인 상태를 확인한다.
2. 관리자가 해당 계정의 `USE_YN = 'N'`으로 변경한다.
3. Redis ACL 키가 즉시 삭제되고 해당 WebSocket 연결이 강제 종료되는 것을 확인한다.

#### 디자인 패턴

- **Plugin / Strategy 패턴**: `CustomAuthenticator` 인터페이스를 통해 인증 구현체를 교체 가능한 구조로 설계. 향후 Vendor사별 상이한 인증 방식(OAuth2, SAML 등)을 별도 플러그인으로 추가 가능.
- **Chain of Responsibility 패턴**: Spring Security Filter Chain 및 WebSocket HandshakeInterceptor를 통해 인증·권한 검사를 단계적으로 처리.

#### Controller

`EumAuthController`는 `POST /api/eum/auth/external` 요청을 수신하여 `VendorAuthPlugin`에 인증을 위임하고, 결과를 `ResponseEntity`로 반환한다. 입력값 유효성 검사(`@Valid`)를 수행하고, 예외는 글로벌 예외 핸들러(`@ControllerAdvice`)로 위임한다.

#### Service

`VendorAuthPlugin`이 인증 서비스 역할을 수행한다. 외부 Vendor API 호출, 내부 정책 체크, 세션 Context 주입, JWT 토큰 생성의 전체 흐름을 단일 `validate()` 메서드 내에서 순차적으로 처리한다. 각 단계는 Private 메서드로 분리하여 가독성 및 단위 테스트 용이성을 확보한다.

#### Repository

`EumVendorUserMapper`는 MyBatis 기반 Mapper 인터페이스로, `EUM_VENDOR_USER_MAP` 테이블에 대한 CRUD를 처리한다. XML Mapper 파일(`EumVendorUserMapper.xml`)에 SQL을 정의한다.

#### DTO

- `VendorCredentialDto`: 요청 DTO. `vendorId`, `extUserId`, `secret` 필드. `@NotBlank` 유효성 검사 적용.
- `VendorAuthResponseDto`: 응답 DTO. `accessToken`, `tokenType`, `expiresIn`, `websocketEndpointUrl` 필드.
- `VendorUserMap`: DB 엔티티 매핑 객체. `EUM_VENDOR_USER_MAP` 테이블 컬럼 전체 포함.

### [트랜잭션 및 예외 처리 (Transaction & Exception)]

**요구사항ID**: REQ-EUM-001 | **참조 문서**: `200AI_C_030.REQ-EUM-001_분석 수행보고서`, `300AI_D_040.REQ-EUM-001_설계 상세 결과서`

#### Transaction

- `EumVendorUserMapper`의 CRUD 작업은 Spring `@Transactional`을 통해 원자성을 보장한다.
- `VendorAuthPlugin`의 인증 흐름은 외부 API 호출을 포함하므로, 트랜잭션 경계를 외부 API 호출 외부로 설정하여 트랜잭션 유지 시간을 최소화한다. 구체적으로 외부 Vendor API 호출 완료 후 DB 조회 및 Redis 저장 단계에서만 트랜잭션을 적용한다.
- Redis 저장(`EumSessionAclManager.save()`)은 트랜잭션 대상에서 제외하며, DB 커밋 완료 후 후속으로 처리한다. DB 성공 후 Redis 저장 실패 시 세션 ACL이 없는 상태가 되므로, 재시도 로직(`@Retryable`) 적용을 검토한다.

#### Custom Exception

| 예외 클래스 | 발생 조건 | HTTP 상태 코드 | 오류 코드 |
|------------|----------|----------------|----------|
| `VendorAuthFailException` | 외부 Vendor API 인증 실패 | 401 | EUM_AUTH_FAIL |
| `VendorApiTimeoutException` | 외부 Vendor API Timeout(3초 초과) | 503 | VENDOR_API_TIMEOUT |
| `VendorUserExpiredException` | 계정 만료(`EXPIRE_DT` 초과) | 401 | EUM_USER_EXPIRED |
| `VendorUserDisabledException` | 계정 비활성화(`USE_YN = 'N'`) | 401 | EUM_USER_DISABLED |
| `ChatChannelAccessDeniedException` | WebSocket 채널 접근 권한 없음 | 403 | EUM_CHANNEL_DENIED |

모든 Custom Exception은 `@ControllerAdvice` 기반 글로벌 예외 핸들러에서 일관된 응답 형식으로 처리한다.

### [테스트 및 성능]

**요구사항ID**: REQ-EUM-001 | **참조 문서**: `200AI_C_030.REQ-EUM-001_분석 수행보고서`, `400AI_E_020.REQ-EUM-001_개발 계획서`

#### 테스트 전략

단위 테스트(Unit Test), 통합 테스트(Integration Test), 인수 테스트(Acceptance Test) 3단계로 테스트를 수행한다.

단위 테스트는 JUnit 5 + Mockito를 사용하여 외부 Vendor API 및 Redis를 Mock으로 대체한 후 각 클래스의 로직을 독립적으로 검증한다. 통합 테스트는 Testcontainers를 활용하여 Redis 컨테이너를 실행한 상태에서 인증 흐름 전체를 검증한다. 인수 테스트는 개발 환경에서 실제 Vendor API 또는 Mock 서버를 연동하여 End-to-End 시나리오를 수행한다.

성능 기준은 다음과 같다. `POST /api/eum/auth/external` 엔드포인트의 평균 응답 시간은 외부 Vendor API 응답 시간을 제외하고 500ms 이내를 목표로 한다. WebSocket 핸드셰이크 단계의 권한 검증 처리 시간은 100ms 이내를 목표로 한다.

#### 모니터링 / 로깅

- **로그 레벨**: `VendorAuthPlugin`의 외부 API 호출 시작·완료·실패 이벤트를 INFO 레벨로 기록한다. 인증 실패, Timeout, 비활성화 계정 접근 시도는 WARN 레벨로 기록한다.
- **로그 항목**: 로그에는 `vendorId`, `extUserId`, 처리 결과(성공/실패), 소요 시간(ms)을 포함한다. `secret` 값은 절대 로그에 기록하지 않는다.
- **모니터링**: Redis ACL 키 생성·삭제 건수, 외부 Vendor API Timeout 발생 건수, 인증 실패 건수를 모니터링 지표로 수집한다. 운영 환경에서 APM(예: Elastic APM, Pinpoint)을 활용하여 인증 흐름 Trace를 수집하는 것을 권고한다.
