### 요구사항 상세 설명

설계 상세 결과서(`300AI_D_040.REQ-ODD-001`)에서 소스코드 최종 재검증(smart4j_msapi.zip, smarts4j_coviexchange.zip, covimail.zip)을 통해 확정된 구현 목표는 아래와 같다. 분석 단계(G-01~G-07)에서 도출된 GAP 항목 7건 전체가 설계 완료 상태이며, 본 개발 계획서는 해당 설계 내용을 구현 관점에서 재검증하여 작성한다.

#### G-01. Exchange/LDAP 세션 안정화 (ExchangeSessionManager 구현)

분석 단계에서 `LdapManager.initLdapConfig()` 메서드가 매 요청마다 `LdapTemplate` 객체를 새로 생성하는 구조(세션 풀 없음)임이 소스코드 재검증을 통해 확인되었다(`smarts4j_coviexchange/LdapManager.java:68`). 또한 기존 `getSession.ps1`은 세션 상태 확인만 수행할 뿐, 세션 만료 후 자동 재연결 로직이 미구현 상태이다. 이를 해결하기 위해 `egovframework.covimail.common.service` 패키지에 `ExchangeSessionManager` 클래스를 신규 구현한다. 해당 클래스는 LDAP 초기화(`initLdapWithRetry`) 및 EWS ExchangeService 획득(`getEwsSession`) 시 최대 3회, Exponential Backoff(1초 → 2초 → 4초) 방식의 자동 재시도 로직을 제공하는 래퍼(Wrapper) 역할을 수행한다.

#### G-02. 인증 다변화 (OutlookAuthStrategy — Kerberos/OAuth2 전략 분리)

기존 `setSession.ps1`은 `-Authentication Kerberos` 고정 방식으로 Exchange On-premise 인증을 처리하고 있으나, `smart4j_msapi/GraphServiceContext.java`에 `accessToken` 필드가 이미 존재함이 재검증을 통해 확인되었다. 이 구조를 활용하여 인증 방식 추상화 인터페이스(`OutlookAuthStrategy`)와 두 가지 구현체를 신규 개발한다. 첫째, `KerberosAuthStrategy`는 기존 `SessionCommonHelper` 기반의 CAUTK 세션 생성 방식을 Strategy 패턴으로 래핑하여 재활용한다. 둘째, `OAuth2AuthStrategy`는 MSAL4J `ClientCredentialParameters`를 이용한 Azure AD 토큰 획득 후 Redis 키(`outlookToken:{domainId}:{mailAddress}`, TTL 3,500초)에 캐싱하는 방식으로 신규 구현한다. 런타임에 `domainId` 기반으로 두 전략 중 하나를 선택하는 `OutlookAuthStrategySelector` 클래스도 함께 구현한다.

#### G-03. 배포 그룹 관리 신규 연동 (MANAGEDISTGROUP)

분석 단계에서는 `setDistributionGroup.ps1`의 이메일 주소 추가 기능만 확인되어 배포 그룹 CRUD 전체 구현이 필요한 것으로 판단하였으나, 설계 단계 소스코드 재검증을 통해 `smart4j_msapi/OrgSvc.java`에 `addGroup()`, `modifyGroup()`, `deleteGroup()`, `setGroupMember()` 메서드가 완전 구현되어 있고, `smarts4j_coviexchange/RestGroupCon.java`에 LDAP + Exchange PowerShell 이중 처리(POST `/v1/org/group`, PATCH, DELETE `/{groupCode}`, PATCH manager)가 완비되어 있음이 확인되었다. 따라서 실제 개발 범위는 `OutlookAddinCon.java`에 `MANAGEDISTGROUP` type 처리를 신규 추가하고, `OutlookAddinSvcImpl.java`에 `manageDistGroup()` 메서드를 구현하여 `CoviMSAPI.org()` 연동 레이어를 연결하는 작업으로 축소된다. `manageDistGroup()`은 opType(CREATE / MODIFY / DELETE / ADD_MEMBER / REMOVE_MEMBER)에 따라 해당 OrgSvc 메서드를 분기 호출하며, Redis Distributed Lock(`lock:outlook:distgroup:{groupCode}`, TTL 30초)을 적용하여 동시 수정을 방지한다.

#### G-04 / G-07. 연락처 동기화 완성 (getContacts — 4단계 폴더링)

기존 `covimail/OutlookAddinSvcImpl.java`의 677번, 717번 라인에 그룹 연락처 처리 및 쿼리 수정 관련 TODO 주석이 2건 미완성 상태로 존재함이 확인되었다. 그러나 설계 단계 재검증 결과, `smarts4j_coviexchange/ContactSvcImpl.java:281`에 `selectContactList()` 메서드가 페이지네이션, 확장속성, 이메일/전화/주소 파싱을 포함한 EWS 완전 구현체로 존재하고, `selectContactGroup()`도 EWS 그룹 연락처(배포 그룹) 조회를 완전 구현한 상태임이 확인되었다. 따라서 `getContacts()` 메서드는 아래 4단계 폴더 구조로 완성한다. 첫째, 개인 폴더는 `CoviMSAPI.contact().selectContactList()`를 통해 EWS에서 조회한다. 둘째, 부서 폴더는 기존 `outlook.addin.selectOrgChartDeptList` SQL 매퍼를 재활용한다. 셋째, 회사 폴더는 기존 `outlook.addin.selectOrgChartUserList` SQL 매퍼를 재활용한다. 넷째, 배포그룹 폴더는 `CoviMSAPI.contact().selectContactGroup()`을 통해 EWS에서 신규 조회한다(G-07 완성). 전체 응답은 `<CPAddressBook>` XML 포맷으로 빌드하여 반환한다. `getContacts()` 실행 시에도 Redis Distributed Lock(`lock:outlook:contacts:{userId}`, TTL 30초)을 적용한다.

#### G-05. Redis Distributed Lock 적용

동일 자원에 대한 동시 접근을 방지하기 위해 `RedisDataUtil.setnx()` 기반의 분산 락을 아래 3개 임계 구역에 적용한다. 첫째, `createTempSession()` — Lock 키: `lock:outlook:session:{mailAddress}`, TTL 10초. 둘째, `getContacts()` — Lock 키: `lock:outlook:contacts:{userId}`, TTL 30초. 셋째, `manageDistGroup()` — Lock 키: `lock:outlook:distgroup:{groupCode}`, TTL 30초. 각 메서드는 Lock 획득 실패 시 즉시 오류 응답을 반환하고, finally 블록에서 반드시 `RedisDataUtil.delete(lockKey)`를 호출하여 Lock을 해제한다.

#### G-06. 조직도 조회 비동기 전환 및 폴링 API 구현

현재 `GETORGCHART` 요청은 동기 방식으로 처리되어 대규모 조직도 조회 시 응답 지연이 발생한다. 이를 해소하기 위해 아래 두 가지를 구현한다. 첫째, `AsyncConfig.java`에 `outlookAsyncExecutor` Bean을 추가한다(corePoolSize: 3, maxPoolSize: 10, queueCapacity: 50). 둘째, `OutlookAddinSvcImpl.getOrgChartAsync()` 메서드에 `@Async("outlookAsyncExecutor")` 어노테이션을 적용하여 비동기로 전환한다. 비동기 실행 결과는 Redis 키(`outlookAsync:{taskId}`, TTL 300초)에 저장하며, `OutlookAddinCon.java`에 폴링 API(`GET /outlook/asyncResult.do?taskId={UUID}`)를 신규 추가하여 클라이언트가 상태(PENDING / DONE / ERROR)를 조회할 수 있도록 한다.

**상** — AD/EWS 연동 및 비동기 처리 포함

### 2-3. 프로그램명세서 검토 범위

설계 상세 결과서(`300AI_D_040.REQ-ODD-001`)에서 확정된 설계 결과물 중 개발자가 구현 전 반드시 검토해야 할 프로그램 명세 항목은 아래와 같다.

#### 컴포넌트 구조 및 시퀀스 다이어그램 (설계서 2-1절)

아래 4개의 시퀀스 다이어그램이 설계 상세 결과서에 확정되어 있으며, 구현 전 전체 흐름을 정확히 파악해야 한다.

- **시퀀스 다이어그램 1 (G-01)**: Exchange 세션 재연결 흐름 — `OutlookAddinSvcImpl` → `ExchangeSessionManager` → `LdapManager/EWSCommon` → Exchange/AD. 최대 3회 재시도(1초, 2초, 4초 간격) 포함.
- **시퀀스 다이어그램 2 (G-03)**: 배포 그룹 CRUD 흐름 — `Outlook Addin` → `OutlookAddinSvcImpl`(Redis Lock) → `CoviMSAPI` → `OrgSvc` → `RestGroupCon`(LDAP + Exchange) → Exchange.
- **시퀀스 다이어그램 3 (G-04, G-07)**: 연락처 동기화 흐름 — `Outlook Addin` → `OutlookAddinSvcImpl`(Redis Lock) → `CoviMSAPI` → `ContactSvc` → `RestContactCon`(EWS) → EWS. 4개 폴더(개인/부서/회사/배포그룹) XML 빌드 포함.
- **시퀀스 다이어그램 4 (G-06)**: 비동기 조직도 조회 흐름 — `Outlook Addin` → `OutlookAddinCon` → `getOrgChartAsync()`(taskId 즉시 반환) → `AsyncExecutor`(비동기 실행) → Redis 결과 저장. 폴링 API(`GET /outlook/asyncResult.do`) 통한 결과 확인 흐름 포함.

#### API 명세 (설계서 2-3절, 2-9절)

기존 Outlook Addin 진입점(`POST /outlook/service.do`) 및 신규 폴링 API(`GET /outlook/asyncResult.do`) 명세, coviexchange 배포 그룹 REST API, coviexchange 연락처 REST API 전체 명세가 설계서에 확정되어 있으며, 구현 시 해당 명세를 정확히 준수해야 한다.

- `POST /outlook/service.do` — CAUTK 파라미터 구조 `{mailAddress}|{timestamp}|{type}|{authKey}|[opType]|[jsonParam]` 및 type별 처리 메서드 매핑(GETCONFIG, GETBIGATTACHINFO, FILEUPLOAD, GETBIGATTACH, GETORGMAP, GETORGCHART[비동기 전환], GETCONTACTS[완성], MANAGEDISTGROUP[신규]) 전체 명세 검토 필요.
- `GET /outlook/asyncResult.do?taskId={UUID}` — 응답 포맷 `{ status: "PENDING"|"DONE"|"ERROR", result: "...|null", errorMessage: null|"..." }` 명세 검토 필요.
- `MANAGEDISTGROUP` opType별 처리 명세 — CREATE(`addGroup`), MODIFY(`modifyGroup`), DELETE(`deleteGroup`), ADD_MEMBER/REMOVE_MEMBER(`setGroupMember`, oldGroup 포함 여부) 처리 로직 검토 필요.

#### 구현 가이드 (Pseudo Code / Logic) (설계서 2-4절)

아래 구현 가이드 코드(Pseudo Code)가 설계서에 포함되어 있으며, 실제 구현 시 해당 로직을 기준으로 작성해야 한다.

- `ExchangeSessionManager.java` — `executeWithRetry()` 메서드의 재시도 루프 로직 및 `CheckedSupplier` 함수형 인터페이스 구현 가이드.
- `OutlookAuthStrategy` 인터페이스 및 `KerberosAuthStrategy`, `OAuth2AuthStrategy` 구현체 — MSAL4J `ConfidentialClientApplication` 빌더 패턴, Redis 토큰 캐시 조회 및 저장 로직.
- `OutlookAddinSvcImpl.manageDistGroup()` — opType 분기 switch문, Redis Lock 획득/해제(finally 블록), `CoviMSAPIUtil.set()` 기반 API 초기화 및 호출 패턴.
- `OutlookAddinSvcImpl.getContacts()` — 4단계 폴더 XML 빌드 로직, Redis Lock 적용, `CoviMSAPI.contact()` 순차 호출 패턴.
- `AsyncConfig.java` — `outlookAsyncExecutor` ThreadPoolTaskExecutor Bean 설정 값(corePoolSize, maxPoolSize, queueCapacity, threadNamePrefix).
- `OutlookAddinSvcImpl.getOrgChartAsync()` — `@Async` 적용, taskId UUID 생성, Redis `setEx("outlookAsync:{taskId}", xmlResult, 300)` 저장 패턴.
- `OutlookAddinCon.java` 폴링 API — Redis `get("outlookAsync:{taskId}")` 조회 후 상태 분기(null→PENDING, "ERROR:"→ERROR, 나머지→DONE) 처리 패턴.
- Redis Distributed Lock 유틸 공통 패턴 — `RedisDataUtil.setnx(lockKey, "1", ttlSeconds)` 획득, finally 블록 `RedisDataUtil.delete(lockKey)` 해제.

#### 패키지 구조 (설계서 2-5절)

설계서에서 확정된 패키지 구조(`covimail`, `smart4j_msapi`, `smarts4j_coviexchange` 각 모듈의 신규/수정/재활용 파일 전체 목록)를 검토하여, 신규 파일 생성 위치 및 기존 파일 수정 대상을 명확히 파악한 후 개발에 착수해야 한다.

#### DTO/VO 및 Service 메서드 시그니처 (설계서 2-7절, 2-8절)

재활용 DTO(`GWGroupDTO`, `GWGroupMemberDTO`, `GWContactDTO`, `GWContactGroupDTO`, `GWContactListDTO`, `MSContactDTO`, `MSContactGroupDTO`, `MSEWSGroupDTO`) 및 신규 VO(`OutlookAsyncResultVO`, `OutlookTokenVO`) 필드 정의, 기존 재활용 메서드 시그니처 목록, 신규 메서드 시그니처 목록 전체를 검토해야 한다.

#### 외부 연동 현황 (설계서 2-10절)

연동 대상별 프로토콜, 현재 구현 상태, 변경 내용 매트릭스(`smart4j_msapi`, `smarts4j_coviexchange`, Active Directory, Exchange Server, MSAL4J, Redis, Smarts4j Core)를 검토하여 연동 방식 및 변경 범위를 정확히 이해해야 한다.

#### 미결 항목 및 전제 조건 (설계서 5절)

설계서에 명시된 미결 항목 3건을 개발 착수 전 반드시 해소해야 한다. 첫째, Azure AD 앱 권한(`Contacts.ReadWrite`, `Group.ReadWrite.All`) — Graph 모듈 사용 시 PM 및 인프라팀 확인 필요. 둘째, `setSession.ps1`의 `-Authentication Kerberos` 고정 값 — OAuth2 전환 시 Exchange 서버 모던 인증 설정 필요(개발자 + 인프라팀). 셋째, `CoviMSAPI` 호출 시 `baseUrl`(coviexchange 서버 URL)의 개발/운영 환경별 분리 확인 필요(개발자).

### 2-6. 개발 가능 여부

**개발 가능 — 개발 필요**

분석 단계에서 요구사항 수용 여부가 **수용**으로 확정되었으며(분석 결과보고서 (2)항), 설계 단계에서 GAP 항목 G-01~G-07 전체에 대해 **설계 완료** 상태가 확인되었다(설계 상세 결과서 (3)절 GAP 해소 최종 현황표). 단, 아래 이슈 및 리스크 항목이 해소되지 않을 경우 개발 범위 및 일정이 변경될 수 있으므로, 착수 전 PM 및 인프라팀과 사전 협의가 반드시 필요하다.

| 번호 | 이슈 / 리스크 | 위험도 | 조치 필요자 |
|------|-------------|--------|-----------|
| 1 | Azure AD 앱의 연락처 및 그룹 관리 권한(`Contacts.ReadWrite`, `Group.ReadWrite.All`) 확보 여부 미확인 (Graph 모듈 사용 시 필수) | 높음 | PM + 인프라팀 |
| 2 | 아웃룩 클라이언트 버전별 OAuth2 지원 여부 확인 필요. OAuth2 미지원 버전에서는 `KerberosAuthStrategy`만 사용 가능 | 높음 | 개발자 + 인프라팀 |
| 3 | 대규모 조직도 비동기 처리 시 Redis 가용 메모리 확인 필요 (`outlookAsync:{taskId}` 키 다수 생성 시 메모리 사용량 증가 가능) | 중간 | 인프라팀 |
| 4 | 베트남 현지 네트워크 레이턴시로 인한 WinRM PowerShell 세션 응답 시간 지연 가능성 (기술적 위험도 4/5). `New-PSSessionOption`으로 `OpenTimeout`, `OperationTimeout` 명시적 설정 필요 | 높음 | 개발자 + 인프라팀 |
| 5 | Exchange On-premise 버전에 따른 PowerShell Remoting 방식 차이 가능성. `New-PSSession`은 PowerShell 5.x 기반이며, PowerShell 7.x(Core) 환경에서는 Exchange 연결 방식 확인 필요 | 중간 | 개발자 + 인프라팀 |
| 6 | MS Exchange On-premise 지원 축소 및 Exchange Online 전환 정책에 따른 PowerShell Remoting 방식 중단 위험 (기술적 위험도 5/5). 베트남 현지 인프라 로드맵 사전 확인 권장 | 매우 높음 | PM + 고객사 |
