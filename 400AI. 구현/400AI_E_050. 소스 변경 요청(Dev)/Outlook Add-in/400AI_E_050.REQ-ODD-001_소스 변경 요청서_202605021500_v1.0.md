#### 2-5-2. 배포 절차

**1단계 — 개발 환경 배포 및 검증**

신규 클래스 7종(`ExchangeSessionManager.java`, `OutlookAuthStrategy` 계열 4종, `OutlookAsyncResultVO.java`, `OutlookTokenVO.java`)이 포함된 빌드 결과물(WAR/JAR)을 개발 서버에 배포한다. 수정 대상 파일 4종(`OutlookAddinCon.java`, `OutlookAddinSvc.java`, `OutlookAddinSvcImpl.java`, `AsyncConfig.java`)의 변경 내용이 반영되었는지 확인한다. 서버 기동 후 단위 테스트(TC ID: UT-ODD-001-01~28) 결과를 확인하고, 인수 테스트 시나리오 6건(AC-ODD-001-01~12)을 순차 수행한다. 모든 테스트 통과 후 운영 배포 단계로 진행한다.

**2단계 — 운영 배포**

배포 사전 체크리스트 8개 항목 전체 확인 완료 후 PM의 최종 승인을 득한다. 기존 운영 WAR/JAR 파일을 `{파일명}_backup_{날짜시간}` 형식으로 백업한다. 신규 빌드를 배포하고 서버를 재기동한다. 기동 로그에서 `outlookAsyncExecutor` Bean 등록 확인 및 기존 기능 정상 동작(GETCONFIG, GETORGMAP 등)을 즉시 검증한다.

#### 2-5-3. 롤백 절차

아래 조건 중 하나라도 발생하면 즉시 롤백을 실행한다.

| 롤백 트리거 | 판단 기준 |
|----------|---------|
| 기존 기능 장애 | GETCONFIG, GETORGMAP 등 기존 type에서 오류 응답 발생 |
| Exchange 세션 전면 장애 | `ExchangeSessionManager` 재시도 3회 연속 실패가 모든 요청에서 발생 |
| Redis Lock 전면 잠금 | Lock 미해제로 전체 Outlook 요청이 오류 응답 반환 |
| Thread Pool 포화 | `outlookAsyncExecutor` Queue Full로 인한 `TaskRejectedException` 전파 |

롤백 실행: 백업해 둔 WAR/JAR로 즉시 교체 후 서버 재기동. Redis 신규 키(`lock:outlook:*`, `outlookAsync:*`, `outlookToken:*`)는 TTL로 자동 만료됨. 장애 원인 분석 후 재배포 계획을 PM에게 보고한다.
