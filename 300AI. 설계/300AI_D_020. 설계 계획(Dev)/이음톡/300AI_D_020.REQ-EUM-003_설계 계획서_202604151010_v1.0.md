## (2) 개발 설계 수행 내용

### 1. AI 및 개발자 의견
- **AI 의견**: 다국어 확장을 위해 하드코딩된 모든 문자열을 리소스 번들로 분리해야 합니다. 베트남어의 경우 특수 성조 기호 및 한국어 대비 긴 텍스트 길이를 고려하여 UI 레이아웃의 유연성(Flexbox 등)을 확보해야 합니다.
- **개발자 의견**: Spring의 `MessageSource`를 활용하고, 클라이언트(Web/App)에서는 유저 세션의 Locale 정보에 따라 언어 팩을 동적 로딩하도록 구현하겠습니다.

### 2. 매핑/GAP 분석 및 대응 방안
- **매핑/GAP 분석**: 현재 한국어/영어만 대응 가능하며, 베트남어(vi) 언어 리소스 및 폰트 설정이 부재함.
- **GAP 대응 방안**: `messages_vi.properties` 파일을 생성하고 전문 번역을 통해 텍스트를 구성함. UTF-8 인코딩을 준수하며 유니코드 폰트를 적용함.
- **분류**: 기능, 기타(UI/UX)
- **수용여부**: 수용

### 3. 기술 설계 상세
- **아키텍처**:
    -     - 구조: `LocaleContextHolder` -> `ResourceBundleMessageSource` -> `Client UI`.
    - 시퀀스: 유저 접속 -> Accept-Language 헤더 또는 유저 프로필 언어 설정 확인 -> 세션 Locale 설정 -> 다국어 텍스트 렌더링.
- **데이터 설계**:
    - `TB_USER_SETTING`에 `PREFERRED_LANGUAGE` (VARCHAR: ko, en, vi) 필드 관리.
- **인터페이스 (API Specification)**:
    - `GET /api/v1/i18n/bundle?lang=vi`: 특정 언어 전체 리소스 팩 수신.
- **구현가이드 (Pseudo Code)**:
    ```javascript
    // 클라이언트 측 다국어 처리 예시 (i18next 활용)
    const t = useTranslation();
    return <span>{t('COMMON_SAVE_SUCCESS')}</span>;
    ```
- **패키지 구조**: `src/main/resources/i18n/`
- **Entity 설계**:
    - 기존 설정 엔티티 내 `langCode` 컬럼 활용.
- **DTO/VO**: `LanguagePackResponse`
- **Service 로직**: `I18nService.getMessage(key, locale)`
- **외부 연동**: 베트남 현지 사용자를 위한 CDN(Content Delivery Network) 최적화 고려.
