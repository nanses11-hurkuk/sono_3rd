### 요구사항 (개요)

유티아이 베트남 그룹웨어 시스템 구축 프로젝트에서 Outlook Addin과 그룹웨어 간 연동 과정에서 식별된 총 **7건의 GAP(G-01 ~ G-07)** 을 해소하기 위해 상세 설계를 수행한다. 해당 GAP 항목들은 전원 **기능** 영역에 해당하며, 분석 완료 단계에서 모두 **"수용"** 판정을 받았다.

핵심 해소 과제는 Exchange 세션 재연결(`ExchangeSessionManager` 구현), OAuth2 기반 인증 전환(Kerberos 탈피), 배포 그룹 CRUD 스크립트 추가, 주소록 동기화 API 연동 확인, Redis Distributed Lock 적용, 비동기 처리 레이어 추가이다.

#### 1. 현황 및 배경

현행 Outlook Addin 연동 모듈(`egovframework.covimail.common`)은 단일 진입점 컨트롤러(`OutlookAddinCon`)와 서비스 구현체(`OutlookAddinSvcImpl`)로 구성되어 있으며, CAUTK 파라미터 기반 type 스위치(`GETCONFIG`, `GETBIGATTACH`, `GETORGCHART`, `GETCONTACTS` 등)로 기능을 분기한다.

```

[OutlookAddinCon.java]          ← 단일 진입 컨트롤러 (outlook/service.do)
    ↓ switch(type)
[OutlookAddinSvcImpl.java]      ← 서비스 구현체

    ├─ getOrgChart()            ← 조직도 연동 (outlook.addin.selectOrgChartUserList 등)
    ├─ getContacts()            ← 연락처 연동 [TODO 미구현 — "그룹 연락처 처리 및 쿼리 수정"]

```

#### 2. 식별된 GAP 목록 및 설계 방향

> **주의**: 분석 결과보고서(`200AI_C_030.REQ-ODD-001`) 원문을 기준으로 하며, 아래는 분석완료 확인서 및 소스코드 분석을 종합한 재구성 내용이다. 상세 설계 착수 전 원본 분석서와 반드시 대조·검증한다.

| GAP ID | GAP 설명 | 현행 문제 | 설계 방향 |
|--------|----------|----------|----------|
| **G-01** | Exchange 세션 재연결 로직 부재 | `createTempSession()` 호출 실패 시 "Can't Create Session" 단순 반환, 재시도 없음 | `ExchangeSessionManager` 클래스 신규 구현 — 세션 만료 감지 → 자동 재연결(Retry 3회, Exponential Backoff) → 실패 시 오류 응답 표준화 |
| **G-02** | OAuth2 인증 미적용 (Kerberos 사용) | CAUTK 파라미터 기반 임시 세션 키 방식, Kerberos 티켓 의존 — 베트남 환경 비적합 | 인증 방식 추상화 인터페이스(`OutlookAuthStrategy`) 설계. 구현체: `KerberosAuthStrategy`(기존), `OAuth2AuthStrategy`(신규, MSAL4J). 런타임 전략 교체 지원 |
| **G-03** | 배포 그룹(Distribution Group) 관리 스크립트 부족 | 조직도(`getOrgChart`)는 부서/사용자 조회만 구현. 배포 그룹 CRUD(생성/수정/삭제/멤버 관리) 미구현 | Exchange PowerShell 또는 Graph API(`/groups`) 기반 배포 그룹 CRUD 스크립트/API 추가 개발. `OutlookAddinSvc`에 `manageDistributionGroup()` 메서드 추가 |
| **G-04** | 주소록 동기화 API 연동 미확인 | `getContacts()` 에 `// TODO: 그룹 연락처 처리 및 쿼리 수정` 미완성. `selectBizCardAllList` 쿼리 존재하나 Outlook 주소록 동기화 연동 미검증 | Outlook 주소록(GAL/개인 연락처) ↔ 그룹웨어 연락처 양방향 동기화 API 인터페이스 설계. `selectBizCardAllList` 쿼리 검증 및 그룹 연락처 쿼리 완성 |
| **G-05** | Redis Distributed Lock 미적용 (동시성 이슈) | 복수 세션 동시 접근 시 `createTempSession()` 내 세션 중복 생성 가능성. 클러스터 환경에서 데이터 정합성 위험 | `RedisDataUtil` 기반 Distributed Lock(`SETNX` 또는 Redisson) 적용. 세션 생성, 연락처 동기화, 배포 그룹 수정 등 임계 구역(Critical Section) 보호 |
| **G-06** | 비동기 처리 레이어 미구현 | 조직도 조회(`getOrgChart`), 연락처 동기화(`getContacts`) 등 대량 데이터 처리 시 동기 방식으로 HTTP 타임아웃 위험 | `AsyncSvc`/`@Async` 패턴(`egovframework.covimail.common.service.AsyncSvc`) 참조하여 비동기 처리 레이어 추가. 결과는 폴링(Polling) 또는 WebSocket/SSE 방식으로 클라이언트에 전달 |
| **G-07** | 배포 그룹 포함 연락처 그룹 분류 미완성 | `getContacts()`의 `folderNameP/D/U`(개인/부서/회사) 분류는 있으나 배포 그룹(Distribution Group) 폴더 미포함. XML 응답 구조 불완전 | 연락처 응답 XML 구조에 배포 그룹 폴더(`folderNameG`) 추가. `selectBizCardAllList` 쿼리에서 `BizCardType='Group'` 항목 포함하여 완전한 주소록 구조 완성 |

#### 3. 아키텍처 설계 방향 (개요)

```
[설계 원칙]

1. ExchangeSessionManager 신규 구현
   ├─ 세션 풀(Session Pool) 관리
   ├─ 세션 만료 감지 (TTL 기반)
   └─ 자동 재연결 (Retry 3회, Exponential Backoff: 1s → 2s → 4s)

2. 인증 추상화 (Strategy Pattern)
   ├─ interface OutlookAuthStrategy
   │    └─ authenticate(CoviMap param): AuthToken
   ├─ KerberosAuthStrategy   ← 기존 유지 (하위 호환)
   └─ OAuth2AuthStrategy     ← 신규 (MSAL4J, ClientCredentials Flow)
        └─ 토큰 캐시: Redis ("outlookToken:{DomainId}:{userId}")

3. 배포 그룹 CRUD
   ├─ OutlookAddinSvc.manageDistributionGroup(type, param)
   ├─ type: CREATE / UPDATE / DELETE / ADD_MEMBER / REMOVE_MEMBER
   └─ Exchange PowerShell 또는 Graph API /groups 연동

4. Redis Distributed Lock 적용 범위
   ├─ createTempSession()       락 키: "lock:outlook:session:{mailAddress}"
   ├─ getContacts() 동기화      락 키: "lock:outlook:contacts:{userId}"
   └─ manageDistributionGroup() 락 키: "lock:outlook:distgroup:{groupId}"

5. 비동기 레이어
   ├─ @Async ("outlookAsyncExecutor") — AsyncConfig.java 별도 Executor 등록
   ├─ 처리 결과 → Redis 임시 저장 ("outlookAsync:{taskId}")
   └─ 클라이언트 폴링: GET /outlook/asyncResult.do?taskId={id}
```

## (2) 개발 설계 수행 범위

### 2-1. 난이도

| GAP ID | 난이도 | 사유 |
|--------|--------|------|
| G-01 (Exchange 세션 재연결) | **중(Medium)** | ExchangeSessionManager 클래스 신규 구현, Retry 로직 설계 필요 |
| G-02 (OAuth2 인증 전환) | **상(High)** | 인증 추상화 인터페이스 설계, MSAL4J 연동, 기존 Kerberos 하위호환 유지 필요 |

| G-04 (주소록 동기화 API) | **중(Medium)** | 기존 쿼리 완성 + Outlook GAL 연동 검증. 양방향 동기화 설계 |
| G-05 (Redis Distributed Lock) | **중(Medium)** | Redis SETNX 또는 Redisson 적용. 기존 RedisDataUtil 패턴 활용 |
| G-06 (비동기 처리 레이어) | **중(Medium)** | 기존 AsyncSvc 패턴 참조. Executor 등록 및 폴링 API 추가 |
| G-07 (연락처 그룹 분류) | **하(Low)** | SQL 쿼리 수정 + XML 응답 구조 보완. 기존 로직 확장 수준 |

> **전체 난이도**: **상(High)** — OAuth2 인증 전환 및 배포 그룹 CRUD 연동이 핵심 고난이도 항목이며, 아키텍처 수준의 인증 추상화 설계 포함

### 2-5. 설계 범위 지정

#### 포함 범위 (In-Scope)

1. **Exchange 세션 관리 설계**
   - `ExchangeSessionManager` 클래스 설계 (세션 풀, TTL, 재연결 Retry)
   - `OutlookAddinCon` 내 세션 오류 처리 표준화 (오류 코드/메시지 체계)

2. **인증 추상화 설계**
   - `OutlookAuthStrategy` 인터페이스 설계
   - `KerberosAuthStrategy` (기존 유지), `OAuth2AuthStrategy` (MSAL4J 기반) 구현 설계
   - Redis 토큰 캐시 키 구조 및 TTL 설계

3. **배포 그룹 CRUD 설계**
