### 2-1. AI 의견

**[요구사항 해석]**

"V1, V2간 채팅기능 구현"은 이음톡 메신저의 구버전(V1)과 신버전(V2)이 동일 서버 내 또는 별도 서버로 운영되는 환경에서, **두 버전의 사용자가 서로 채팅 메시지를 주고받을 수 있도록 상호운용성(Interoperability)을 제공**하는 요구사항이다. 이는 유티아이 베트남 법인의 이음톡 도입 시 기존 V1 사용 그룹과 신규 V2 도입 그룹이 혼재하는 전환(Migration) 기간을 고려한 요청으로 해석된다.

**[기술적 타당성 검토]**

| 검토 항목 | 기술적 판단 | 근거 |
|---------|-----------|------|
| 동일 서버 내 V1/V2 공존 | ✅ 가능 | 버전별 엔드포인트 분리(`/v1/`, `/v2/`) 또는 Feature Flag 패턴 적용 |
| 별도 서버 간 메시지 교환 | ✅ 가능 | 메시지 브로커(Kafka, RabbitMQ) 또는 Webhook 방식으로 서버 간 메시지 중계 |
| WebSocket 프로토콜 호환 | ⚠️ 검토 필요 | V1과 V2의 WebSocket 메시지 포맷(JSON 스키마)이 다를 경우 변환 레이어 필요 |
| 채팅 메시지 스키마 호환 | ⚠️ 검토 필요 | 메시지 ID 채계, 타임스탬프 형식, 첨부파일 참조 방식 등 버전 간 차이 확인 필요 |
| 사용자 ID 체계 통합 | ⚠️ 검토 필요 | V1과 V2의 사용자 ID(PK)가 동일하다면 단순 연동, 다르다면 매핑 테이블 필요 |
| 실시간성 보장 | ✅ 가능 | 메시지 브로커 기반 Pub/Sub 패턴으로 ms 단위 지연 내 메시지 전달 가능 |

**[설계 옵션]**

**옵션 A — 통합 메시지 버스 방식** (권장)
- Kafka 또는 Redis Pub/Sub로 V1/V2 서버 간 메시지 중계
- V1 → Topic 발행 → V2 구독(Consumer) → V2 사용자에게 전달
- 장점: 서버 간 직접 의존성 없음, 확장성 우수
- 단점: 메시지 브로커 인프라 추가 필요

**옵션 B — API Gateway Webhook 방식**
- V1 메시지 발송 시 Webhook으로 V2 서버에 POST 전달
- 장점: 구현 단순
- 단점: 동기 방식 → V2 서버 장애 시 V1 메시지 손실 위험

**AI 권장**: **옵션 A (메시지 브로커 방식)** — 안정성 및 확장성 우수

### 2-9. 구현 가이드 (Pseudo Code / Logic)

> 문서 ID: 300AI_D_030.REQ-EUM-002 | 설계 계획 ID: 300AI_D_020.REQ-EUM-002

```java
// [신규] MessageBridgeProducer.java (V2 서버)
// 메시지 전송 후 브릿지 대상 채팅방이면 브로커에 발행
public void sendToBridge(ChatMessage v2Message) {
    // 1. cross_version_room에서 V2 → V1 매핑 조회 (Redis 캐시 우선)
    String v1RoomId = roomMappingCache.getV1RoomId(v2Message.getRoomId());
    if (v1RoomId == null) return; // 브릿지 미설정 채팅방

    // 2. V2 사용자 → V1 사용자 ID 매핑
    String v1SenderId = userMappingCache.getV1UserId(v2Message.getSenderId());

    // 3. 공통 브릿지 포맷으로 변환
    BridgeMessageDTO bridgeMsg = converter.toV2toV1Format(v2Message, v1RoomId, v1SenderId);

    // 4. Redis Pub/Sub 또는 Kafka 발행
    messageBroker.publish("chat.bridge.v2-to-v1", bridgeMsg);

    // 5. 로그 기록
    bridgeLogRepo.save(new BridgeMessageLog(v2Message.getMsgId(), "V2", "V1", "PENDING"));
}

// [신규] MessageBridgeConsumer.java (V2 서버 — V1 메시지 수신)
@EventListener("chat.bridge.v1-to-v2")
public void onV1Message(BridgeMessageDTO bridgeMsg) {
    try {
        // 1. V1 sender → V2 sender 매핑
        String v2SenderId = userMappingCache.getV2UserId(bridgeMsg.getSenderId());

        // 2. 채팅방 ID 확인
        String v2RoomId = roomMappingCache.getV2RoomId(bridgeMsg.getSourceRoomId());

        // 3. V2 DB에 메시지 저장 (발신자: V1 브릿지 표시)
        ChatMessage v2Message = converter.fromBridgeFormat(bridgeMsg, v2RoomId, v2SenderId);
        v2Message.setBridgedFrom("V1");    // V1에서 수신된 메시지임을 표시
        chatMessageRepo.save(v2Message);

        // 4. V2 사용자에게 WebSocket Push
        webSocketBroadcaster.broadcast(v2RoomId, v2Message);

        // 5. 전달 성공 로그
        bridgeLogRepo.updateStatus(bridgeMsg.getBridgeMessageId(), "DELIVERED");
    } catch (Exception e) {
        bridgeLogRepo.updateStatus(bridgeMsg.getBridgeMessageId(), "FAILED");
        // 재시도 큐 등록 (Redis pending)
    }
}

// [신규] MessageBridgeConverter.java
public BridgeMessageDTO toV2toV1Format(ChatMessage v2Msg, String v1RoomId, String v1SenderId) {
    return BridgeMessageDTO.builder()
        .bridgeMessageId(UUID.randomUUID().toString())
        .sourceVersion("V2").sourceRoomId(v2Msg.getRoomId())
        .targetVersion("V1").targetRoomId(v1RoomId)
        .senderId(v2Msg.getSenderId()).senderName(v2Msg.getSenderName())
        .content(v2Msg.getContent()).contentType(v2Msg.getContentType())
        .sentAt(v2Msg.getSentAt())
        .build();
}
```
