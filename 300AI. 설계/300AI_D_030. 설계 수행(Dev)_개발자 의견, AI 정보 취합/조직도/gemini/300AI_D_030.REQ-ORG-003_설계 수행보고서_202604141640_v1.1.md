## (2) 개발 설계 수행 내용

### 1. AI 및 개발자 의견 취합
- **AI 의견**: `smarts4j.covicore`의 조직도 트리 컴포넌트인 `coviTree`를 확장하여 계열사 통합 뷰를 구현할 것을 제안함. `domainID` 파라미터를 동적으로 처리하여 전체 계열사(ALL) 또는 특정 계열사 트리 로딩이 가능하도록 `OrgApiCon`을 고도화함.
- **개발자 의견**: 통합 조직도 검색 시 동명이인 구분을 위해 부서명 앞에 회사명을 병기(예: [V1] 인사팀)하는 UX 가이드를 적용함. 대량 조회 시 성능 저하를 방지하기 위해 상위 부서 클릭 시 하위 노드를 로드하는 `Lazy Loading` 방식을 필수로 적용함.

### 2. 매핑/GAP 분석 및 대응
- **매핑/GAP 분석**: 기존 SaaS 아키텍처는 도메인 간 데이터 격리가 기본 정책이나, 이번 요구사항은 이를 상호 조회 가능하도록 완화해야 함.
- **GAP 대응 방안**: `sys_base_config`에 `IsGlobalOrgChart` 옵션을 추가하여, 해당 옵션이 활성화된 경우에만 도메인 필터를 우회하는 전용 Mapper Query를 개발함.
- **분류**: 기능
- **수용여부**: 수용

### 3. 상세 설계 내역

#### A. 아키텍처 (Architecture)
- **컴포넌트 구조**: 
    - `UnifiedOrgTree`: V1, V2를 통합하여 렌더링하는 UI 컴포넌트.
    - `OrgSvc`: 도메인 통합 쿼리를 지원하는 서비스 레이어.
    - `ScopeFilter`: 메뉴별 노출 범위를 제어하는 보안 필터.
- **시퀀스 다이어그램**:
    1. 사용자가 조직도 팝업 실행.
    2. `OrgApiCon.getUnifiedDeptList.do` 호출.
    3. `OrgSvc`가 전체 도메인 목록 조회 후 각 도메인의 Root 부서 취합.
    4. `coviTree`에 통합 JSON 데이터 전달 및 렌더링.

#### B. 데이터 설계 (Data Design)
- **조회 전략**: `sys_object_user`, `sys_object_dept` 테이블을 `domainID` 기준으로 그룹화하여 조회.
- **캐시 전략**: 통합 조직도 데이터를 Redis에 캐싱하여 반복 조회 시 DB 부하 최소화.

#### C. 인터페이스 (API Specification)
- **IF-COV-USR-290**: LDAP/통합 수신처 조직도 조회 인터페이스 패턴 재활용.
- **REST API**: `GET /covicore/api/org/getUnifiedList.do`
