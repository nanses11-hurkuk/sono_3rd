## (2) 개발 설계 수행 내용

### AI 의견

- **GW DB 스키마가 이미 다중 회사 구조:** `orgs_orgperson`과 `orgs_orgdept`의 PK가 `(UserCode, CompanyCode)` / `(GroupCode, CompanyCode)` 복합키로 설계되어 있습니다. V1·V2 데이터가 동일 테이블에 `CompanyCode`로 구분되어 공존하는 구조가 이미 적용되어 있습니다. 별도 통합 작업 불필요.
- **기존 API가 이미 CompanyCode 파라미터 지원:** `OrgApiCon`의 `org/user/list.do`, `org/dept/list.do`, `org/user/add.do` 등이 `CompanyCode`를 파라미터로 사용하고 있습니다. 통합 조회는 `CompanyCode`를 공백/전체로 설정하는 방식으로 기존 API를 재활용합니다.
- **getavailablecompanylist.do 재활용:** `OrgManageCon.getavailablecompanylist`가 사용 가능한 회사 목록을 반환합니다. 계열사 목록 조회에 바로 재활용 가능합니다.
- **조직도 공통 팝업(F-COR-SHR-001) 재활용:** `FUNCTIONAL_SPEC_CORE.md` 기준, `IF-COV-USR-243` 조직도 컨트롤(공통 팝업)이 이미 구현되어 있습니다. 전자결재·이음톡에서 사용하는 조직도 선택 팝업이 이 공통 컴포넌트를 사용하므로, 통합 조회(CompanyCode 전체 조건)만 적용하면 결재선 선택 시 타 계열사 임직원도 자동으로 노출됩니다.
- **모바일 조직도 API 재활용:** `MobileOrgCon.getSubList`, `getOrgInfo`, `getOrgPathInfo`가 모바일 환경의 조직도 조회를 담당합니다. 이음톡(모바일) 대화 상대 검색에 재활용 가능합니다.
- **계열사 간 정보 노출 정책 미확정:** 현재 조회 API가 전체 컬럼을 반환하는 구조이므로, 노출 속성 제한이 필요한 경우 API 응답 필터링 레이어를 추가해야 합니다. 정책 확정 전까지는 기본 조회 컬럼(`UserCode`, `DisplayName`, `DeptCode`, `JobPositionCode`, `PhotoPath`, `MailAddress`) 기준으로 적용합니다.
- **사용자 사진(F-COR-SHR-002) 재활용:** `FUNCTIONAL_SPEC_CORE.md` 기준, 사용자 사진 조회 공통 서비스(`IF-COV-USR-385`)가 이미 구현되어 있어 V1·V2 통합 조직도 UI에서 별도 개발 없이 프로필 사진 표시 가능합니다.
