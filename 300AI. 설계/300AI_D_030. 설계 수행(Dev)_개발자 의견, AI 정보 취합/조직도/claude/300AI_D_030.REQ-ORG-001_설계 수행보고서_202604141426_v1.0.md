## (2) 개발 설계 수행 내용

### AI 의견

- **기존 동기화 인프라 재활용 가능:** `smarts4j.covicore`에는 `OrgSyncCon`(인사 동기화 실행·모니터링·로그), `SysDatabaseSyncCon`(외부 DB 동기화) 컨트롤러가 이미 구현되어 있습니다. V2(영림원) 연동은 `SysDatabaseSyncCon.transferExtDatabase` 및 `OrgSyncCon.synchronize` 흐름을 재활용하는 방향으로 설계합니다.
- **HR 원천 테이블 확인 완료:** 영림원 MSSQL DB명 `UTICommon`, 테이블 `dbo.GW_Department` / `dbo.GW_User` (item/HR/HRConnectCode.java 확인). DB 접속 성공 검증 완료(item/HR/ConnectionResult.md).
- **GW DB 스키마 확인 완료:** 동기화 DB는 `covi_syncdata`. HR 스테이징 테이블(`orgs_hr_dept`, `orgs_hr_person`)→ GW 마스터(`orgs_orgdept`, `orgs_orgperson`)→ 임시/비교 테이블(`orgs_temp_*`, `orgs_compare_*`) 구조 이미 존재.
- **식별키 확정:** `item/HR/Data_Keys.md` 분석 결과, 사용자 식별키는 `EmpNo`(HR 원천 식별자, 200건 전체 유니크율 1.0). 부서 식별키는 `GroupCode`.
- **충돌 항목 사전 확인:** `item/SYNC/FieldMapping.sql` 분석 결과, `insertHRJobLevel`, `insertHRJobPosition`, `insertHRJobTitle`은 영림원이 데이터를 미제공하여 0건 → **직위/직책/직급은 자동 동기화 불가, GW 수동관리 확정**(인터뷰 합의와 일치).
- **관리자 UI 재활용:** `OrgManageCon`의 `insertUpdateUserInfo`, `insertDeptInfo`, `deleteUser`, `updateIsUseUser` 등 기존 API를 V1 수동관리에 그대로 사용합니다.
- **비밀번호 정책:** `OrgManageCon.usePWPolicyCheck`, `resetuserpassword` 기존 구현 활용. 초기 비밀번호(`password`) 세팅 시 회사 비밀번호 정책 체크 로직 우회 또는 관리자 권한 적용 필요.
- **사진 업로드:** `OrgManageCon.insertUpdateUserInfo`에 프로필 이미지 업로드가 통합 처리되어 있음. HR팀 전용 접근 권한 설정만 추가하면 됩니다.
- **퇴직 처리:** `updateIsUseUser`(`IsUse=N`)로 소프트 삭제. `RetireDate` 컬럼이 `orgs_orgperson`에 이미 존재.

### 구현 가이드 (Pseudo Code / Logic)

```
[V2 자동 동기화 흐름 - 기존 구현 활용]

1. Scheduler(매일 새벽 1시) → OrgSyncCon.organizationSynchronize() 호출
   - 또는 SysDatabaseSyncCon.transferExtDatabase() (비동기) 호출

2. createSyncData():
   - JDBC로 UTICommon.dbo.GW_Department 조회 → INSERT orgs_hr_dept
   - JDBC로 UTICommon.dbo.GW_User 조회 → INSERT orgs_hr_person
   - 식별키: dept=GroupCode, user=EmpNo (Data_Keys.md 확정)
   - INSERT 대상: 11개 dept 컬럼, 26개 person 컬럼 (FieldMapping.sql 기준)
   - JobLevel/Position/Title: HR 미제공 → 0건 처리, 오류 아님

3. synchronize():
   - orgs_hr_dept ↔ orgs_orgdept 비교 (orgs_compare_object_group 활용)
     → INSERT: 신규 부서
     → UPDATE: DisplayName, SortKey 등 변경
     → SOFT-DELETE: IsUse=N (HR에 없는 부서)
   - orgs_hr_person ↔ orgs_orgperson 비교 (orgs_compare_object_user 활용)
     → INSERT: 신규 사용자 (초기 비밀번호 정책 처리 포함)
     → UPDATE: DeptCode, DisplayName 등 변경
     → SOFT-DELETE: IsUse=N, RetireDate=오늘 (HR에 없는 사용자)

4. moniterSyncStatus() / getLogList():
   - 동기화 진행 건수 실시간 모니터링
   - 오류 발생 시 syncErrorMailToManager() 담당자 메일 발송

[V1 수동관리 - 기존 API 재활용]
- 사용자 등록/수정: OrgManageCon.insertUpdateUserInfo.do
  - 프로필 이미지 업로드 통합 처리 포함
  - ROLE_HR_ADMIN 권한 체크 추가 필요
- 부서 관리: OrgManageCon.insertDeptInfo.do / deleteDept.do
- 직위/직책/직급 수동관리: OrgManageCon.registarbitarygroup.do
- 퇴직 처리: OrgManageCon.updateIsUseUser.do (IsUse=N)
- 비밀번호 초기화: OrgManageCon.resetuserpassword.do
- 중복 체크: getIsDuplicateUserCode, getIsDuplicateEmpno, getIsDuplicateDeptCode
```
