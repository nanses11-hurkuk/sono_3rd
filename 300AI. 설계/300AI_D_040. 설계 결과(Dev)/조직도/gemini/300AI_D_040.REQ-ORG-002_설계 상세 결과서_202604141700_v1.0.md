## (2) 개발 최종 설계 내용

### 1. 아키텍처 (Architecture)
- **컴포넌트 구조**: 
    - **LdapManager (Component)**: `smarts4j.coviexchange` 내의 LDAP Context 관리 컴포넌트 (기존 재활용).
    - **ADSyncSvc (Business Layer)**: GW 인사 변경 이벤트를 수신하여 AD 속성을 동기화하는 서비스.
    - **ExchangeAPIBridge**: M365 라이선스 및 사서함 제어를 위한 API Proxy.
- **시퀀스 다이어그램**:
    1. 인사 동기화 성공 후 `ADSyncJob` 트리거.
    2. `ADSyncSvc` -> `LdapUserSvc.modifyUser()` 호출하여 Local AD 반영.
    3. AD 변경 확인 후 `ExchangeRestProxy` -> `/v1/org/user` 호출하여 M365 속성 갱신.
    4. AADC(Azure AD Connect)에 의해 M365(Entra ID) 최종 동기화 완료 대기.

### 2. 데이터 설계 (Data Design)
- **AD Mapping 명세**:
    - `sAMAccountName` <- `LogonID`
    - `userPrincipalName` <- `LogonID@uti.com`
    - `extensionAttribute1` <- `EmpNo`
- **저장 전략**: AD를 인증 Master로 활용하되, GW DB를 인사 Master로 하여 단방향 추종(Push) 방식 적용.

### 3. 인터페이스 (API Specification)
- **IF-EXCH-006 (Reuse)**:
    - **Endpoint**: `v1/org/user` (POST)
    - **Payload**: `MSEWSUserDTO` 객체.

### 4. 구현 가이드 (Pseudo Code/Logic)
```java
// ADSyncSvcImpl.java (coviexchange 모듈 활용)
public void pushToAD(UserDTO user) {
    MSEWSUserDTO ldapUser = converter.toMSEWSUserDTO(user);
    ldapUserSvc.modifyUser(ldapUser); // 기존 coviexchange 서비스 호출
    
    if(user.isMailUser()) {
        exchUserSvc.enableUser(ldapUser); // Exchange 사서함 활성화
    }
}
```
