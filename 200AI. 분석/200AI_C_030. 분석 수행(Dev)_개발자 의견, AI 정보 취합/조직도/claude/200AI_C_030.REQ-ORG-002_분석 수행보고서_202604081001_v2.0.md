### GAP 대응 방안

1. **[GAP-001] GW→AD→AADC→EntraID 흐름 명확화**: GW SYNC 배치의 AD 연동 범위는 Local AD Object(사용자 계정, 부서 그룹) 생성·수정·비활성화까지이다. AADC가 이를 Entra ID에 자동 전파하므로, GW에서 Microsoft Graph API를 직접 호출하는 로직은 불필요하다. 이 흐름을 명확히 설계 문서에 기재하여 개발 범위를 확정한다.

2. **[GAP-002] AD 통합 선행 조건 협의**: V1·V2 AD 통합은 인프라 팀이 담당한다. 개발팀은 인프라 팀과 AD 통합 완료 예정일을 협의하고, 완료 전에는 V1·V2 각각의 AD에 대해 독립적으로 동기화 배치를 구성하는 2단계 접근 방식을 제안한다.

3. **[GAP-003] AD 배포단위 확정 요청**: 배포단위(부서, 직위만 구성)가 UTI 내부 검토 중이다. 분석/설계 단계 완료 전(T+2주 이내)에 확정을 요청한다. 확정 전에는 Smarts4j 표준 AD 구조(배포단위: 부서/직위/직급/직책, 조직단위: 회사 OU → 부서 OU → 사용자)를 기준으로 설계한다.

4. **[GAP-004] 개발 테넌트 확보 공식 요청**: 개발 테넌트 없이는 Exchange Online 메일그룹 생성, AADC 동기화 결과 검증 등 핵심 테스트가 불가하다. PM을 통해 UTI 측에 개발 테넌트 제공을 공식 요청한다. 미확보 시 Microsoft 365 Developer Program 활용을 대안으로 제안한다.

5. **[GAP-005] 메일 도메인 전환 기간 대응**: utikorea.com → utivina.com 전환 기간 중 사용자가 두 도메인 계정을 함께 보유한다. GW에서는 `sys_object_user.MailAddress`를 utivina.com 기준으로 설정하고, ProxyAddresses에 utikorea.com 주소를 보조 주소로 유지하여 메일 수발신이 단절되지 않도록 한다.

6. **[GAP-006] Exchange 메일그룹 동기화**: GW SYNC 배치에서 부서/직위 생성 시 Exchange Online 메일그룹(부서코드@utivina.com, 직위코드@utivina.com)을 자동 생성하고, 삭제 시 비활성화(삭제 아님)하는 로직을 `smarts4j.coviexchange` 모듈에 추가한다.

#### 제약 사항

1. **AD 통합 미완료 (개발 착수 최대 블로커)**: V1·V2 AD 통합이 완료되어야 통합 기준 AD 동기화 개발 착수 가능. 통합 전에는 V1·V2 각각 독립 배치로 임시 구성.
2. **AD 배포단위 미확정**: 배포단위(부서, 직위만) UTI 내부 검토 중. 확정 전 AD Object 구조 구현 불가.
3. **개발 테넌트 없음**: Exchange Online 메일그룹, AADC 동기화 결과 검증 불가. Microsoft 365 Developer Program 활용 대안 검토.
4. **AADC Throttling/Latency**: AADC 동기화 주기(기본 30분)를 고려하면 GW 배치 완료 후 M365 반영까지 최대 30분 지연 발생 가능. 이를 운영 정책으로 공유해야 함.
5. **utikorea.com → utivina.com 전환 기간**: 두 도메인 계정 공존으로 GW MailAddress 일관성 확보 전략 필요.
6. **ProxyAddresses 관리**: 퇴사 시 사서함 비활성화하나 ProxyAddresses에서 기존 주소 제거하지 않음(회의록_Exch.md 항목 4). 동일 메일주소 재사용 방지 로직 필요.

## REQ-ORG-002 GAP 분석 (개발 + AI 의견 취합)

### 개발자 의견 (이윤정)

> 아래 내용은 분석 계획서 및 인터뷰 자료를 기반으로 한 개발자(이윤정)의 기술적 검토 의견입니다.

1. **이 요구사항의 핵심은 GW→AD까지**: M365 동기화의 실질적 GW 개발 범위는 Local AD Object를 올바르게 생성·수정·비활성화하는 것이다. AADC가 AD 변경 내용을 Entra ID에 자동 전파하므로 GW에서 Microsoft Graph API를 직접 호출하는 복잡한 로직은 불필요하다. 이 범위 명확화가 개발 복잡도를 크게 낮춘다.

2. **AD 통합이 모든 것의 전제 조건**: V1·V2 AD 통합이 완료되기 전에는 통합 기준의 설계를 완성할 수 없다. 인프라 팀의 AD 통합 완료 예정일을 조기에 확인하고, 지연 시 V1·V2 분리 배치로 우선 진행하는 방안을 PM에게 보고해야 한다.

3. **AD 배포단위 '부서, 직위만' 확정 시 개발 영향**: 현재 Smarts4j 표준은 부서/직위/직급/직책 4개를 배포단위로 사용한다. UTI에서 부서, 직위만 사용하기로 확정되면 직급·직책 배포단위 생성 로직을 생략할 수 있어 개발 공수가 줄어든다. 단, 향후 추가 가능성을 고려해 코드 구조는 유연하게 설계한다.

4. **개발 테넌트 확보는 PM 통해 즉시 요청 필요**: 개발 테넌트 없이는 Exchange Online 메일그룹 API 호출 자체를 테스트할 수 없다. Microsoft 365 Developer Program(무료 90일 개발 테넌트)을 대안으로 활용하되, 운영 도메인(utivina.com) 설정이 불가하므로 한계가 있다. PM을 통해 UTI 운영 테넌트와 별도의 개발 서브 테넌트 제공을 요청하는 것이 최선이다.

5. **utikorea.com → utivina.com 전환 일정 파악 필요**: 병행 기간 중 GW가 어느 도메인을 기준 MailAddress로 설정해야 하는지 명확한 기준이 없다. 전환 완료 일정과 병행 기간 정책을 고객에게 확인하고 GW 개발 기준을 정의해야 한다.

### AI 의견 취합

> 아래 내용은 분석 요청서, 인터뷰 자료, 분석 계획서를 종합하여 AI가 도출한 기술적 분석 의견입니다.

1. **GW → AD → AADC → Entra ID 흐름 검증 포인트**:
   - GW 배치에서 AD Object를 생성·수정한 후, AADC 동기화(기본 30분 주기) 이후 Entra ID에 올바르게 반영되었는지 검증하는 프로세스가 필요하다.
   - 검증 방법: Azure Portal 또는 Graph API의 `GET /users/{id}` 호출로 Entra ID 사용자 속성 확인. 개발 테넌트 확보 후 자동화 가능.

2. **AD Object 생성 시 주요 속성 설정 가이드**:
   - `cn`: 사용자 코드(식별키)
   - `sAMAccountName`: 사용자 LogonID (= 메일ID, 회의록_공통.md 항목 10 확인)
   - `userPrincipalName`: 사용자 LogonID@utivina.com
   - `memberOf`: 소속 부서·직위 그룹 DN
   - 퇴사 시: `userAccountControl` 비활성화(0x0002) + 퇴직부서 OU로 이동

3. **Exchange 메일그룹 생성 권장 방식**:
   - 그룹 타입: Distribution Group (메일 수발신 목적)
   - 주소 형식: 부서코드@utivina.com, 직위코드@utivina.com
   - 외부 도메인 메일 수신 여부(회의록_Exch.md 항목 2, 3): 미확정 → 고객 확인 필요
   - 삭제 정책: 부서/직위 삭제 시 Exchange 그룹 즉시 삭제가 아닌 비활성화 처리 권장(메일 유실 방지)

4. **M365 라이선스 수동관리 유지 확인**:
   - 회의록_공통.md 항목 3에서 "M365에서 라이선스 수동관리>tobe에서도 프로세스 동일" 명시.
   - GW에서 M365_IsUse 필드를 표시는 하되, 자동 라이선스 부여/제거 로직은 개발하지 않음.
   - 관리자 화면에서 M365_IsUse 수동 변경 기능만 제공하는 것으로 범위 확정.

5. **미확정 항목 추적 테이블**:

### 종합 결론

REQ-ORG-002는 **수용 가능**하나, AD 통합, AD 배포단위 확정, 개발 테넌트 확보 등 3개의 외부 선행 조건이 동시에 미충족 상태이다. GW 개발 범위는 Local AD Object 관리까지이며, AADC 이후 Entra ID 반영은 AADC에 위임한다는 원칙을 명확히 하면 개발 복잡도를 크게 낮출 수 있다. PM을 통해 선행 조건 해소 일정을 즉시 확인하고, 지연 시 V1·V2 분리 배치로 단계적 구현하는 계획을 수립할 것을 권고한다.
