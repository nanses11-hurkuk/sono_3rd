### GAP 대응 방안

1. **[GAP-001] SaaS 멀티 도메인 가시성 설정 활용**: Smarts4j의 `sys_object_domain` 테이블 및 `DomainID=0`(그룹사) 기반 통합 조직도 트리 구성 기능을 활용한다. `selectInitOrgTreeList` 쿼리에 도메인 필터 확장을 적용하여 통합 조회 파라미터 전달 시 V1·V2 통합 트리를 반환하도록 한다.

2. **[GAP-002] isSaaS 파라미터 적용**: SYNC FieldMapping.sql의 `insertTempObjectDept`, `insertTempObjectUser` 등에서 `isSaaS=Y` 설정 시 GroupCode에 `CompanyCode_` 접두어를 붙여 V1·V2 간 코드 충돌을 방지한다. 단, 현재 FieldMapping.sql에서 QUERY 타입 sourceField 내 `#{_parameter[isSaaS]}` 표현식이 JDBC 파라미터 바인딩 오류를 유발하는 버그가 확인되었다. 이는 Java 서비스 단에서 사전 치환 로직을 추가하여 해결한다.

3. **[GAP-003] 전자결재 조직도 팝업 확장**: 결재선·참조 선택 팝업 호출 시 `isDisplayAllDomain=true` 또는 동등한 파라미터를 추가하여 V1·V2 통합 조직도를 표시한다. 단, 결재 권한(ACL)은 각 도메인 정책을 유지하여 타 도메인 사용자를 참조에는 추가할 수 있으나 권한 외 기안은 제한하는 방향으로 설계한다.

4. **[GAP-004] 이음톡·포탈 연동 API 확장**: 이음톡과 포탈의 조직도 조회 API에 멀티 도메인 파라미터를 추가하여 V1·V2 통합 데이터를 반환하도록 확장한다.

5. **[GAP-005] 계열사 간 정보 노출 범위 협의**: 타 계열사 사용자의 전화번호·메일·사진 등 민감 정보 노출 범위를 고객과 협의하여 `sys_object_domain` 설정 또는 ACL 정책으로 반영한다. 협의 전까지는 '읽기 허용(기본 정보만)' 기본값으로 개발한다.

6. **[GAP-006] 양사 조직도 UI 구분 표시**: 통합 조직도 트리에서 V1·V2를 시각적으로 구분할 수 있도록 트리 최상위에 회사명 노드를 표시하는 UI/UX 설계를 가이드에 포함한다.

#### 제약 사항

1. **`isSaaS` 파라미터 JDBC 버그 (즉시 수정 필요)**: 현재 FieldMapping.sql의 MemberOf, GroupCode QUERY 타입 sourceField 내 `#{_parameter[isSaaS]}` 표현식이 MariaDB Connector/J의 `INSERT INTO ... SELECT` 구문에서 다중 파라미터 바인딩 오류(`SQLSyntaxErrorException: Could not set parameter at position 2, values was 'N'`)를 유발한다. 이 버그가 수정되지 않으면 `isSaaS=N` 환경에서 SYNC 배치가 실패한다. Java 서비스 단 사전 치환 로직으로 해결 완료 필요.

2. **FieldMapping.sql CONSTANT 타입 빈 문자열 버그 (수정 완료)**: GroupPath, OUPath, SortPath 등 CONSTANT 타입에서 빈 문자열 값이 `'\''`(단일 따옴표)로 잘못 저장되어 SQL 구조가 깨지는 현상. 수정 완료(`'\'\''`로 교정)되었으나 DB 재삽입 및 재검증 필요.

3. **계열사 간 정보 노출 범위 미확정**: 타 계열사 사용자의 전화번호·메일·사진 노출 범위를 고객과 협의해야 한다. 미확정 시 '기본 정보(이름, 소속부서, 직위)만 노출'을 기본값으로 임시 구현한다.

4. **통합 조직도 성능 저하 가능성**: V1+V2 통합 조직도 로드 시 단일 도메인 대비 데이터량이 2배 증가할 수 있다. 초기 로드 시 레벨 1~2만 반환하고 하위는 Lazy Loading으로 처리해야 한다.

5. **이음톡 모듈 수정 범위 미파악**: `AI_Folder\smarts4j.covicore` 폴더 미존재로 이음톡 연동 모듈의 코드 레벨 분석 불가. 수정 범위는 모듈 코드 확인 후 재산정 필요.

## REQ-ORG-003 GAP 분석 (개발 + AI 의견 취합)

### 개발자 의견 (이윤정)

> 아래 내용은 분석 계획서 및 인터뷰 자료를 기반으로 한 개발자(이윤정)의 기술적 검토 의견입니다.

1. **`isSaaS` JDBC 버그가 현재 가장 급한 이슈**: REQ-ORG-003 구현의 기반은 SYNC 배치에서 `isSaaS=Y`로 GroupCode에 `CompanyCode_` 접두어를 올바르게 붙이는 것이다. 그런데 현재 FieldMapping.sql의 MemberOf·GroupCode QUERY 타입 sourceField 내 `#{_parameter[isSaaS]}` 가 JDBC 오류를 유발한다. 이 버그를 먼저 수정하지 않으면 isSaaS 관련 기능 전체가 동작하지 않는다. Java 서비스 단에서 FieldMapping 목록 조회 후 QUERY 타입 sourceField의 `#{_parameter[xxx]}` 패턴을 실제 파라미터 값으로 치환하는 전처리 메서드를 추가해야 한다.

2. **ACL 정책은 조기 협의 필수**: 타 계열사 사용자의 어떤 정보를 어느 범위까지 보여줄지는 고객의 보안 정책 결정 사항이다. 개발팀이 임의로 결정할 수 없으므로 PM을 통해 조기에 협의를 요청한다. 협의 전까지는 이름·소속부서·직위 등 기본 정보만 노출하는 것을 기본값으로 설계한다.

3. **V1·V2 UI 구분 표시 방안 제안**: 통합 조직도 트리에서 양사 조직도가 혼재할 경우 사용자 혼선이 발생할 수 있다. 트리 최상위에 '(V1) UTI Vietnam' / '(V2) UTI Korea Branch' 와 같이 회사명 노드를 별도로 구성하거나, 사용자 명함에 소속 회사 배지(Badge)를 표시하는 방안을 UI/UX 담당자와 협의할 것을 제안한다.

4. **Redis 캐시 정책 재설계 필요**: 현재 조직도 캐시가 단일 도메인 기준이라면, V1+V2 통합 캐시 키를 별도로 설계해야 한다. V1만 변경되어도 V1+V2 통합 캐시를 Invalidate해야 하므로, 캐시 키 구조를 `ORG_TREE_{DomainID}` 뿐 아니라 `ORG_TREE_ALL`도 관리하도록 수정이 필요하다.

5. **이음톡 모듈 코드 확인 요청**: `AI_Folder\smarts4j.covicore` 자료가 없어 이음톡 연동 API의 코드 레벨 분석이 불가하다. 이음톡 모듈의 조직도 API 코드를 확인한 후 수정 범위를 재산정하고 공수를 조정할 것을 요청한다.

### AI 의견 취합

> 아래 내용은 분석 요청서, 인터뷰 자료, 분석 계획서를 종합하여 AI가 도출한 기술적 분석 의견입니다.

1. **`isSaaS` JDBC 이슈 즉시 수정 코드 가이드**:
   - 수정 위치: `OrgSyncService` 또는 `OrgSyncCon`의 FieldMapping 조회 후 배치 실행 전
   - 수정 방법:
     ```java
     // isSaaS 값을 실제 SQL 리터럴로 치환
     String isSaaSValue = (String) paramMap.get("isSaaS");
     for (OrgSyncFieldMapping mapping : fieldMappingList) {
         if ("QUERY".equals(mapping.getSourceType())) {
             mapping.setSourceField(
                 mapping.getSourceField()
                     .replaceAll("#\\{_parameter\\[\\s*isSaaS\\s*\\]\\s*\\}", "'" + isSaaSValue + "'")
             );
         }
     }
     ```
   - 이 방식으로 JDBC `?` 파라미터 대신 SQL 리터럴로 처리되어 MariaDB 바인딩 오류 해소.

2. **`sys_object_domain` 설정 기반 멀티 도메인 가시성 구현 방안**:
   - `DomainID=0` 설정이 존재한다면 그룹사(최상위) 도메인으로 활용하여 V1·V2를 하위 도메인으로 구성.
   - `selectInitOrgTreeList` 쿼리에 `@domainScope = 'ALL'` 파라미터 추가 시 `sys_object_domain.DomainID IN (SELECT DomainID FROM sys_object_domain WHERE ParentDomainID = 0)` 형태로 확장.
   - `AI_Folder` 자료 미제공으로 실제 쿼리 구조 확인 불가. 자료 제공 후 정확한 구현 방안 도출 필요.

3. **GroupCode 충돌 방지 전략 검증**:
   - `isSaaS=Y` 적용 전: V1의 `DEPT001`, V2의 `DEPT001` → 충돌
   - `isSaaS=Y` 적용 후: V1의 `V1_DEPT001`, V2의 `V2_DEPT001` → 충돌 없음
   - MemberOf 값도 동일하게 처리됨: `CASE WHEN MemberOf = CompanyCode THEN MemberOf ELSE CONCAT(CompanyCode + '_', MemberOf) END`
   - 이 로직은 FieldMapping.sql에서 QUERY 타입으로 이미 정의되어 있음. `isSaaS` JDBC 버그 수정 후 정상 동작 예상.

4. **Redis 캐시 키 설계 권장안**:
   - 단일 도메인 캐시: `ORG_TREE:{DomainID}:{Level}` 형식
   - 멀티 도메인 통합 캐시: `ORG_TREE:ALL:{Level}` 형식
   - Invalidation: V1 또는 V2 SYNC 완료 시 `ORG_TREE:V1:*`, `ORG_TREE:V2:*`, `ORG_TREE:ALL:*` 모두 Flush
