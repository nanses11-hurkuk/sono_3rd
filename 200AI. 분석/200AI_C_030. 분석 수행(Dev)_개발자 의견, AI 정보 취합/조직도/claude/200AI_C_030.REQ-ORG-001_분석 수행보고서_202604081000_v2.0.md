### GAP 대응 방안

1. **[GAP-001] 이기종 DB 연동 설계**: Smarts4j의 표준 DataSource 확장 패턴을 사용하여 `applicationContext-mssql.xml`에 MSSQL 전용 DataSource Bean을 별도 등록한다. MSSQL용 MyBatis SqlSessionFactory도 분리하여 기존 MariaDB 연동에 영향을 주지 않는다. MSSQL JDBC Driver(`mssql-jdbc-12.x.jar`)를 WAS lib에 추가한다.

2. **[GAP-002] V1/V2 분기 처리**: `OrgSyncCon` 배치 진입점에서 `CompanyCode` 기준으로 V1(수동), V2(자동 연동) 분기를 구현한다. V1은 동기화 배치 대상에서 제외하며, V2만 영림원 View 연동을 수행한다.

3. **[GAP-003] SYNC FieldMapping 흐름 활용**: 표준 `SYNC_FieldMapping.sql`의 아래 흐름을 준수한다.
   - 1단계: `insertHRPerson`, `insertHRDept` (영림원 View → HR 중간 테이블, required)
   - 2단계: `insertOrgPerson`, `insertOrgDept` (HR → Org 테이블)
   - 3단계: `insertTempObject*` (Org → Temp 비교 테이블)
   - 4단계: `insertCompareObject*` (Temp 기준 INSERT/UPDATE/DELETE 비교)
   - 5단계: `sys_object_user`, `sys_object_group` 최종 반영

4. **[GAP-004] FieldMapping.sql 버그 수정**: 현재 `SYNC\FieldMapping.sql`에서 CONSTANT 타입의 빈 문자열 값이 `'\''`(단일 따옴표)로 잘못 저장되어 있어 SQL 구조가 깨지는 현상을 확인하였다. 이를 `'\'\''`(SQL 빈 문자열 리터럴)로 수정 완료하였다. 또한 QUERY 타입 sourceField 내 `#{_parameter[isSaaS]}` 표현식이 JDBC 다중 파라미터 바인딩 오류(`SQLSyntaxErrorException: Could not set parameter at position N`)를 일으키는 것을 확인하였다. 이는 Java 서비스 단에서 사전 치환 로직을 추가하여 해결한다.

5. **[GAP-005] 사용자 식별키 확정 대기**: UTI 내부 확인이 완료될 때까지 개발 착수를 보류한다. 확정 후 `UserCode` 또는 `EmpNo` 기준으로 DB 스키마 및 FieldMapping 매핑을 결정한다.

#### 제약 사항

1. **영림원 View 스키마 미확정 (개발 착수 최대 블로커)**: View 컬럼 목록, 데이터 타입, 인코딩 정보 없이는 `insertHRPerson`, `insertHRDept` FieldMapping 작성 불가. 영림원 측에 `orgs_hr_person`, `orgs_hr_dept` 스키마 문서를 제공하고 동일 컬럼 구조의 View 제공을 요청한다.
2. **사용자 식별키 미확정 (개발 착수 최대 블로커)**: UserCode vs EmpNo 결정이 DB Primary Key, FieldMapping Key 설계 전반에 영향을 미친다. UTI 담당자의 공식 답변 전까지 해당 영역 개발 착수 불가.
3. **개발 환경 MSSQL 접근 불가**: 방화벽 오픈 및 테스트 계정이 없으면 개발 단계 통합 테스트 불가. 인프라 팀에 개발서버 → 영림원 테스트 DB 경로 오픈 요청 필요.
4. **베트남 현지 네트워크 Latency**: GW WAS ↔ 영림원 MSSQL 간 실측 Latency 미확인. Connection Timeout 설정값은 현지 실측 후 조정 필요.
5. **FieldMapping.sql 버그 (수정 완료, 재배포 필요)**: `ELSE ''` 이슈 및 QUERY 타입 JDBC 이슈 수정 완료. 수정본 배포 및 DB 재삽입 필요.

## REQ-ORG-001 GAP 분석 (개발 + AI 의견 취합)

### 개발자 의견 (이윤정)

> 아래 내용은 분석 계획서 및 인터뷰 자료를 기반으로 한 개발자(이윤정)의 기술적 검토 의견입니다.

1. **영림원 View 스키마 확보가 최우선**: 현재 분석 단계에서 가장 큰 블로커는 영림원 MSSQL View의 컬럼 정의가 없다는 점이다. Smarts4j `orgs_hr_person`, `orgs_hr_dept` 스키마 문서를 영림원 측에 전달하고, View 컬럼이 동일 구조로 제공되도록 요청하는 것이 가장 빠른 진행 방법이다.

2. **사용자 식별키는 EmpNo로 임시 설계 후 확정 대기**: 회의록에서 LogonID=메일ID임을 확인하였으나, UserCode(PK)가 사번인지 다른 값인지는 미확정이다. 현재 `orgs_hr_person` 스키마에서 EmpNo가 Key로 정의되어 있으므로, 임시적으로 EmpNo 기반으로 설계하고 확정 시 조정하는 방식을 제안한다.

3. **FieldMapping.sql JDBC 이슈는 즉시 수정 필요**: 현재 MemberOf, GroupCode QUERY 타입 sourceField 내 `#{_parameter[isSaaS]}` 표현식이 `INSERT INTO ... SELECT` 구문에서 MariaDB JDBC 다중 파라미터 바인딩 오류를 발생시킨다. `OrgSyncService` 단에서 FieldMapping 조회 후 QUERY 타입 sourceField의 `#{}` 패턴을 실제값으로 치환하는 전처리 메서드를 추가해야 한다. 이 버그가 미수정 상태이면 isSaaS=N 환경(현재 대부분의 테스트 환경)에서 배치가 실패한다.

4. **V1 수동 관리 UI 확인 필요**: V1 계열사의 경우 자동 연동이 없으므로, 관리자가 사용자/부서 데이터를 수동으로 등록/수정/삭제하는 화면이 그룹웨어에 이미 있는지 확인이 필요하다. 없다면 해당 UI 개발 공수를 별도로 산정해야 한다.

5. **초기 비밀번호 `password` 보안 검토**: 오픈 시 전 사용자에게 동일 비밀번호를 부여하는 것은 보안상 위험하다. 최초 로그인 시 비밀번호 변경을 강제하는 정책 적용을 고객에게 제안할 것을 권장한다.

### AI 의견 취합

> 아래 내용은 분석 요청서, 인터뷰 자료, 분석 계획서를 종합하여 AI가 도출한 기술적 분석 의견입니다.

1. **MSSQL-MariaDB 연동 기술 스택 권장안**:
   - JDBC Driver: `mssql-jdbc-12.4.2.jre11.jar` (Microsoft 공식, JTDS 비권장)
   - Connection Pool: HikariCP 설정 (`maximumPoolSize=5`, `connectionTimeout=30000ms`, `idleTimeout=600000ms`)
   - MSSQL Timezone: UTC 설정 후 GW 단에서 UTC+7(Asia/Ho_Chi_Minh) 변환 처리
   - 인코딩: `sendStringParametersAsUnicode=true` JDBC URL 파라미터 추가(MSSQL nvarchar 대응)

2. **Truncate-and-Load vs UPSERT 전략 평가**:
   - `orgs_hr_*` 중간 테이블은 **Truncate-and-Load** 방식이 안전하다. 이유: 외부 ERP View는 전체 현재 상태를 항상 제공하므로, 누락/갱신 판단 없이 전체 교체가 정합성 확보에 유리하다.
   - `orgs_orgperson`, `orgs_orgdept`는 **UPSERT** 방식이 적합하다. 이유: 히스토리 및 GW에서 직접 추가한 정보(직위/직책 수동 등록 등)가 포함될 수 있으므로 덮어쓰기 위험을 최소화해야 한다.

3. **Smarts4j SYNC 아키텍처 상 주의점**:
   - `insertTempObject*` → `insertCompareObject*` 흐름에서 `isSaaS=Y`일 때 GroupCode에 `CompanyCode_` 접두어가 붙는다. 이로 인해 동일 배치를 isSaaS 값 변경 없이 재수행하면 GroupCode 키가 달라져 데이터 중복이 발생할 수 있다. 배치 파라미터 `isSaaS` 값의 일관성을 배치 수행 전 검증하는 로직이 필요하다.
   - `insertHRPerson`, `insertHRDept`는 `FieldMapping_Info.md` 기준 **required** 액션이다. 해당 액션의 FieldMapping 행이 없으면 SYNC 배치 전체가 실행되지 않을 수 있다. 영림원 View 스키마 확정 후 즉시 해당 매핑을 작성해야 한다.

4. **DELETE 안전 임계값 구현 방안**:
   - `orgs_compare_object_user`에서 DELETE 건수가 10명(1일 최대 퇴사자 수)을 초과하면 배치를 중단하고 관리자에게 알림을 전송한다.
   - 구현: `SELECT COUNT(*) FROM orgs_compare_object_user WHERE action='DELETE' AND CompanyCode='V2'` → 10 초과 시 예외 발생 및 배치 롤백.

5. **미확정 항목 추적 테이블**:
