#### DB 변경점

- `form_entInfo.xml`의 `selectEntInfoList` 쿼리에서 `sys_object_domain.DomainCode` 체계가 이미 존재하므로 V1, V2 도메인 코드 등록이 필요합니다. `assignedDomain` 파라미터로 도메인 접근 권한이 제어되고 있어, V1↔V2 간 상호 도메인 접근 권한 설정을 추가해야 합니다.
- `admin_affiliatesAggregateInformation.xml`에서 `jwf_workitem`과 `jwf_process` 테이블을 `CompanyCode`로 통계 처리하고 있으므로, 사간결재 문서의 `CompanyCode` 기록 정합성 확인이 필요합니다.
- 사간결재 라우팅 테이블 및 권한 매핑 테이블 구성을 검토해야 합니다.

#### 인프라 변경점

- 베트남 현지 네트워크 환경에서 V1↔V2 간 결재 데이터 전송 시 지연(Latency) 최소화를 위한 캐싱 전략이 필요합니다.
- `FormCon.java`에서 `RedisDataUtil`을 사용하고 있으므로, 결재선 정보 캐싱에 동일한 Redis Cache 방식을 적용할 수 있습니다.
- 인터뷰 자료에서 알림 매체 및 알림 Event를 사용하는 것으로 확인되었으므로, 사간결재 시 알림 발송 대상이 타 계열사 사용자로 확장될 때 `sendMessage.do`의 도메인 간 발송 처리가 정상 동작해야 합니다.

### 3-2. 기술적 타당성

#### 구현 난이도

- 소스 분석 결과, `EntInfoSvc.getEntinfoListData()` 및 `AffiliatesAggregateInformationCon.java`에서 이미 계열사 구분 체계(`CompanyCode`, `DomainCode`)가 구현되어 있습니다.
- `AutoApprovalLineService.setFormAutoApprovalData()`에서 자동결재선 설정 시 `strSessionDNCode`(도메인 코드) 파라미터를 이미 받고 있어, 사간결재 확장이 가능한 구조입니다.
- 인터뷰 자료에서 확인된 결재 프로세스(보고/품의, 신청, 배포) 중 보고/품의의 모든 서브 프로세스(개인협조, 부서협조, 참조, 회람 등)를 사간결재에서도 지원해야 하므로, `AutoApprovalLineService`의 자동결재선 처리(CCAfter, CCBefore, Approval, Agree 등)에 계열사 간 라우팅 분기가 추가되어야 합니다.
- 구현 난이도는 **중간** 수준으로 판단됩니다.

#### 제약 사항

- 계열사 간 결재 데이터의 가시성(Visibility) 및 권한(Authorization) 처리에 대한 명확한 정책 수립이 선행되어야 합니다.
- `form_entInfo.xml`의 `assignedDomain` 파라미터로 도메인 접근 권한이 제어되고 있으므로, V1↔V2 간 상호 도메인 접근 정책 설정이 필요합니다.
- V1, V2 간 조직도 데이터의 실시간 동기화 또는 배치 동기화 방식에 대한 결정이 필요합니다.
- 인터뷰 자료에서 기밀 문서 기능을 사용하는 것으로 확인되었으므로, 사간결재 시 기밀 등급 문서의 계열사 간 열람 권한 정책을 별도 수립해야 합니다.

#### 대안 제시

- 기존 `EntInfoSvc` 및 `DomainCode` 체계를 활용하여 사간결재 표준 기능을 활성화하고, 계열사별 결재선/양식 분리 전략을 적용하는 것이 코어 수정 최소화 관점에서 가장 효율적입니다.
- 성능 이슈 발생 시 결재선 캐싱(Redis) 및 비동기 알림(`MessagingQueueManager`) 처리를 추가 적용할 수 있습니다.
- 인터뷰 자료에서 문서번호 발번 규칙이 "부서명-YYYY-일련번호(4자리)"로 확인되었으므로, 사간결재 시 문서번호 체계에 계열사 구분이 필요한지 고객사 확인이 필요합니다.

### 3-6. 데이터 정합성 및 트랜잭션 전략

- `ApvProcessSvc.doCreateInstance()` 메서드에서 결재 인스턴스 생성 시 트랜잭션이 관리되고 있으며, 사간결재 시에도 동일한 트랜잭션 경계 내에서 원자성(Atomicity)을 보장해야 합니다.
- `admin_affiliatesAggregateInformation.xml`에서 `jwf_process` 테이블의 `BusinessState`, `ProcessState`로 결재 상태를 관리하고 있으므로, 분산 환경에서 V1↔V2 간 결재 문서 이동 시 상태값의 일관성(Consistency) 유지를 위한 보상 트랜잭션(Compensating Transaction) 전략을 수립해야 합니다.
- 인터뷰에서 승인취소, 기안취소/회수 기능을 사용하는 것으로 확인되었으므로, 사간결재 문서의 취소/회수 시 양 계열사 DB의 상태값 동기화 정합성이 보장되어야 합니다.
- 동시 결재 처리(일괄결재/연속결재) 시 낙관적 잠금(Optimistic Lock) 또는 비관적 잠금(Pessimistic Lock) 방식의 적용을 검토해야 합니다.
