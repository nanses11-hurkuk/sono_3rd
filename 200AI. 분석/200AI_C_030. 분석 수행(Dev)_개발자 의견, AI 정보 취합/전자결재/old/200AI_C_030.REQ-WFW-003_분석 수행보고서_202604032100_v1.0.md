#### DB 변경점

- `form_entInfo.xml`에서 `covi_smart4j.Fn_BaseGetDictionary_S(#{lang}, dn.MultiDisplayName)` DB 함수로 다국어 변환을 수행하고 있으므로, `sys_base_dictionary` 테이블에 베트남어(vi-VN) 언어 코드 및 번역 데이터 등록이 필요합니다.
- 인터뷰에서 확인된 전자결재 고유 용어의 베트남어 번역 데이터가 필요합니다:

  - 서브 프로세스: 개인협조, 부서협조, 참조, 사전참조, 사후참조, 회람
  - 결재 기능: 서명표시, 결재암호, 지정반려, 기안취소, 회수, 승인취소, 내용수정, 일괄결재, 1인결재, 기밀
  - 문서 속성: 작성자, 작성부서, 작성일자, 보존년한, 보안등급, 문서분류, 문서번호, 제목
  - 배포: 수신자, 수신부서

#### 인프라 변경점

- TECHNICAL_GUIDE_COMMON_SERVICES.md에 명시된 대로 시스템 구동 시 `sys_base_dictionary` 데이터를 Redis에 캐싱하고 있으므로, 베트남어 사전 데이터 추가 시 Redis Cache 용량 및 갱신 전략 검토가 필요합니다.
- 다국어 사전 데이터 갱신 시 Redis 캐시 무효화(Invalidation) 전략을 수립해야 합니다.

### 3-2. 기술적 타당성

#### 구현 난이도

- 소스 분석 결과, `DicHelper`를 통한 다국어 변환 체계가 전자결재 모듈 전반에 이미 적용되어 있습니다(`FormCon.java`, `ApvProcessCon.java`, `XFormEditorCon.java`, `AutoApprovalLineService.java` 등 다수 클래스에서 import 확인).
- `form_entInfo.xml`의 `Fn_BaseGetDictionary_S()` DB 함수가 `#{lang}` 파라미터를 받아 동적으로 다국어 변환을 수행하는 구조이므로, 베트남어 언어 팩 추가 및 사전 데이터 등록 방식으로 구현이 가능합니다.
- 인터뷰 자료에서 확인된 결재 용어/기능/문서 속성의 전체 목록을 기반으로 번역 대상 항목을 구체적으로 파악할 수 있어, 번역 범위 산정이 용이합니다.
- 구현 난이도는 **중간** 수준이나, 번역 데이터 확보 및 고객사 검수 기간이 일정에 영향을 미칩니다.

#### 제약 사항

- 요구사항정의서 원본에 "양식 포멧의 언어는 가능하나 본문내용의 언어는 변환 불가함"으로 명시되어 있으므로, 사용자가 작성한 본문 내용의 자동 번역은 불가합니다. 이 사항은 고객사에 사전 안내가 필요합니다.
- 베트남어 번역 데이터의 정확성 검증을 위해 고객사(현지) 확인 기간이 필요합니다.
- `AutoApprovalLineService.java`에서 `SessionHelper.getSession("lang")` 값이 클래스 레벨 필드로 선언되어 있어, Bean 스코프에 따라 세션 언어 변경 시 동적 반영 안정성을 확인해야 합니다. (`@Service` 어노테이션으로 Singleton 스코프일 경우 이슈 가능성 있음)
- 인터뷰에서 확인된 결재 용어 중 "신청" 유형의 담당자/담당부서/담당업무함 관련 항목이 "검토 후 결정" 상태이므로, 해당 항목의 다국어 번역도 확정 대기 상태입니다.

#### 대안 제시

- 1단계: 기존 `DicHelper` / `Fn_BaseGetDictionary_S()` 체계에 베트남어 언어 코드 추가 및 시스템 공통 영역(메뉴, 버튼, 상태값) 다국어 적용
- 2단계: 전자결재 양식별 필드명/안내문구 다국어 적용 및 고객사 검수
- 인터뷰에서 확인된 결재 용어/기능 목록을 베트남어 번역 대상 목록으로 확정하여 고객사에 번역 요청
- 본문 내용 번역이 향후 요구될 경우, 외부 번역 API 연동을 별도 요구사항으로 관리하는 것을 권장합니다.

- **양식 관리자 API** (`XFormEditorCon.java`, `AdminFormCon.java`): 양식 필드명/안내문구의 다국어 데이터를 반환하는 인터페이스 확장이 필요합니다.

- **통합 메시징 서비스** (`sendMessage.do`): 알림 발송 시 수신자의 언어 설정에 맞는 알림 내용 발송이 필요합니다. `MessageHelper`의 매체별 어댑터가 다국어 알림 내용을 지원하는지 확인해야 합니다.

### 3-6. 데이터 정합성 및 트랜잭션 전략

- TECHNICAL_GUIDE_COMMON_SERVICES.md에 명시된 대로 `sys_base_dictionary` 데이터가 Redis에 캐싱되고 있으므로, 다국어 사전 데이터 등록/수정 시 DB와 Redis 캐시 간의 데이터 일관성을 보장해야 합니다.
- 사전 데이터 갱신 시 캐시 무효화(Cache Invalidation) 후 재로딩하는 전략을 적용해야 합니다.
- 다국어 데이터가 누락된 경우 기본 언어(한국어 또는 영어)로 폴백(Fallback)하는 처리 로직이 필요합니다. `Fn_BaseGetDictionary_S()` 함수의 폴백 동작을 확인해야 합니다.
- 인터뷰에서 확인된 결재 용어/기능 전체 목록에 대한 베트남어 번역 데이터의 완전성(Completeness)을 검증해야 합니다. 누락된 번역이 있을 경우 폴백 언어로 표시되어 사용자 경험에 영향을 줄 수 있습니다.
