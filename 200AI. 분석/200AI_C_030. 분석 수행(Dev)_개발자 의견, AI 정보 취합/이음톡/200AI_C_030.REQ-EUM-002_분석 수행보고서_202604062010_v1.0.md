## (3) 분석 범위

### 3-1. 영향도 분석 (기존 모듈 영향도 / DB·인프라 변경점)

#### (가) 기존 모듈 영향도

사간(社間) 채팅은 V1과 V2 두 계열사가 독립적인 Smarts4j 인스턴스를 운영하는 환경에서 계열사 경계를 넘어 메시지를 주고받는 기능입니다. 이는 단순한 채팅 기능 추가를 넘어서 분산 시스템 간 메시지 라우팅 아키텍처 변경을 수반하므로, 아래 모듈에 영향이 발생합니다.

**이음톡 채팅 서버 모듈 영향 (핵심)**: 현재 이음톡 WebSocket 채팅 서버는 단일 인스턴스 내에서 세션을 관리합니다. V1 서버와 V2 서버가 물리적으로 분리된 환경에서 사간 채팅을 구현하려면 두 서버 간 메시지를 교환하는 메시지 브로커 또는 메시지 중계 서비스가 필요합니다. 대표적인 구현 방안은 Redis Pub/Sub 또는 외부 메시지 큐(RabbitMQ, Kafka)를 통해 두 인스턴스를 연결하는 방식입니다.

분석 요청서에서 Redis Cache 의존성을 명시하고 있으므로, Redis Pub/Sub를 활용하여 V1·V2 채팅 서버 간 메시지를 실시간으로 중계하는 방식이 우선 검토 대상입니다.

**사용자 식별 및 조직도 모듈 영향**: V1 직원이 V2 직원을 채팅 상대로 검색·초대하려면 상대방 계열사의 조직도 정보를 조회할 수 있어야 합니다. 현재 각 계열사 조직도는 해당 인스턴스의 인사 DB에 독립적으로 저장되므로, 사간 조직도 조회를 위한 공유 디렉터리 또는 상대 인스턴스 API 호출 방식이 필요합니다.

**채팅방 및 메시지 DB 모듈 영향**: 사간 채팅방의 소유권(어느 인스턴스의 DB에 저장할지)과 메시지 저장 정책을 결정해야 합니다. 각 계열사 인스턴스에 각자 메시지 사본을 저장하는 방식과, 공유 메시지 DB에 단일 저장하는 방식 중 하나를 선택해야 합니다. 데이터 격리 정책(V1·V2 데이터 분리)과의 상충 가능성이 있으므로 PM의 추가 의사결정이 필요합니다.

**알림(Push Notification) 모듈 영향**: V1 직원이 V2 인스턴스에서 메시지를 수신할 때 알림이 정상 동작하려면, V2 서버가 V1 직원의 Push 토큰 또는 알림 엔드포인트를 알고 있어야 합니다. 이는 계열사 간 알림 정보 공유 또는 중계 알림 서비스가 필요함을 의미합니다.

**SaaS 모드(`isSaaS=Y`) 영향**: 분석 요청서에서 Smarts4j 프레임워크 전반의 `isSaaS=Y` 영향도 검토를 요청하였습니다. SaaS 모드의 테넌트 분리 정책은 계열사 간 데이터 접근을 기본적으로 차단합니다. 사간 채팅을 위해 테넌트 경계를 선택적으로 허용하는 예외 처리 로직이 필요하며, 이는 기존 테넌트 격리 정책과 충돌합니다. 이 상충 관계는 분석 요청서 PM 의견에서도 명시적으로 해결을 요청하고 있습니다.

#### (나) DB 및 인프라 변경점

**DB 변경**: 사간 채팅방을 식별하기 위한 채팅방 유형 컬럼(내부 단일 계열사 / 사간) 추가가 필요합니다. 메시지 DB에 발신자 계열사 식별자 컬럼 추가도 필요합니다. 공유 조직도 조회 방식을 API 호출로 결정하는 경우 별도 DB 변경은 최소화할 수 있습니다.

**인프라 변경**: V1 서버와 V2 서버 간 네트워크 통신 경로가 필요합니다. Redis Pub/Sub 방식을 선택하는 경우 V1·V2 공유 Redis 서버(또는 Redis Cluster)가 필요하며, 이는 기존 각 계열사 독립 Redis 구성에서 공유 Redis 구성으로의 전환 또는 Redis Replication 설정이 필요함을 의미합니다. 베트남 현지 네트워크 환경에서 공유 Redis 서버의 위치와 접근 Latency를 고려해야 합니다.

### 3-2. 기술적 타당성 (구현 난이도 / 제약 사항 / 대안 제시)

#### (가) 구현 난이도

분석 요청서에서 표준 아키텍처(egovframework) 준수 하에 구현 가능으로 평가하였으나, REQ-EUM-002는 세 가지 EUM 요구사항 중 기술 복잡도가 가장 높습니다. V1·V2 인스턴스 간 실시간 메시지 교환, 데이터 격리 정책과의 상충 해소, 분산 환경에서의 채팅방 상태 동기화가 모두 해결되어야 하기 때문입니다. 분석 요청서의 PM 제시 공수(4 M/D)는 분석·설계 단계 기준이며, 실제 구현 공수는 별도 산정이 필요합니다.

#### (나) 제약 사항

- **데이터 격리 정책 충돌**: V1·V2 계열사 데이터 격리 정책과 사간 채팅 메시지 공유 필요성이 상충합니다. 어느 범위까지 데이터를 공유할지에 대한 사업 의사결정이 선행되어야 합니다.
- **네트워크 분리 환경**: V1과 V2가 물리적으로 분리된 네트워크에서 운영될 경우, 두 서버 간 직접 통신 경로를 확보해야 합니다. 방화벽 정책 변경이 필요할 수 있습니다.
- **메시지 순서 보장**: 분산 환경에서 메시지 순서가 보장되어야 합니다. Redis Pub/Sub는 메시지 순서를 보장하지 않으므로, 메시지 시퀀스 번호 관리가 필요합니다.

#### (다) 대안 제시

- **페더레이션(Federation) 방식**: 각 인스턴스는 독립성을 유지하면서 사간 채팅을 위한 경량 페더레이션 프로토콜을 구현하는 방식입니다. 각 서버가 상대 서버의 API를 직접 호출하여 메시지를 전달합니다.
- **중앙 메시지 브로커 방식**: V1·V2가 공통으로 연결하는 중앙 메시지 브로커(RabbitMQ, Kafka)를 구축하여 계열사 간 메시지를 중계합니다. 메시지 순서 보장과 내구성이 Redis Pub/Sub보다 우수합니다.
- **단계적 구현**: 우선 V1·V2 공통 상위 서버에 사간 채팅 전용 채팅방을 운영하고, 각 계열사 이음톡 클라이언트에서 해당 서버에 접속하는 방식으로 1단계 구현 후, 완전한 사간 채팅 통합을 2단계로 추진하는 단계적 접근을 제안합니다.

### 3-5. API 및 서비스 인터페이스 영향도

소스코드가 제공되지 않아 구체적인 API 목록은 확인할 수 없습니다. 분석 요청서와 사간 채팅 도메인 지식을 기반으로 영향이 예상되는 인터페이스는 다음과 같습니다.

- **사간 채팅방 생성 API**: 참여자에 V1 직원과 V2 직원이 혼재하는 채팅방을 생성하는 API가 필요합니다. 현재 단일 계열사 내 채팅방 생성 API에서 계열사 경계를 넘는 참여자를 허용하는 로직 추가가 필요합니다.
- **계열사 간 사용자 검색 API**: V1 사용자가 V2 조직도에서 대화 상대를 검색하거나, V2 사용자가 V1 조직도를 검색할 수 있는 API가 필요합니다. 이는 상대 계열사 인스턴스의 조직도 API를 호출하거나, 공유 디렉터리 서비스를 구축하는 방식으로 구현합니다.
- **메시지 수신 알림 API**: V1 서버에서 발생한 메시지가 V2 사용자에게 Push 알림으로 전달되려면, V2 서버가 V1 메시지 이벤트를 수신하여 해당 사용자에게 알림을 발송하는 인터페이스가 필요합니다.
- **외부 인사 DB 연동**: 분석 요청서에서 외부 인사 DB 연동 영향도 검토를 요청하였습니다. 사간 채팅에서는 상대 계열사 인사 정보 조회가 필요하므로, 상대 계열사의 인사 DB를 직접 조회하거나 공유 API를 통해 간접 조회하는 방식이 필요합니다. 직접 DB 접근은 데이터 격리 정책 위반 위험이 있으므로 API 기반 간접 조회를 권장합니다.

### 3-6. 데이터 정합성 및 트랜잭션 전략

**메시지 전달 보장**: 네트워크 장애 또는 서버 재기동 시 사간 채팅 메시지가 유실되지 않도록 메시지 At-Least-Once 전달 보장이 필요합니다. Redis Pub/Sub는 구독자가 오프라인 상태일 때 메시지를 유실하는 특성이 있으므로, 이를 보완하는 메시지 영속화 처리(DB 저장 후 전달)가 필요합니다.

**데이터 격리와 공유의 균형**: V1·V2 계열사 간 데이터 격리 원칙 하에서 사간 채팅 메시지는 발신자 계열사 DB에 저장하고, 수신자 계열사 DB에는 메시지 참조 정보(발신자 계열사 코드, 메시지 ID)만 저장하는 방식으로 격리와 공유를 균형있게 처리하는 방안을 검토합니다.
