## (3) 분석 범위

### 3-1. 영향도 분석 (기존 모듈 영향도 / DB·인프라 변경점)

#### (가) 기존 모듈 영향도

다국어(베트남어) 적용은 이음톡 기능 로직 변경 없이 UI 텍스트 리소스를 추가하는 작업으로, 세 가지 EUM 요구사항 중 기술 영향도가 가장 낮습니다. 다만, Smarts4j 이음톡 모듈의 현행 다국어 처리 방식에 따라 영향 범위가 달라집니다.

**Smarts4j Core i18n 프레임워크 영향**: Smarts4j가 표준 Spring i18n(`MessageSource`, `LocaleResolver`, `LocaleChangeInterceptor`)을 사용하는 경우, 베트남어 언어 리소스 파일(`messages_vi.properties` 또는 `i18n_vi.json`)을 추가하고 `vi` 로케일을 등록하는 것만으로 대응이 완료됩니다. 이 경우 코어 수정은 필요 없습니다.

**이음톡 메신저 클라이언트 영향**: 이음톡이 Web 클라이언트(JavaScript SPA)로 구현된 경우, 서버 사이드 i18n 외에도 클라이언트 사이드 i18n 처리가 필요합니다. JavaScript i18n 라이브러리(예: i18next, vue-i18n, react-i18next 등)를 사용하는 경우 베트남어 언어 파일을 해당 라이브러리 형식으로 추가해야 합니다. 현재 이음톡 클라이언트의 프론트엔드 기술 스택이 확인되지 않아 소스코드 분석 후 구체적인 대응 방법을 확정해야 합니다.

**모바일 앱 영향**: 이음톡에 Android/iOS 앱이 존재하는 경우, 앱 내 다국어 리소스(Android: `res/values-vi/strings.xml`, iOS: `vi.lproj/Localizable.strings`)도 추가해야 합니다. 앱의 베트남어 지원 여부 및 앱스토어 베트남어 지역 설정도 확인이 필요합니다.

**번역 관련 영향**: 베트남어 번역 품질이 사용자 경험에 직접적인 영향을 미칩니다. IT 전문 용어(채팅, 알림, 수신, 파일 첨부 등)의 베트남어 표현이 현지 비즈니스 환경에서 자연스러운지 베트남어 원어민 검수가 필요합니다.

**SaaS 모드(`isSaaS=Y`) 영향**: 분석 요청서에서 Smarts4j 프레임워크 전반의 `isSaaS=Y` 영향도 검토를 요청하였습니다. SaaS 모드에서 사용자별 또는 테넌트별 언어 설정이 지원되어야 합니다. V1·V2 계열사의 기본 언어를 개별적으로 설정할 수 있어야 하며, 개별 사용자가 언어를 선택할 수 있는 개인 설정 기능이 필요합니다.

#### (나) DB 및 인프라 변경점

**DB 변경**: 사용자 언어 설정을 저장하는 컬럼(예: `LANG_CD` 또는 `LOCALE`)이 사용자 프로필 테이블에 없는 경우 추가가 필요합니다. 기본 언어 설정이 전체 시스템 설정에서 관리되는 경우 시스템 설정 테이블에 `vi` 로케일 항목 추가도 필요합니다.

**인프라 변경**: 베트남어는 유니코드(UTF-8)로 완전히 표현 가능하므로 DB 문자셋이 UTF-8 또는 UTF-8MB4로 이미 설정되어 있다면 별도 인프라 변경이 필요 없습니다. 문자셋이 다른 경우(예: EUC-KR) DB 문자셋 변경이 필요하며, 이는 시스템 전체에 영향을 미치는 중대한 변경입니다.

### 3-2. 기술적 타당성 (구현 난이도 / 제약 사항 / 대안 제시)

#### (가) 구현 난이도

분석 요청서에서 표준 아키텍처(egovframework) 준수 하에 구현 가능으로 평가하였습니다. 이음톡이 Smarts4j Core의 표준 i18n 프레임워크를 사용하고 있다면, 베트남어 언어 리소스 파일 추가와 번역 텍스트 작성이 전체 작업의 대부분을 차지합니다. 구현 난이도 자체는 낮으나, 번역 품질 확보(현지 원어민 검수 포함)와 하드코딩된 텍스트의 발굴·처리가 실질적인 작업 부담입니다.

#### (나) 제약 사항

- **하드코딩 텍스트 처리**: 이음톡 소스코드에 i18n 키가 아닌 하드코딩된 한국어 또는 영어 텍스트가 있는 경우, 해당 텍스트를 모두 i18n 키로 교체하는 추가 작업이 발생합니다. 소스코드 분석 전에는 이 작업의 규모를 예측할 수 없습니다.
- **번역 범위 정의**: 이음톡 UI 전체의 번역 대상 텍스트 목록(String Catalog)을 사전에 추출하여 베트남어 번역을 의뢰해야 합니다. 번역 의뢰에서 완료까지 소요 시간이 개발 일정에 영향을 미칠 수 있습니다.
- **우측에서 좌측(RTL) 방향 처리 불필요**: 베트남어는 좌측에서 우측(LTR) 쓰기 방향을 사용하므로, 아랍어 등 RTL 언어 지원과 달리 레이아웃 변경은 필요 없습니다.
- **특수 문자 렌더링**: 베트남어는 다수의 발음 구별 부호(diacritics)를 사용합니다. 이음톡 UI에서 사용 중인 폰트가 베트남어 특수 문자를 완전히 지원하는지 확인해야 합니다. 현재 적용 폰트가 베트남어 문자를 지원하지 않는 경우 폰트 변경 또는 폰트 스택 보완이 필요합니다.

#### (다) 대안 제시

- **기계 번역 초안 활용**: Google Translate API 또는 DeepL API를 사용하여 번역 초안을 자동 생성하고, 현지 베트남어 담당자가 검수하는 방식으로 번역 작업 시간을 단축할 수 있습니다.
- **단계적 적용**: 이음톡의 핵심 기능(채팅 메시지 작성·읽기, 채팅방 목록, 알림) UI를 우선 베트남어로 적용하고, 설정 화면·부가 기능은 2단계로 적용하는 방식으로 초기 적용 범위를 줄일 수 있습니다.

### 3-5. API 및 서비스 인터페이스 영향도

다국어 적용은 API 인터페이스 변경이 최소화되는 영역이나, 아래 인터페이스에 영향이 예상됩니다.

- **사용자 설정 저장 API**: 사용자가 언어 설정을 변경할 때 선택한 로케일(`vi`, `ko`, `en`)을 서버에 저장하는 API가 필요합니다. 이미 언어 설정 저장 API가 있다면 `vi` 로케일 허용 값 추가만으로 대응됩니다. 없는 경우 신규 개발이 필요합니다.
- **i18n 리소스 조회 API**: Web 클라이언트가 언어 리소스를 서버에서 동적으로 로드하는 방식이라면, 해당 API에서 `vi` 로케일 리소스를 반환하도록 수정해야 합니다.
- **알림 메시지 API**: Push 알림 또는 시스템 알림 메시지를 서버에서 생성하는 경우, 수신자의 언어 설정에 맞춰 알림 텍스트를 베트남어로 발송하는 로직이 필요합니다.
- **외부 인사 DB 연동**: 분석 요청서에서 외부 인사 DB 연동 영향도 검토를 요청하였습니다. 다국어 기능에서는 외부 인사 DB와의 직접적인 영향이 없으나, 사용자 프로필 테이블에 언어 설정 컬럼을 추가하는 경우 인사 DB 동기화 처리에서 해당 컬럼을 포함해야 합니다.

### 3-6. 데이터 정합성 및 트랜잭션 전략

**언어 설정 저장 일관성**: 사용자 언어 설정 변경은 단일 DB 레코드 업데이트로, 트랜잭션 복잡도가 매우 낮습니다. Spring `@Transactional`로 원자성을 보장합니다.

**언어 리소스 캐시 일관성**: Redis에 언어 리소스를 캐싱하는 경우, 언어 파일 업데이트(번역 수정) 시 Redis 캐시를 무효화해야 합니다. 배포 후 자동 캐시 무효화 처리 또는 관리자 캐시 초기화 기능을 제공해야 합니다.

**테넌트별 언어 기본값 설정**: V1·V2 계열사 각각에 대해 기본 언어(베트남어 또는 한국어)를 시스템 설정에서 관리하며, 테넌트 설정 테이블과의 정합성을 유지해야 합니다. 테넌트 기본 언어 변경 시 해당 테넌트 내 모든 사용자에게 즉시 적용할지, 사용자가 명시적으로 설정한 경우 제외할지 정책을 정의해야 합니다.
