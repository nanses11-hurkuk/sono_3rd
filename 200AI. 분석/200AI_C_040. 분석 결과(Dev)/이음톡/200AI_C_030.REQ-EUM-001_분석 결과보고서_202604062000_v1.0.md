## (3) 분석 범위

### 3-1. 영향도 분석 (기존 모듈 영향도 / DB·인프라 변경점)

#### (가) 기존 모듈 영향도

이음톡은 Smarts4j 그룹웨어 내 실시간 메신저 서브시스템입니다. 현재 시스템은 내부 임직원을 대상으로 운영되고 있으며, 외부 사용자(Vendor) 지원을 위해서는 아래 모듈에 영향이 발생합니다.

**인증 및 회원 관리 모듈 영향**: Smarts4j 표준 인증 체계는 내부 인사 DB 기반으로 동작합니다. Vendor 사용자는 내부 인사 DB에 등록되지 않은 외부 사용자이므로, 외부 사용자 전용 계정 저장소(별도 테이블 또는 별도 디렉터리)와 인증 흐름이 필요합니다. Spring Security의 `UserDetailsService` 구현체를 확장하여 내부 사용자와 외부 사용자를 동일한 인증 파이프라인에서 처리하되, 권한(Role)을 분리하는 방식으로 대응할 수 있습니다.

**이음톡 채팅 서버 모듈 영향**: 현재 WebSocket 또는 STOMP 기반으로 동작하는 이음톡 채팅 서버는 세션 인증 정보를 기반으로 사용자를 식별합니다. 외부 사용자가 채팅 서버에 접속할 수 있도록 WebSocket 핸드셰이크 단계에서 외부 사용자 토큰 검증 로직을 추가해야 합니다.

**조직도 및 대화상대 검색 모듈 영향**: 내부 임직원 검색은 인사 DB 기반으로 동작하나, 외부 Vendor 사용자는 별도 관리 테이블에서 조회해야 합니다. 대화상대 검색 API에서 내부/외부 사용자를 통합 검색하거나 분리 검색하는 방식을 결정해야 합니다.

**알림(Push Notification) 모듈 영향**: 내부 사용자는 Smarts4j 내부 알림 체계를 사용하나, 외부 Vendor 사용자는 별도 Push 수신 수단(FCM, APNs 또는 이메일 알림)이 필요합니다.

**SaaS 모드(`isSaaS=Y`) 영향**: 분석 요청서에서 Smarts4j 프레임워크 전반의 `isSaaS=Y` 영향도 검토를 요청하였습니다. SaaS 모드에서는 테넌트 분리 정책이 적용되므로, Vendor 사용자를 특정 테넌트에 귀속시키는 방식으로 데이터 격리를 달성할 수 있습니다. V1·V2 계열사 간 데이터 격리 정책과 충돌하지 않도록 Vendor 계정의 테넌트 귀속 기준을 명확히 정의해야 합니다.

#### (나) DB 및 인프라 변경점

**DB 변경**: 외부 사용자(Vendor) 계정 저장을 위한 신규 테이블(예: `TB_VENDOR_USER`, `TB_VENDOR_COMPANY`)이 필요합니다. 테이블에는 최소한 계정 ID, 비밀번호(해시), 소속 Vendor사, 연락처, 접속 권한 범위, 사용 여부, 등록일시, 만료일시 필드가 포함되어야 합니다.

**인프라 변경**: 외부 인터넷에서 이음톡 채팅 서버로의 접근을 허용하려면 방화벽 정책 변경 및 외부 접근용 도메인·SSL 인증서 설정이 필요합니다. 베트남 현지 네트워크 환경에서 외부 Vendor가 접속하는 경우 WebSocket 연결의 안정성을 보장하기 위해 리버스 프록시(Nginx 등) 설정 최적화가 필요합니다.

### 3-2. 기술적 타당성 (구현 난이도 / 제약 사항 / 대안 제시)

#### (가) 구현 난이도

분석 요청서에서 표준 아키텍처(egovframework) 준수 하에 구현 가능으로 평가하였습니다. 전자정부 표준 프레임워크 기반의 Spring Security 확장과 Smarts4j Extension Point를 활용하면 코어 수정 없이 외부 사용자 지원을 추가할 수 있습니다. 다만, 내부 사용자와 외부 사용자의 세션 격리 및 권한 분리 설계가 핵심 난이도 요소이며, 특히 이음톡 WebSocket 세션 관리에서 외부 사용자를 어떻게 식별하고 격리할지에 대한 세부 설계가 필요합니다.

#### (나) 제약 사항

- **보안 위험**: 외부 인터넷에서 접근 가능한 채팅 기능은 내부 시스템 보안 위협이 증가합니다. 외부 Vendor 사용자가 접근 가능한 채팅방과 내부 전용 채팅방을 명확히 분리해야 합니다.
- **계정 관리 정책**: Vendor 사용자 계정의 등록·승인·만료·삭제 프로세스가 정의되어야 합니다. 현재 이음톡은 이 프로세스가 없으므로 신규 개발이 필요합니다.
- **베트남 현지 외부 네트워크 접근성**: 베트남 네트워크 환경에서 외부 Vendor(다른 국가 포함 가능)의 접속 품질이 보장되어야 합니다. WebSocket Long-polling 폴백 지원 여부를 검토해야 합니다.

#### (다) 대안 제시

- **초대 링크 방식**: 내부 직원이 Vendor 사용자에게 일회성 채팅 초대 링크를 발송하고, Vendor 사용자는 별도 계정 없이 임시 토큰으로 특정 채팅방에만 참여하는 방식을 대안으로 검토할 수 있습니다. 이 경우 Vendor 계정 관리 시스템을 구축하지 않아도 됩니다.
- **별도 Vendor 포털 채널 운영**: 이음톡 내부에 Vendor 사용자를 통합하는 대신, 별도 Vendor 전용 채널(예: 이메일 알림 기반 메시지 전달)을 운영하는 방식도 검토 가능합니다.

### 3-5. API 및 서비스 인터페이스 영향도

소스코드가 제공되지 않아 구체적인 API 목록 및 인터페이스는 확인할 수 없습니다. 분석 요청서와 이음톡 기능 도메인 지식을 기반으로 영향이 예상되는 인터페이스는 다음과 같습니다.

- **사용자 인증 API**: 현재 내부 사용자 전용 로그인 API에 외부 사용자 인증 경로 추가가 필요합니다. 외부 사용자는 별도 로그인 엔드포인트(예: `/api/vendor/login`) 또는 토큰 발급 엔드포인트를 통해 인증하는 방식을 검토합니다.
- **대화방 생성·참여 API**: 내부 사용자가 외부 Vendor를 초대하는 흐름, 외부 Vendor가 접근 가능한 채팅방을 제한하는 접근 제어 로직이 필요합니다.

### 3-6. 데이터 정합성 및 트랜잭션 전략

**Vendor 계정 데이터 무결성**: Vendor 사용자 계정은 내부 인사 DB와 독립적으로 관리되므로, 계정 상태(활성/만료/삭제)가 이음톡 채팅 서비스 접근 권한과 항상 동기화되어야 합니다. 계정 비활성화 시 해당 사용자의 WebSocket 세션을 즉시 종료하는 처리가 필요합니다.

**V1·V2 계열사 데이터 격리**: 분석 요청서에서 PM이 V1·V2 계열사 간 데이터 격리와 결재 연동의 상충 관계 해결을 요청하였습니다. Vendor 사용자가 특정 계열사(V1 또는 V2)에 귀속되는 경우, 해당 계열사의 테넌트 범위 내에서만 채팅 접근이 허용되도록 설계해야 합니다. 테넌트 키(`companyCode` 또는 `tenantId`)를 Vendor 계정 테이블의 외래키로 연결하는 방식을 검토합니다.

**트랜잭션 범위**: Vendor 계정 등록·수정·삭제 작업은 단일 DB 트랜잭션 내에서 처리하며, Spring `@Transactional`을 통해 원자성을 보장합니다.
