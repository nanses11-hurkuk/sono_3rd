## (4) 영향도 분석 — 기존 모듈

### 4.1 패키지·레이어

| 구분 | 영향 대상 | 영향 내용 |
|------|-----------|-----------|
| **smarts4j.covicore** | 도메인·사용량·시스템 관리 SQL(`sys_object_domain` 등) | 도메인 추가에 따른 관리 화면·통계 쿼리 결과 행 증가, 필터 조건 점검 |
| **smarts4j.framework** | `SessionHelper`, `RedisDataUtil`, 로그인·캐시 | 세션 `DN_ID`, 도메인별 Redis 기초설정 로딩 경로는 **변경 빈도 낮음**, 설정값만 증가 |
| **smarts4j.groupware** | 공통 서비스·각 업무 모듈 | 이미 `SessionHelper.getSession("DN_ID")` + `getBaseConfig(..., DN_ID)` 패턴 사용(예: 연동 모듈). **신규 도메인만 추가 시 대부분 설정·데이터** |
| **인프라** | Redis, WAS, 리버스 프록시 | 캐시 키에 도메인 포함 여부·세션 클러스터 시 도메인 일관성 |

### 4.2 코드·설정 근거(추적)

- **도메인 마스터**: `covicore` SQL맵에서 `sys_object_domain` 참조(도메인 목록·집계).
- **도메인별 기초설정**: `egovframework.coviframework.util.RedisDataUtil#getBaseConfig(String, String)` — 두 번째 인자에 도메인 ID 전달 패턴.
- **세션**: `DN_ID`는 `groupware` 자원·일정 등 다수 서비스에서 이미 사용.

### 4.3 DB·데이터

| 객체 | 영향 |
|------|------|
| **sys_object_domain** | V1, V2 행 추가(핵심) |
| **사용자·부서·객체 매핑 테이블** | 각 행의 `DomainID`가 V1/V2 중 하나를 가리키도록 데이터 이관·신규 입력 규칙 |
| **sys_base_config(도메인별)** | 연동·기능 플래그를 도메인별로 다르게 둘 경우 행 추가 |
