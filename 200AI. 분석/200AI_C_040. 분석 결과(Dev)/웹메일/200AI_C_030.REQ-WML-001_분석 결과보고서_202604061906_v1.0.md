#### (2) DB·인프라 변경점

제공된 소스 파일에서는 DB 스키마(DDL) 파일이 포함되어 있지 않아 DB 변경점의 직접 확인은 불가합니다. 코드 분석을 통해 추정 가능한 내용은 다음과 같습니다.

- `RestApiUtil.java`의 `makeHeader` 메서드 내 config 항목 분석 결과, Exchange 연동에는 다음의 설정 항목이 존재합니다: `adUrl`, `adUser`, `adPwd`, `adBaseDn`, `adRootName`, `exchangeUser`, `exchangePwd`, `exchangeUrl`, `exchangeStorageServer`, `exchangeIsResetMailAddress`, `ewsUrl`, `ewsUser`, `ewsPwd`, `ewsDomain`, `ewsVersion`, `pwshPoolSize`, `pwshCertPath`, `pwshCertPassword`, `pwshAppID`, `pwshOrganization`. 해당 설정들은 도메인별(`domainId`)로 저장·관리되는 구조이며, DB 내 Exchange 연동 설정 테이블이 존재할 것으로 추정됩니다.
- 메일 규칙(`MsMailSettingsRules`)은 Exchange 서버에 직접 저장되는 구조인지 또는 로컬 DB에도 저장되는지 여부는 백엔드 서비스 소스 미제공으로 확인 불가합니다.
- Redis 세션 필터가 주석 처리된 상태이므로, 현재 구성에서 Redis는 세션 공유 목적으로 사용되지 않고 있습니다. 캐시 용도로 별도 사용 여부는 미확인입니다.

### 3-2. 기술적 타당성 (구현 난이도 / 제약 사항 / 대안 제시)

#### (1) 구현 난이도

| 구분 | 난이도 | 이유 |
|---|---|---|
| Exchange On-Premise(EWS) 연동 | 상(高) | EWS는 Microsoft의 레거시 프로토콜로, 버전별 API 호환성 차이가 크고 NTLM/Kerberos 인증 처리가 복잡합니다. `RestApiUtil`의 `ewsVersion` 설정으로 버전을 관리하고 있으나, 버전 불일치 시 오류 발생 위험이 높습니다. |
| Exchange Online(Graph API) 연동 | 중(中) | OAuth2 기반의 Graph API는 표준화된 REST 방식으로 EWS 대비 연동이 용이합니다. 단, AccessToken의 만료 및 갱신(Refresh Token) 처리 로직이 현재 코드에서 확인되지 않아, 장시간 세션 유지 시 토큰 만료 문제가 발생할 수 있습니다. |
| 메일 규칙(Rules) 설정 | 중(中) | 화면 UI(JSP)는 구현되어 있으나, 실제 Exchange 서버에 규칙을 생성/수정/삭제하는 백엔드 서비스 소스가 제공되지 않아 전체 구현 완료 여부 확인이 필요합니다. |
| 연락처(Contacts) 관리 | 중(中) | 연락처 화면 UI(ContactList.jsp 등)는 구현되어 있으나, 백엔드 서비스 소스 미제공으로 전체 구현 완료 여부 확인이 필요합니다. |
| 폴더 구조 동기화 | 상(高) | Exchange 폴더와 covimail 폴더 간 양방향 매핑(`WellKnownFolderName`, `getHashFromString`)이 구현되어 있으나, Exchange 서버의 사용자 정의 폴더 계층 구조가 깊어질 경우 재귀 탐색 성능 이슈가 발생할 수 있습니다. |
| 대용량 첨부파일 처리 | 상(高) | REQ-ODD-001에서 대용량 처리가 언급되었으나, 현재 제공된 소스에서는 대용량 처리 전략이 확인되지 않습니다. coviexchange 서버 내부 구현 확인이 필요합니다. |
| 베트남 현지 고지연 환경 대응 | 상(高) | `RestApiUtil`의 `RestTemplate`에서 읽기 타임아웃과 연결 타임아웃이 주석 처리된 상태입니다. 베트남 현지 네트워크 환경(고지연)에서 Exchange 서버와 통신 시 응답 지연으로 인한 서비스 행(hang) 위험이 있습니다. |

#### (2) 제약 사항

1. **smarts4j-covicore.zip 및 smarts4j-coviexchange.zip 미제공**: 분석 요청서에 명시된 주요 입력 파일(smarts4j-covicore.zip, smarts4j-coviexchange.zip)이 제공되지 않았습니다. 이로 인해 코어 모듈 의존성, coviexchange 서버 내부 API 구현, DB 매퍼(MyBatis XML), 서비스 구현 클래스(ServiceImpl) 전체를 확인할 수 없었습니다. 본 보고서는 제공된 Java 유틸 클래스 및 JSP 화면 파일만을 기반으로 작성하였습니다.

2. **AccessToken 갱신 로직 미확인**: `RestApiUtil`의 `makeHeader`에서 `AccessToken`을 헤더에 직접 설정하는 방식만 확인되었으며, 토큰 만료 시 자동 갱신(Refresh Token Flow)하는 로직은 제공된 소스에서 확인되지 않았습니다. Graph API 연동 시 장시간 세션 유지 환경에서 토큰 만료 오류 발생 위험이 있습니다.

3. **Redis 세션 비활성**: `web.xml`에서 `springSessionRepositoryFilter`가 주석 처리되어 있어, 현재 구성에서는 다중 WAS 환경(클러스터)에서 세션 공유가 불가능합니다. 베트남 이중 계열사(V1, V2) 환경에서 세션 격리 정책을 확인해야 합니다.

4. **타임존 처리 의존성**: `DataConverter`에서 사용자 세션의 `UR_TimeZoneDisplay` 값을 사용하여 UTC 시간을 변환합니다. 사용자 세션에 타임존 정보가 없는 경우 `GMT+9`를 기본값으로 사용합니다. 베트남 현지 사용자의 타임존(`Asia/Ho_Chi_Minh`, UTC+7) 처리가 올바르게 세션에 설정되어 있는지 확인이 필요합니다.

#### (3) 대안 제시

| 제약/위험 | 대안 |
|---|---|
| EWS 레거시 프로토콜 의존 | MS의 EWS 지원 종료(Deprecation) 일정을 고려하여 중장기적으로 Graph API 단일 연동으로 전환하는 방향을 검토합니다. 현재 EWS/Graph 이중 분기 구조는 전환 유예 기간 동안 유지합니다. |

| 베트남 고지연 환경 | RestTemplate의 읽기 타임아웃 및 연결 타임아웃 값을 환경별 프로퍼티(`exchange.properties`)에서 관리하도록 설정합니다. 또한 coviexchange 서버를 베트남 현지 가까운 위치에 배치하는 네트워크 토폴로지 검토를 권장합니다. |
| 폴더 재귀 탐색 성능 | 깊은 폴더 계층 조회 시 Redis 캐시에 폴더 트리를 캐싱하여 반복 조회 비용을 절감하는 방안을 검토합니다. |
| Redis 세션 비활성 | 클러스터 환경 적용 전 `springSessionRepositoryFilter` 활성화 및 Redis Sentinel/Cluster 설정을 검토합니다. |

### 3-6. 데이터 정합성 및 트랜잭션 전략

#### (1) 현재 확인된 트랜잭션 처리 방식

- 제공된 소스 파일에서 `@Transactional` 어노테이션이나 트랜잭션 경계 설정은 확인되지 않았습니다. 실제 트랜잭션 처리는 서비스 구현 클래스(ServiceImpl, 소스 미제공)에서 이루어질 것으로 추정됩니다.
- `DataConverter.convertToLocalTime`은 `SessionHelper.getSession`을 통해 사용자 타임존을 가져오므로, 세션이 만료되거나 타임존 정보가 없는 경우 기본값(`GMT+9`) 또는 예외가 발생할 수 있습니다.

#### (2) 데이터 정합성 이슈 및 위험

| 이슈 | 상세 설명 | 위험도 |
|---|---|---|
| **Exchange와 GW DB 간 이중 저장 정합성** | 메일 규칙, 연락처 등이 Exchange 서버와 로컬 DB에 이중으로 저장되는 경우, 네트워크 오류나 Exchange API 실패 시 양쪽 데이터 불일치가 발생할 수 있습니다. | 높음 |
| **베트남 계열사(V1, V2) 간 데이터 격리** | PM 의견에서 언급한 V1, V2 계열사 간 데이터 격리 요구사항이 있습니다. 현재 `domainId` 헤더를 통해 도메인 격리를 시도하는 구조이나, 서비스 레이어에서의 격리 정책은 소스 미제공으로 확인 불가합니다. | 높음 |
| **반복 일정 변환 버그 가능성** | `DataConverter.convertToMSRecurrence`에서 `repeatAppointType`이 "A"인 조건이 두 번 중복되어 있습니다 (line 221, 223). `"A".equals(repeatAppointType)` 조건이 반복되어 두 번째 분기(`daysOfWeek` 설정)가 실행되지 않는 버그가 의심됩니다. 반복 일정 저장 시 데이터 불일치가 발생할 수 있습니다. | 중간 |
| **타임존 불일치** | `DataConverter.convertToLocalTime`은 세션 타임존을 사용하지만, `convertToUTC`는 세션 타임존이 없는 경우 `GMT+9`를 기본값으로 사용합니다. 베트남(UTC+7) 사용자 환경에서 시간 데이터 불일치가 발생할 수 있습니다. | 중간 |
| **SHA-256 해시 기반 폴더 ID 충돌 위험** | `GWMailUtil.getHashFromString`은 MS Exchange 폴더 ID를 SHA-256 해시로 변환하여 covimail의 폴더 GUID로 사용합니다. 이론적 충돌 가능성은 낮으나, 원본 Exchange 폴더 ID와의 역방향 매핑은 불가능하므로 장애 시 디버깅이 어렵습니다. | 낮음 |

#### (3) 권장 트랜잭션 전략

- Exchange API 호출과 로컬 DB 저장이 병행되는 경우, Exchange API 성공 후 로컬 DB 저장을 시도하는 순서를 원칙으로 합니다. Exchange API 실패 시 로컬 DB에도 반영하지 않도록 트랜잭션 경계를 명확히 설정합니다.
- Exchange 서버와의 통신은 외부 시스템 호출이므로 Spring의 선언적 트랜잭션(`@Transactional`)만으로는 분산 트랜잭션을 보장할 수 없습니다. 보상 트랜잭션(Compensating Transaction) 또는 이벤트 기반 Eventually Consistent 전략을 검토합니다.
- `DataConverter.convertToMSRecurrence`의 `repeatAppointType` 조건 중복 버그를 반드시 수정 후 배포합니다.

### 4-2. 개발자 의견 (이동준)

1. **타임아웃 설정 즉시 반영 필요**: RestTemplate 타임아웃이 주석 처리된 채 배포되면 베트남 현지 네트워크 환경(평균 RTT 50~200ms 예상)에서 Exchange API 응답 지연 발생 시 Tomcat 스레드 고갈로 전체 서비스 장애로 이어질 수 있습니다. 개발 완료 전 반드시 적용이 필요합니다.

2. **EWS vs Graph API 이중 구조 장기 관리 부담**: 현재 `Module.EXCHANGE`(EWS)와 `Module.GRAPH`(Graph API)를 `ServiceContext`로 분기하는 구조는 단기적으로는 유연하지만, 두 프로토콜의 동작 차이(응답 형식, 오류 처리 방식 등)로 인해 유지보수 부담이 증가합니다. EWS 지원 계획 기간을 PM과 협의하여 Graph API 단일화 시점을 명확히 정의하는 것을 권장합니다.

3. **DataConverter 반복 일정 버그 수정 우선 처리**: 코드 라인 221~224에서 `"A".equals(repeatAppointType)` 조건이 두 번 중복되어 있어, 월별/연별 반복 일정의 요일 지정 방식(B타입)이 실제로 처리되지 않습니다. 이는 일정 데이터 정합성에 직접적인 영향을 주므로 단위 테스트 작성과 함께 우선 수정을 권장합니다.

4. **전체 소스 확보 후 재분석 필요**: smarts4j-covicore.zip 및 smarts4j-coviexchange.zip이 제공되지 않아 서비스 구현 클래스, DB 매퍼, coviexchange 내부 API 전체를 확인하지 못하였습니다. 본 보고서는 제공된 유틸 클래스 및 JSP 파일만을 기반으로 한 부분 분석임을 명시합니다. 전체 소스 확보 후 재분석이 반드시 필요합니다.

### 4-3. AI 분석 의견

1. **보안 강화 권고**: `RestApiUtil.makeHeader`에서 Exchange/AD/EWS 비밀번호를 포함한 설정 정보를 Base64로 인코딩하여 HTTP 헤더에 전달하는 구조는 Base64가 암호화가 아닌 단순 인코딩임을 고려할 때, HTTPS 미적용 환경에서는 패킷 스니핑만으로 비밀번호가 노출됩니다. 설정 전달 방식을 공개키 암호화(RSA) 또는 환경 변수 기반으로 개선하는 것을 권고합니다.

2. **재시도 로직 개선 권고**: `sendRequest`의 최대 5회 재시도 로직에서 재시도 간격(Backoff)이 없어 빠른 연속 재시도가 발생합니다. Exchange 서버 부하 상황에서 재시도 폭주(Retry Storm)를 방지하기 위해 지수 백오프(Exponential Backoff)와 재시도 간격을 추가하는 것을 권고합니다.

3. **GraphUtils 확장성 검토**: 현재 `GraphUtils`는 `subject`, `from`, `to`, `receivedDateTime` 4가지 검색 조건만 지원합니다. 웹메일 요구사항에서 추가 검색 조건(예: hasAttachments, 읽음 여부, 폴더 지정 등)이 필요할 경우 `graphDTOInfos` 맵에 항목을 추가하는 방식으로 확장이 가능한 구조입니다.

4. **Converter 클래스 Jackson 버전 호환성**: `Converter.java`의 `objectMapper.convertValue`는 Jackson 2.x에서 정상 동작합니다. Spring Boot 3.x 이상 사용 시 Jackson 3.x와의 호환성을 확인해야 합니다.
