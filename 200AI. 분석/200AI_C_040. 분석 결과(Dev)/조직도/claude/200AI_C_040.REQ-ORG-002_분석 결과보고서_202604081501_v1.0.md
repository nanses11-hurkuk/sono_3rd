#### 분석 범위

REQ-ORG-002의 분석 범위는 다음과 같다.

1. **GW → Local AD 동기화**: GW SYNC 배치 결과를 Local AD Object(사용자 계정, 부서 그룹)로 생성·수정·비활성화하는 모듈 개발. AADC가 이를 Entra ID에 자동 전파하므로, GW에서 Microsoft Graph API를 직접 호출하는 복잡한 로직은 불필요하다.
2. **AD 통합 대응**: V1·V2 AD 통합 전후 2단계 접근. 통합 전에는 V1·V2 각각 독립 배치, 통합 후에는 단일 경로로 전환.
3. **Exchange 메일그룹 동기화**: 부서코드@utivina.com, 직위코드@utivina.com 메일그룹 자동 생성·비활성화 로직을 `smarts4j.coviexchange` 모듈에 추가.
4. **메일 도메인 전환 대응**: utikorea.com → utivina.com 전환 기간 중 ProxyAddresses 관리 전략 수립.

#### 제약 사항

1. **AD 통합 미완료 (개발 착수 최대 블로커)**: V1·V2 AD 통합이 완료되어야 통합 기준 AD 동기화 개발 착수 가능. 통합 전에는 V1·V2 각각 독립 배치로 임시 구성. (산출 근거: 회의록_AD.md 항목 1)
2. **AD 배포단위 미확정**: 배포단위(부서, 직위만) UTI 내부 검토 중. 확정 전 AD Object 구조 구현 불가. (산출 근거: 회의록_AD.md 항목 2)
3. **개발 테넌트 없음**: Exchange Online 메일그룹, AADC 동기화 결과 검증 불가. Microsoft 365 Developer Program 활용 대안 검토. (산출 근거: 회의록_ETC.md 항목 1)
4. **AADC Throttling/Latency**: AADC 동기화 주기(기본 30분)를 고려하면 GW 배치 완료 후 M365 반영까지 최대 30분 지연 발생 가능. 이를 운영 정책으로 공유해야 함. (산출 근거: Claude 분석 수행보고서 (3-2))
5. **utikorea.com → utivina.com 전환 기간**: 두 도메인 계정 공존으로 GW MailAddress 일관성 확보 전략 필요. (산출 근거: 회의록_ETC.md 항목 1)
6. **ProxyAddresses 관리**: 퇴사 시 사서함 비활성화하나 ProxyAddresses에서 기존 주소 제거하지 않음. 동일 메일주소 재사용 방지 로직 필요. (산출 근거: 회의록_Exch.md 항목 4)
7. **AD 암호 정책**: AD 암호 정책과 GW 암호 정책의 동기화 여부 결정 필요(SSO 사용 시 AD 우선). (산출 근거: Gemini 분석 수행보고서 (2))
