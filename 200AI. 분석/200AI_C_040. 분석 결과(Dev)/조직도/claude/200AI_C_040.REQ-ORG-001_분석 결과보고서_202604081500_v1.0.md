#### 분석 범위

REQ-ORG-001의 분석 범위는 다음과 같다.

1. **영림원 ERP(MSSQL) → 그룹웨어(MariaDB) DB-to-DB 연동 배치 모듈 신규 개발**: V2 계열사에 한하여 영림원 MSSQL View에서 사용자·부서 데이터를 조회하여 Smarts4j 표준 SYNC FieldMapping 흐름(HR 중간 테이블 → Org 테이블 → Temp 비교 테이블 → Compare 테이블 → sys_object 최종 반영)으로 조직도 데이터를 동기화한다.
2. **V1/V2 CompanyCode 기반 분기 처리**: V1은 배치 대상에서 제외하고 수동관리를 유지하며, V2만 자동 연동을 수행하는 분기 로직을 구현한다.
3. **Local AD 연동**: 동기화 배치 내 AD 계정 생성/수정/비활성화 연동을 포함한다.
4. **FieldMapping.sql 버그 수정**: CONSTANT 타입 빈 문자열 이슈 및 QUERY 타입 `#{_parameter[isSaaS]}` JDBC 바인딩 오류 수정을 포함한다.

#### 제약 사항

1. **영림원 View 스키마 미확정 (개발 착수 최대 블로커)**: View 컬럼 목록, 데이터 타입, 인코딩 정보 없이는 `insertHRPerson`, `insertHRDept` FieldMapping 작성 불가. 영림원 측에 `orgs_hr_person`, `orgs_hr_dept` 스키마 문서를 제공하고 동일 컬럼 구조의 View 제공을 요청해야 한다. (산출 근거: Claude 분석 수행보고서 GAP-001, Gemini 분석 수행보고서 (3))
2. **사용자 식별키 미확정 (개발 착수 최대 블로커)**: UserCode vs EmpNo 결정이 DB Primary Key, FieldMapping Key 설계 전반에 영향을 미친다. UTI 담당자의 공식 답변 전까지 해당 영역 개발 착수 불가. (산출 근거: 회의록_공통.md 항목 9)
3. **개발 환경 MSSQL 접근 불가**: 방화벽 오픈 및 테스트 계정이 없으면 개발 단계 통합 테스트 불가. (산출 근거: Claude 분석 수행보고서 (3-2))
4. **베트남 현지 네트워크 Latency**: GW WAS ↔ 영림원 MSSQL 간 실측 Latency 미확인. Connection Timeout 설정값은 현지 실측 후 조정 필요. (산출 근거: Gemini 분석 수행보고서 (3), Claude 분석 수행보고서 (3-7))
5. **FieldMapping.sql 버그 (수정 완료, 재배포 필요)**: `ELSE ''` 이슈 및 QUERY 타입 JDBC 이슈 수정 완료. 수정본 배포 및 DB 재삽입 필요. (산출 근거: `AI_Folder\item\SYNC\FieldMapping.sql` 분석)

### 2-7. 데이터 정합성 및 트랜잭션 전략

| 처리 단계 | 트랜잭션 전략 | 오류 처리 | 산출 근거 |
|----------|------------|----------|----------|
| **1단계: 영림원 View 조회** | READ_ONLY (MSSQL 트랜잭션) | 오류 시 전체 배치 중단, 관리자 알림, 이전 상태 유지 | Claude 분석 수행보고서 (3-6) |
| **2단계: orgs_hr_* 적재** | V2 CompanyCode 기준 Truncate-and-Load (MariaDB 트랜잭션) | 실패 시 롤백. 이전 배치 데이터는 Truncate 전이면 보존 | Claude 분석 수행보고서 (3-6) |
| **3단계: orgs_org* 반영** | UPSERT (INSERT ON DUPLICATE KEY UPDATE) | 키 충돌 시 UPDATE로 대체, 신규 건은 INSERT | Claude 분석 수행보고서 (3-6) |
| **4단계: orgs_temp_object_* 적재** | 배치 시작 시 전체 Truncate 후 재삽입 | 실패 시 롤백. Truncate 후 삽입 실패 시 공백 상태 방지 주의 | Claude 분석 수행보고서 (3-6) |
| **5단계: orgs_compare_object_* 비교** | INSERT/UPDATE/DELETE 각각 별도 트랜잭션 | DELETE 건수 10명 초과 시 배치 중단, 관리자 확인 후 수동 처리 | Claude 분석 수행보고서 (3-6) |
| **6단계: sys_object_user/group 반영** | sys_object_user_base 정합성 검증 후 커밋 | 정합성 불일치 시 커밋 보류 및 알림 | Claude 분석 수행보고서 (3-6) |
| **7단계: Redis 캐시 무효화** | 전체 커밋 완료 후 수행 (트랜잭션 외부) | Flush 실패 시 재시도. 최대 재시도 3회 | Claude 분석 수행보고서 (3-6) |

> **Truncate-and-Load vs UPSERT 전략 평가** (산출 근거: Claude AI 의견):
> - `orgs_hr_*` 중간 테이블은 Truncate-and-Load 방식이 안전하다. 이유: 외부 ERP View는 전체 현재 상태를 항상 제공하므로, 누락/갱신 판단 없이 전체 교체가 정합성 확보에 유리하다.
> - `orgs_orgperson`, `orgs_orgdept`는 UPSERT 방식이 적합하다. 이유: 히스토리 및 GW에서 직접 추가한 정보(직위/직책 수동 등록 등)가 포함될 수 있으므로 덮어쓰기 위험을 최소화해야 한다.

> **Interface Table(Staging) 전략** (산출 근거: Gemini 분석 수행보고서 (3)):
> Interface Table(Staging)에 우선 적립 후, 유효성 검사(Validation)를 통과한 데이터만 본 테이블에 Batch Insert/Update 수행한다.
