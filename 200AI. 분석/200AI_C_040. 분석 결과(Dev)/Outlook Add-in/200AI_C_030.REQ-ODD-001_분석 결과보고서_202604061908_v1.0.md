#### 3-1-2. DB 변경점

분석된 소스 기준으로 DB 스키마 DDL 변경이 명시된 파일은 없다. 단, 아래 항목들은 DB 연동 가능성이 있으므로 추가 확인이 필요하다.

- 주소록(Contact) 데이터: `ContactList.jsp`, `ContactAddPopup.jsp` 화면에서 표시하는 연락처 데이터가 내부 DB에 저장되는지, Exchange 서버에서 실시간 조회하는지 구조 확인 필요.
- 메일 설정 규칙(JunkEmailConfiguration): 설정 값이 Exchange 서버에만 저장되는지, 내부 DB에도 이력을 저장하는지 확인 필요.
- 메일박스 프로비저닝 이력: `setMailbox.ps1` 실행 결과를 별도 로그 테이블에 기록하는지 여부 확인 필요.

#### 3-1-3. 인프라 변경점

- PowerShell Remoting 포트(TCP 5985/5986) 방화벽 개방 필요: `setSession.ps1`이 `New-PSSession`으로 Exchange 서버에 접속하므로, WinRM(Windows Remote Management) 포트가 서버 간 방화벽에서 허용되어야 한다.
- Kerberos 인증을 위한 Active Directory 도메인 환경 전제: 베트남 현지 서버에도 동일한 AD 도메인 또는 신뢰 관계가 구성되어 있어야 한다.
- Java 애플리케이션 서버의 PowerShell 실행 권한: WAS(Tomcat/JBoss 등)가 운영되는 Linux/Windows 서버에서 PowerShell 스크립트를 `Runtime.exec()` 또는 `ProcessBuilder`로 실행할 수 있는 환경 구성 필요.

#### 3-2-1. 구현 난이도

| 구현 항목 | 난이도 | 사유 |
|---|---|---|

| 정크메일 규칙 관리 (add/remove Sender) | ★★★☆☆ (중간) | 파라미터 파싱 로직 단순, 단 실시간 적용 및 오류 처리 보완 필요 |
| 배포 그룹 이메일 추가 (`setDistributionGroup.ps1`) | ★★★☆☆ (중간) | 재시도 로직(최대 5회) 내장되어 있어 비교적 안정적, Exchange 복제 지연 대응 포함 |
| 메일박스 프로비저닝 (`setMailbox.ps1`) | ★★★☆☆ (중간) | 단순 Set-Mailbox 명령이나, DB 연계 및 롤백 전략 설계 필요 |
| covimail UI(JSP) Exchange 연동 | ★★☆☆☆ (낮음) | 기존 Spring MVC 패턴 그대로 활용 가능, 컨트롤러 및 서비스 레이어 추가 구현 필요 |

#### 3-2-2. 제약 사항

1. Exchange On-premise 버전 의존성: `setSession.ps1`의 `-ConfigurationName Microsoft.Exchange`는 Exchange On-premise 전용 설정이다. Exchange Online(Microsoft 365)의 경우 EXO V3 PowerShell 모듈 및 OAuth2 인증 방식으로 변경이 필요하다.
2. PowerShell 버전 호환성: `New-PSSession`을 사용하는 기존 방식은 PowerShell 5.x 기반이다. PowerShell 7.x(Core) 환경에서는 `New-PSSession`의 Exchange 연결 방식이 다를 수 있으므로 버전 확인이 필요하다.
3. 세션 수명 관리: `getSession.ps1`에서 `$ExSession.State`를 확인하는 구조이나, 세션이 Broken/Closed 상태일 경우의 자동 재연결 로직이 `setSession.ps1`에 포함되어 있지 않으므로 서비스 레이어에서 보완 필요.
4. Linux WAS 환경 제약: Java 애플리케이션이 Linux 서버에서 운영될 경우, PowerShell Core(pwsh)가 설치되어 있어야 하며, `New-PSSession`의 Kerberos 인증 지원을 위해 `krb5` 라이브러리 설정이 필요하다.

#### 3-2-3. 대안 제시

| 현재 방식 | 대안 | 비고 |
|---|---|---|
| PowerShell Remote Session (Kerberos) | Microsoft Graph API (OAuth2/REST) | Exchange Online 환경 시 필수 대안. 단, On-premise에서는 Hybrid Connector 필요 |
| `Runtime.exec()` / `ProcessBuilder`로 PS 실행 | EWS(Exchange Web Services) Java API | Exchange On-premise에서 직접 Java 연동 가능, 단 EWS는 Microsoft가 지원 축소 중 |
| 개별 스크립트 파일 실행 방식 | Exchange PowerShell 통합 실행 서비스화 | Spring Service Bean으로 PS 명령어 풀 관리 및 재사용성 향상 |

### 3-5. API 및 서비스 인터페이스 영향도

#### 3-5-1. 내부 모듈 간 인터페이스

```
[covimail UI (JSP/Spring MVC)]
        |
        | HTTP Request
        v
[covimail Controller Layer]
        |
        | Service Method Call
        v
[coviexchange Service Layer]
        |
        | ProcessBuilder / Runtime.exec()
        v
[PowerShell Script 실행 (setSession, setMailbox, setDistributionGroup, setMailboxJunkEmailConfiguration 등)]
        |
        | WinRM / PSRemoting (TCP 5985/5986, Kerberos)
        v
[Exchange On-premise Server (EMS: Exchange Management Shell)]
```

- covimail -> coviexchange 호출 방식: 현재 소스 기준으로 Java에서 PowerShell 스크립트를 직접 실행하는 방식(`ProcessBuilder` 또는 `Runtime.exec()`)으로 추정된다. 서비스 간 REST API 호출 방식 여부는 추가 소스 확인 필요.
- 트랜잭션 전파(Propagation): Java -> PowerShell 실행 구조상 Spring의 `@Transactional`이 PowerShell 실행 단위까지 전파되지 않는다. PowerShell 스크립트 실행 결과(`END_OF_SCRIPT` 출력 여부)를 Java 측에서 파싱하여 성공/실패를 판단하는 구조이므로, 실패 시 수동 보상 트랜잭션(Compensating Transaction) 전략이 필요하다.

#### 3-5-2. 외부 인터페이스

| 외부 시스템 | 연동 방식 | 확인 사항 |
|---|---|---|
| Exchange On-premise | PowerShell Remote Session (Kerberos/WinRM) | AD 계정 권한, 방화벽 포트 개방 필요 |
| Active Directory | Kerberos 인증 (`setSession.ps1`) | 서비스 계정(serviceUserName) AD 그룹 정책 확인 필요 |
| 인사 DB (외부) | PM 요청서 기준 연동 언급 | 추가 소스 및 인터페이스 정의서 필요 |

### 3-6. 데이터 정합성 및 트랜잭션 전략

#### 3-6-1. 핵심 문제점

Java 애플리케이션(Spring)과 PowerShell Exchange 명령 실행 간에는 분산 트랜잭션이 기본적으로 지원되지 않는다. 즉, Java DB 트랜잭션이 Commit된 이후 PowerShell 스크립트 실행이 실패하면 데이터 불일치가 발생한다.

#### 3-6-2. 대응 전략

| 시나리오 | 문제점 | 대응 방안 |
|---|---|---|
| DB Insert 성공 -> PS 스크립트 실행 실패 | Exchange 설정 누락, DB와 Exchange 데이터 불일치 | PS 실행 후 `END_OF_SCRIPT` 확인 전까지 DB Commit 보류, 또는 실패 시 DB Rollback |
| PS 스크립트 실행 성공 -> DB Commit 실패 | Exchange에는 설정 적용, DB에는 미반영 | 보상 트랜잭션(PS 역실행) 또는 별도 이력 테이블에 Pending 상태 기록 후 재처리 |
| 배포 그룹 생성 중 Exchange 복제 지연 | `setDistributionGroup.ps1` 내 재시도 로직(최대 5회, 1초 간격) 존재 | 현재 구현된 재시도 로직으로 일부 대응 가능. 단, 5회 초과 시 명시적 오류 처리 필요 |
| 정크메일 설정 add/remove 동시 요청 | Race Condition 발생 가능 | 사용자별 요청 직렬화(Queue) 또는 Redis Distributed Lock 적용 권장 |

#### 3-6-3. ACID 원칙 적용 방안

- Atomicity: Java DB 트랜잭션과 PS 실행을 묶기 위해 "PS 실행 성공 확인 후 DB Commit" 패턴 적용.
- Consistency: PS 스크립트 실행 결과의 `END_OF_SCRIPT` 토큰 및 오류 메시지를 파싱하여 결과 검증.
- Isolation: 동일 메일박스에 대한 동시 설정 변경 방지를 위해 Redis 분산 락 적용 검토.
- Durability: PS 실행 요청 및 결과를 별도 이력 테이블(`exchange_cmd_history`)에 기록하여 장애 복구 지원.
