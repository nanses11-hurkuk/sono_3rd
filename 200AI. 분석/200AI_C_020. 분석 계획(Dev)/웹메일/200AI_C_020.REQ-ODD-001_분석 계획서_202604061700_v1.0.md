## (3) 분석 계획 범위

### 3-1. 영향도 분석 (기존 모듈 영향도 / DB·인프라 변경점)

#### (가) smarts4j_msapi 모듈 구조 — 소스코드 직접 분석

`smarts4j_msapi` 모듈의 패키지 루트는 `egovframework.msapi`이며, 전체 Java 소스 파일 기준으로 196개 파일이 포함되어 있습니다. 모듈은 크게 API 파사드, 공통 컨텍스트, DTO, 엔드포인트 정의, 서비스 구현, 유틸리티의 여섯 계층으로 구성됩니다.

**[API 파사드 계층 — `api/CoviMSAPI.java`]**

`CoviMSAPI`는 외부(Smarts4j 내부 다른 모듈)에서 Microsoft 서비스를 호출하기 위한 단일 진입점입니다. 소스코드 내 주석에 명시된 사용 예시는 다음과 같습니다.

```java
CoviMSAPI coviMSAPI = new CoviMSAPI();
ServiceContext serviceContext = new ServiceContext();
serviceContext.setAccessToken("XXX");
serviceContext.setBaseUrl("http://localhost:8082");
coviMSAPI.setServiceContext(serviceContext);
coviMSAPI.service("graph").org().viewUser(param);
```

`CoviMSAPI`가 제공하는 서비스 영역은 `org()`, `mail()`, `contact()`, `app()`, `calendar()`, `teamwork()`, `onedrive()`, `todo()` 총 8개입니다. 각각 `ServiceFactory`를 통해 구체 구현체를 획득합니다.

**[공통 컨텍스트 계층 — `common/`]**

- **`CoviMSAPIEnums`**: 모듈 유형을 `GRAPH`와 `EXCHANGE` 두 가지로 정의합니다. 이 열거값이 `ServiceFactory`의 분기 기준이 됩니다.

- **`ServiceContext`**: 연결 설정 전체를 담는 컨텍스트 객체입니다. 주요 필드는 다음과 같습니다.
  - 공통: `module` (GRAPH/EXCHANGE), `baseUrl`, `domainId`
  - Exchange On-premise AD 접속: `adUrl`, `adUser`, `adPwd`, `adBaseDn`, `adRootName`
  - Exchange 관리: `exchangeUser`, `exchangePwd`, `exchangeUrl`, `exchangeStorageServer`, `exchangeIsResetMailAddress`
  - EWS: `ewsUrl`, `ewsUser`, `ewsPwd`, `ewsDomain`, `ewsVersion`
  - PowerShell (On-premise): `pwshPoolSize`
  - PowerShell (Graph/Certificate): `pwshCertPath`, `pwshCertPassword`, `pwshAppID`, `pwshOrganization`

- **`GraphServiceContext`**: `ServiceContext`를 상속하며 OAuth2 액세스 토큰(`accessToken`)을 추가로 보유합니다. Graph API 호출 시 이 서브클래스를 사용합니다.

**[ServiceFactory — Strategy 패턴 구현]**

`ServiceFactory`는 `ServiceContext.getModule()` 값에 따라 구체 구현체를 선택합니다. 각 서비스별 지원 모듈은 다음과 같습니다.

| 서비스 | GRAPH 구현체 | EXCHANGE 구현체 | 비고 |
|--------|-------------|-----------------|------|
| OrgSvc | GraphOrgSvcImpl | ExchangeOrgSvcImpl | 두 모듈 모두 지원 |
| MailSvc | GraphMailSvcImpl | ExchangeMailSvcImpl | 두 모듈 모두 지원 |
| ContactSvc | GraphContactSvcImpl | ExchangeContactSvcImpl | 두 모듈 모두 지원 |
| AppSvc | GraphAppSvcImpl | 미지원 | GRAPH 전용 |
| CalendarSvc | CalendarSvcImpl | CalendarSvcImpl | 구현체 단일화 (모듈 무관) |
| OneDriveSvc | GraphOneDriveSvcImpl | 미지원 | GRAPH 전용 |
| TeamWorkSvc | GraphTeamWorkSvcImpl | 미지원 | GRAPH 전용 |
| ToDoSvc | GraphToDoSvcImpl | 미지원 | GRAPH 전용 |

AppSvc, OneDriveSvc, TeamWorkSvc, ToDoSvc는 EXCHANGE 모듈을 지정하면 `UnsupportedOperationException`이 발생합니다. 이는 해당 서비스들이 Microsoft 365 Graph API에만 존재하기 때문입니다.

**[엔드포인트 정의 계층 — `endpoint/`]**

엔드포인트는 Java `enum`으로 정의되어 URL, HTTP 메서드, 요청 DTO 클래스, EWS 응답 DTO 클래스, Graph 응답 DTO 클래스를 함께 보유합니다. `BaseEndPointSvc` 인터페이스로 추상화되어 있으며, `RestApiUtil`이 이 인터페이스를 통해 HTTP 요청을 일관되게 처리합니다.

확인된 엔드포인트 열거 파일은 다음과 같습니다.

- **`MailEndPoint`**: 메일 관련 45개 이상의 엔드포인트. 폴더 CRUD, 메시지 CRUD(발송·회신·전달·재발송·삭제·이동·복사·회수·내보내기·가져오기), 자동응답·규칙·범주·구독·스팸·용량 조회 등 전체 웹메일 기능 포함.
- **`CalendarEndPoint`**: 일정 CRUD(생성·수정·삭제·수락·거부·취소·미정), 기간별 조회, 자원 중복 체크, 첨부파일 다운로드 등 15개 엔드포인트.
- **`ContactEndPoint`**: 연락처 폴더 CRUD, 연락처 CRUD(개인·그룹), 이동·복사 등 15개 엔드포인트.
- **`OrgEndPoint`**: 사용자 CRUD(추가·수정·비활성화·비밀번호변경·라이선스변경·그룹변경·관리자설정·메일포워딩), 그룹 CRUD(추가·수정·삭제·멤버설정·관리자설정) 등 16개 엔드포인트.
- **`AppEndPoint`**: 앱 조회, 암호인증서 생성, Access Token 생성(AppAuth·UserAuth·DelegateAuth), 라이선스 목록 조회 등 6개 엔드포인트. OAuth2 인증 3가지 흐름 모두 지원.
- **`TeamWorkEndPoint`**: Teams 팀·채널 CRUD, 멤버 관리, Presence(접속상태) 조회 등 11개 엔드포인트.
- **`OneDriveEndPoint`**: 폴더 목록 조회, 파일 검색, 파일 다운로드 3개 엔드포인트.
- **`ToDoEndPoint`**: 사용자 To-Do 작업 목록 및 작업 조회 2개 엔드포인트.
- **`AdminEndPoint`**: Exchange MDB(Mailbox Database) 목록 조회 1개 엔드포인트.
- **`ConfigEndPoint`**: coviexchange 서버에 Exchange 설정 등록(`/v1/config/add`) 1개 엔드포인트.

**[유틸리티 계층 — `util/`]**

- **`RestApiUtil`**: `smarts4j_msapi`에서 모든 HTTP 통신을 담당하는 핵심 유틸리티입니다. 직접 확인된 동작은 다음과 같습니다.
  - Spring `RestTemplate`을 사용하여 coviexchange REST API 서버에 HTTP 요청을 전송합니다.
  - 최대 5회 자동 재시도 로직이 내장되어 있습니다(`maxRetries = 5`).
  - 응답에서 `NOTFOUND_CONFIG` 오류 코드를 수신하면 자동으로 `ConfigEndPoint.ADD_CONFIG`를 호출하여 Exchange 설정을 등록한 뒤 재시도합니다.
  - URL은 `serviceContext.getBaseUrl() + serviceContext.getModule().toString().toLowerCase() + endpointUrl` 형식으로 조합됩니다. 예시: `http://localhost:8082/exchange/v1/mail/{email}/selectMessageList` 또는 `http://localhost:8082/graph/v1/mail/{email}/selectMessageList`
  - `HttpComponentsClientHttpRequestFactory`를 사용하나, 현재 소스코드에서 `readTimeout`과 `connectTimeout` 설정 라인이 주석 처리된 상태입니다. 베트남 Latency 환경에서 타임아웃 미설정은 높은 위험 요소입니다.

- **`DataConverter`**: EWS 응답 데이터와 Graph API 응답 데이터를 Smarts4j 내부 DTO 형식으로 변환합니다. 파일 크기가 13,960 bytes로 복잡한 변환 로직을 포함합니다.

- **`GWMailUtil`**: 메일 처리 관련 유틸리티. 10,963 bytes 규모입니다.

- **`GraphUtils`**: MS Graph API 호출 관련 공통 유틸리티. 액세스 토큰 처리 등을 포함합니다.

**[DTO 계층 — `dto/`]**

DTO는 요청(GW 접두어)과 응답(MS 접두어)으로 분리되며, 연동 대상(Graph/EWS)에 따라 하위 패키지로 구분됩니다.

- **요청 DTO**: `GWUserDTO`에서 `@JsonAlias` 어노테이션을 통해 Smarts4j 내부 필드명(예: `MailAddress`, `EmpNo`, `Department`)을 MS API 필드명으로 매핑하는 방식을 사용합니다. 이는 Smarts4j 기존 인사 DB 필드명과 MS Graph API 필드명의 불일치를 처리하기 위한 설계입니다.
- **응답 DTO**: EWS 응답용(`dto/response/org/ews/`, `dto/response/mail/`)과 Graph API 응답용(`dto/response/org/graph/`, `dto/response/app/graph/`, `dto/response/teamwork/graph/` 등)으로 이중 구조를 유지합니다.

#### (나) 기존 모듈 영향도

**smarts4j_coviexchange와의 관계**: `smarts4j_msapi`는 `smarts4j_coviexchange` REST API 서버를 HTTP로 호출하는 클라이언트 라이브러리입니다. `smarts4j_coviexchange`가 서버 역할을 하고, `smarts4j_msapi`가 클라이언트 역할을 합니다. `RestApiUtil`의 URL 조합 방식(`baseUrl + module + endpoint`)과 `ConfigEndPoint.ADD_CONFIG` 자동 호출 로직이 이를 명확히 보여줍니다.

**smarts4j_covimail과의 관계**: `smarts4j_covimail` 어드민 모듈이 Exchange 연동 기능이 필요한 경우, `smarts4j_msapi`를 통해 `smarts4j_coviexchange` 서버로 요청을 전달하는 계층 구조가 됩니다. covimail은 `smarts4j_msapi`를 직접 의존성으로 추가하거나, 또는 별도 REST 호출로 coviexchange를 직접 호출하는 두 가지 방식 중 하나를 선택해야 합니다.

**Smarts4j 내부 그룹웨어 모듈과의 관계**: 웹메일 UI 모듈, 캘린더 모듈, 주소록 모듈, Teams 연동 모듈 등 Smarts4j 그룹웨어의 각 기능 모듈들은 `smarts4j_msapi`의 `CoviMSAPI` 파사드를 통해 MS 서비스를 호출합니다. 이 라이브러리가 없거나 버전이 맞지 않으면 MS 연동 기능 전체가 동작하지 않습니다.

#### (다) DB 및 인프라 변경점

- **DB 변경 없음**: `smarts4j_msapi` 모듈은 순수 클라이언트 라이브러리로, DB 접근 코드가 존재하지 않습니다. 모든 데이터는 MS 서비스(Exchange Store, Microsoft 365)에 저장됩니다.
- **인프라 추가 필요**: Graph API를 사용하는 경우 Azure AD 애플리케이션 등록(`appId`, `tenantId`, `clientSecret`)이 필요합니다. `GWTenantDTO`에서 이 필드들이 확인됩니다. 베트남 현지 환경에서 Azure 서비스에 대한 네트워크 접근성(방화벽 정책, HTTPS 포트 443 개방)을 사전 확인해야 합니다.

### 3-2. 기술적 타당성 (구현 난이도 / 제약 사항 / 대안 제시)

#### (가) 구현 난이도

`smarts4j_msapi`는 소스코드 분석 결과 이미 상당 부분 구현이 완료된 상태입니다. Graph API와 EWS 양쪽에 대한 구현체가 모두 존재하며, 엔드포인트 정의, DTO, 유틸리티까지 갖추어진 성숙한 클라이언트 라이브러리입니다. 따라서 신규 개발보다는 베트남 현지 환경 적용 및 통합 테스트가 주요 작업입니다. 다만 아래와 같은 기술적 복잡도 요소가 존재합니다.

- `RestApiUtil`의 타임아웃 설정이 주석 처리된 상태입니다. 베트남 네트워크 환경에서 `readTimeout`과 `connectTimeout` 미설정은 응답 지연 시 스레드가 무한 대기하는 장애로 이어질 수 있습니다. 이 설정의 활성화 및 적절한 값 설정이 반드시 필요합니다.
- `maxRetries = 5`가 소스코드에 하드코딩되어 있습니다. 이 값은 외부 설정(property 파일 또는 Redis 설정)으로 분리하는 것이 바람직합니다. 소스코드 내 주석(`> property?`)에서도 개발자가 이 점을 인지하고 있음을 확인할 수 있습니다.
- OAuth2 인증 3가지 흐름(App Auth, User Auth, Delegate Auth)을 지원하나, 베트남 현지 Azure AD 테넌트 환경에서 각 흐름의 실제 동작 여부를 검증해야 합니다.

#### (나) 제약 사항

- **그래프 API Throttling**: Microsoft Graph API는 요청 빈도 제한(Throttling)이 있습니다. Smarts4j 전체 사용자의 동시 요청이 Graph API 제한을 초과하는 경우 429(Too Many Requests) 오류가 발생합니다. `RestApiUtil`의 재시도 로직은 현재 모든 오류에 동일하게 적용되나, 429 오류의 경우 `Retry-After` 헤더를 읽어 대기 후 재시도해야 하며 이 처리가 현재 구현에서 확인되지 않습니다.
- **GRAPH 전용 기능 제약**: AppSvc, OneDriveSvc, TeamWorkSvc, ToDoSvc는 GRAPH 모듈에서만 동작합니다. 베트남 현지 환경이 Exchange On-premise만으로 구성된 경우 이 네 가지 서비스를 사용할 수 없습니다. Microsoft 365 라이선스 확보가 선행되어야 합니다.
- **CalendarSvc 단일 구현체**: `ServiceFactory`에서 `CalendarSvc`는 GRAPH/EXCHANGE 구분 없이 `CalendarSvcImpl` 단일 구현체를 반환합니다. `CalendarSvcImpl`(30,118 bytes)이 내부적으로 두 프로토콜을 모두 처리하는지, 또는 한 가지만 지원하는지 추가 코드 분석이 필요합니다.
- **`@JsonAlias` 필드 매핑 관리**: `GWUserDTO`에서 `@JsonAlias`로 Smarts4j 내부 필드명과 MS API 필드명을 매핑합니다. 인사 DB 필드명이 변경되거나 MS API 필드 스펙이 변경되는 경우 이 매핑을 추적하여 수정해야 합니다.
- **베트남 Latency**: `RestTemplate`은 기본적으로 블로킹 I/O를 사용합니다. 베트남 현지에서 MS 서비스(Graph API 서버 또는 Exchange Server)까지의 네트워크 RTT가 높은 경우 응답 대기 시간이 길어집니다. 타임아웃 미설정 상태에서는 모든 서버 스레드가 대기 상태로 고갈될 수 있습니다.

#### (다) 대안 제시

- **타임아웃 즉시 활성화**: `RestApiUtil` 내 주석 처리된 `factory.setReadTimeout()` 및 `factory.setConnectTimeout()` 라인을 활성화하고, 베트남 네트워크 측정 결과를 기반으로 적절한 값을 설정해야 합니다. 초기값으로 connectTimeout 5,000ms, readTimeout 30,000ms를 제안합니다.
- **재시도 횟수 외부화**: `maxRetries` 값을 `properties` 파일 또는 Redis 설정으로 분리하여 운영 중 조정이 가능하도록 합니다.
- **Graph API Throttling 대응**: `RestApiUtil`의 재시도 로직에 HTTP 429 응답 시 `Retry-After` 헤더를 읽어 대기하는 전용 처리 로직을 추가합니다.
- **비동기 호출 전환 검토**: 베트남 Latency 환경에서 대용량 데이터(메일 목록, 첨부파일) 조회 시 Spring의 `AsyncRestTemplate` 또는 WebClient(Reactive) 전환을 중장기 개선 방안으로 검토합니다.

### 3-6. 데이터 정합성 및 트랜잭션 전략

**[멱등성 및 재시도 전략]**

`RestApiUtil`의 최대 5회 자동 재시도 로직은 네트워크 일시 장애 시 유용하나, POST 요청(메일 발송, 사용자 생성 등)에 동일하게 적용될 경우 동일 데이터가 중복 처리될 위험이 있습니다. 멱등성이 보장되지 않는 작업(메일 발송, 일정 생성, 사용자 추가 등)에는 재시도를 제한하거나, MS 서버 측 중복 방지 처리(`If-Match` 헤더 활용, `internetMessageId` 중복 체크 등)를 함께 적용해야 합니다.

**[Smarts4j DB와 MS 서비스 간 정합성]**

조직 관리 시 Smarts4j 인사 DB와 MS AD/Azure AD의 사용자 정보가 동기화되어야 합니다. `smarts4j_msapi`가 MS AD/Azure AD에 변경을 반영한 후, Smarts4j DB 업데이트 전에 장애가 발생하면 두 시스템 간 데이터 불일치가 발생할 수 있습니다. 이를 해결하기 위해 변경 이력(이벤트 로그)을 Smarts4j DB에 먼저 기록하고, MS 연동 완료 후 상태를 업데이트하는 Outbox 패턴 적용을 검토합니다.

**[V1·V2 계열사 데이터 격리]**

`ServiceContext`의 `domainId` 필드와 Exchange 연동의 `adBaseDn` 설정을 통해 계열사 간 데이터를 격리합니다. `smarts4j_msapi` 호출 코드에서 도메인 컨텍스트가 혼재되지 않도록, 호출 전 반드시 `ServiceContext.setDomainId()`를 통해 대상 도메인을 명시해야 합니다. `smarts4j_coviexchange`의 `ResourceManager`가 `domainId`를 키로 설정을 관리하므로, domainId 값이 두 모듈 간에 일치해야 합니다.

**[액세스 토큰 만료 처리]**

`MSGraphAccessTokenDTO`에는 `accessToken`과 `expiresAt` 두 필드가 있습니다. 그러나 현재 소스코드에서 토큰 만료 시 자동 갱신(Token Refresh)을 처리하는 로직이 확인되지 않습니다. 운영 환경에서 장시간 동작하는 배치 작업이나 세션 유지 중 토큰이 만료될 경우 401 오류가 발생합니다. 토큰 만료 전 자동 갱신 처리가 필요합니다.

### 3-7. 기술적 위험도 점수

소스코드 분석 결과를 기반으로 산출한 위험도 평가는 다음과 같습니다.

| 위험 항목 | 위험 수준 | 근거 | 대응 방안 |
|-----------|----------|------|-----------|
| 타임아웃 미설정으로 인한 스레드 고갈 | 높음 | `RestApiUtil.java`에서 `readTimeout`/`connectTimeout` 설정 라인이 주석 처리된 상태. 베트남 Latency 환경에서 응답 지연 시 모든 서버 스레드가 블로킹될 위험 | 즉시 타임아웃 값 활성화 필요. 적정 값(connectTimeout 5s, readTimeout 30s) 설정 |
| Graph API Throttling 미처리 | 높음 | 재시도 로직이 429 응답에 대한 `Retry-After` 헤더 처리 없이 단순 횟수 기반 재시도를 수행. 전체 사용자 동시 요청 시 Throttling 발생 가능 | 429 전용 재시도 처리 로직 추가 |
| OAuth2 액세스 토큰 자동 갱신 미구현 | 높음 | `MSGraphAccessTokenDTO.expiresAt` 필드 존재하나 만료 시 자동 갱신 코드 미확인 | Token Refresh 처리 로직 구현 필요 |
| AppSvc·OneDriveSvc·TeamWorkSvc·ToDoSvc EXCHANGE 모듈 미지원 | 중간 | `ServiceFactory`에서 EXCHANGE 모듈 지정 시 `UnsupportedOperationException` 발생. 베트남 환경이 M365 미도입 경우 해당 서비스 전체 사용 불가 | M365 라이선스 확보 선행. 미지원 기능은 요구사항에서 제외 처리 |
| maxRetries 하드코딩 | 낮음 | `RestApiUtil.java` 소스코드에 `int maxRetries = 5`가 하드코딩. 운영 중 조정 불가 | 외부 설정으로 분리(properties 또는 Redis) |
| 멱등성 없는 POST 재시도 위험 | 중간 | 메일 발송·일정 생성 등 POST 요청도 최대 5회 재시도. 중복 처리 위험 | 작업 유형별 재시도 정책 분리 |
| CalendarSvc 단일 구현체 미확인 | 낮음 | GRAPH/EXCHANGE 구분 없이 단일 `CalendarSvcImpl`을 반환하나, 내부에서 두 프로토콜을 모두 지원하는지 미확인 | CalendarSvcImpl 소스 추가 분석 필요 |
| V1·V2 domainId 혼재 위험 | 중간 | 호출 코드에서 `ServiceContext.domainId` 설정을 누락하면 잘못된 도메인의 Exchange 설정이 사용됨 | 도메인 컨텍스트 주입 검증 로직 및 통합 테스트 필수 |

**종합 기술적 위험도**: **4/5** — 구현이 상당 부분 완료되어 있으나, 타임아웃 미설정과 Graph API Throttling 미처리, 토큰 자동 갱신 미구현이 운영 환경에서 치명적 장애로 이어질 수 있어 높은 위험도를 유지합니다.
