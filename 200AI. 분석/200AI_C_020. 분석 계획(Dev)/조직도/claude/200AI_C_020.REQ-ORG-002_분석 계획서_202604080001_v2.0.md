### GAP 대응 방안

1. **AD 통합 선행 조건 정의**: V1·V2 AD 통합은 인프라 팀의 AD 마이그레이션 작업이 선행되어야 한다. 개발팀은 인프라 팀과 AD 통합 완료 시점을 협의하고, 통합 전까지는 V1·V2 각각의 AD에 대해 동기화를 수행하는 분기 로직을 구현한다.
2. **AD 배포단위 구조 확정 대기**: 배포단위(부서, 직위만 구성)가 UTI 내부 검토 중이므로, 확정 후 AD Object 생성 로직 구현. 현재 분석 단계에서는 Smarts4j 표준 AD 구조를 기준으로 설계하고 확정 시 조정.
3. **Graph API 방식 적용**: Smarts4j의 `smarts4j.coviexchange` 모듈에서 Microsoft Graph API(`User.Read.All`, `Group.ReadWrite.All`, `Directory.Read.All` 스코프)를 사용하여 Entra ID와 직접 통신. AADC 경유 방식(AD → Entra ID 자동 싱크)이 주 경로이므로, Graph API는 보완적 검증 또는 라이선스 조회 목적으로만 사용.
4. **개발 테넌트 확보 요청**: 개발 테넌트 미제공 시 M365 연동 테스트가 불가하므로, 프로젝트 초기에 UTI 측에 개발 테넌트 제공을 공식 요청한다.
5. **메일 도메인 병행 기간 처리**: utikorea.com → utivina.com 전환 기간 중 두 도메인의 메일 주소가 공존하므로, 사용자 마이그레이션 시 `ProxyAddresses` 관리 전략을 수립한다.

### 3-2. 기술적 타당성

#### 구현 난이도

| 항목 | 난이도 | 근거 |
|------|--------|------|
| AADC 기반 AD → Entra ID 자동 싱크 활용 | 하~중 | 이미 AADC 운영 중이므로 AD 변경 내용이 Entra ID에 자동 반영. GW 배치의 역할은 AD까지만 반영하면 됨 |
| AD Object 생성/수정/비활성화 로직 | 중 | 배포단위(부서·직위) + 조직단위(OU·사용자) 이중 구조, 사용자 소속 그룹 매핑 |
| V1·V2 통합 AD 구조 적용 | 고 | AD 통합이 선행되어야 하며, 기존 1레벨 OU 구조에서 표준 구조로 전환 시 기존 AD 정책/권한 영향 검토 필요 |
| Exchange 메일그룹 동기화 | 중 | 부서/직위 코드 기반 메일그룹 생성·갱신·삭제 로직. 퇴사자 사서함 비활성화 포함 |
| Graph API 연동 | 중 | Microsoft Graph SDK 사용, 인증 토큰 만료 처리, Throttling 대응 필요 |
| 개발 테넌트 미확보 | 고 위험 | 테스트 불가 상황. 본 위험은 외부 요인(UTI 의사결정)이므로 조기 협의 필요 |

#### 제약 사항

1. **AD 통합 미완료**: V1·V2 AD 통합이 완료되지 않으면 통합 기준의 AD 동기화 개발 착수 불가.
2. **AD 배포단위 미확정**: "배포단위는 부서, 직위만 구성 - UTI 내부 검토 후 확정" 상태. 확정 전 AD Object 생성 로직 구현 불가.
3. **개발 테넌트 없음**: M365 Graph API, Exchange Online 연동 테스트 환경 부재.
4. **Entra ID Throttling**: Graph API 대량 호출 시 Throttling(429 Too Many Requests) 발생 가능. 지수 백오프(Exponential Backoff) 구현 필요.
5. **utikorea.com → utivina.com 도메인 전환 기간**: 전환 기간 중 두 도메인 계정이 공존하므로, 그룹웨어 사용자 메일주소 일관성 확보 전략 필요.

#### 대안 제시

- **개발 테넌트 미확보 시**: Microsoft 365 개발자 프로그램(Dev Tenant) 신청하여 임시 테넌트 확보. 운영 테넌트와 동일한 도메인 구성은 불가하나 API 연동 테스트 가능.
- **AD 통합 지연 시**: V1·V2 각각의 AD에 대해 독립적으로 동기화 배치를 구성하고, 통합 완료 후 단일 AD 경로로 전환하는 2단계 접근 방식 적용.
