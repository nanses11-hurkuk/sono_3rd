### GAP 대응 방안

1. **이기종 DB 연동**: Smarts4j의 `CoviMapperOne` 외에 MSSQL 전용 DataSource를 별도 `applicationContext-mssql.xml`로 구성하고, MyBatis SqlSessionFactory를 분리 등록한다. 영림원 MSSQL View는 `covi_syncdata.v_orgs_orgperson`, `covi_syncdata.v_orgs_orgdept`에 대응하는 외부 View로 읽어 중간 테이블(`orgs_hr_person`, `orgs_hr_dept`)에 삽입한다.
2. **V1 수동 처리**: V1 계열사는 관리자 화면을 통해 사용자/부서 데이터를 직접 등록·수정하며, 동기화 배치 대상에서 제외(`CompanyCode = V1` 조건 분기).
3. **코어 수정 최소화**: Smarts4j Extension Point(Plugin Architecture)를 활용하여 `OrgSyncCon`의 배치 진입점만 확장하고, 핵심 동기화 로직(insertHR* → insertOrg* → insertTempObject* → insertCompareObject* 순서)은 표준 `SYNC_FieldMapping.sql` 구조를 그대로 활용한다.
4. **사용자 식별키 미확정 대응**: UTI 내부 확인이 완료될 때까지 `EmpNo`를 임시 식별키로 사용하되, 확정 후 `UserCode` 매핑을 검토한다.

| 사용자 식별키 매핑 | 고 | 식별키 미확정 상태, 확정 후 DB 스키마 및 매핑 재검토 필요 |
| 퇴사자 처리 로직 | 중 | IsHR=Y인데 인사데이터에 없는 경우 자동 감지 및 관리자 알림 로직 |

#### 제약 사항

1. **영림원 View 스키마 미확정**: View 컬럼 목록 및 데이터 타입 확인 전까지 FieldMapping.sql 완성 불가. 영림원 측 스키마 정의서 요청 필요.
2. **사용자 식별키 미확정**: UTI 내부 확인 후 `UserCode` 또는 `EmpNo` 결정. 개발 착수 전 필수 확정 항목.
3. **개발 환경 MSSQL 접근**: 개발 서버에서 영림원 테스트 DB 접근 가능 여부 확인 필요 (방화벽, VPN 등).
4. **베트남 현지 네트워크 지연**: GW WAS ↔ 영림원 MSSQL 간 네트워크 Latency 측정 후 Connection Timeout 값 설정 필요.

#### 대안 제시

- **영림원 View 스키마 확보 전 대안**: 현재 `orgs_hr_person`, `orgs_hr_dept` 스키마(`HR\Scheme.md`)를 영림원 측에 제공하고, View가 동일 컬럼명을 제공하도록 요청한다. 불일치 컬럼은 FieldMapping의 `sourceField`에 SQL 변환 표현식(QUERY/FUNCTION 타입)으로 처리한다.
- **네트워크 지연 대안**: 영림원 View 데이터를 GW DB 내 별도 Mirror 테이블에 사전 복제(Linked Server 방식)하고, GW 배치는 Mirror 테이블을 참조하는 방식으로 전환 가능.
