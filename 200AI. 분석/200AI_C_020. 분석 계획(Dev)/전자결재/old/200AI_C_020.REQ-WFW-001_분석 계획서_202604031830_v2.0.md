## (3) 분석 계획 범위

### 3-1. 영향도 분석

#### 기존 모듈 영향도

소스 코드 분석 결과, 계열사간 결재(사간결재) 기능 구현 시 다음 모듈에 직접적인 영향이 있습니다.

| 영향 모듈 | 소스 위치 | 영향 내용 |
|---|---|---|
| 결재 프로세스 엔진 | `coviflow.form.service.ApvProcessSvc` / `ApvProcessCon.java` | 기안, 승인, 반려, 합의 등 프로세스 시작 시 계열사 간 라우팅 분기 로직 추가 필요 |
| 결재선 자동설정 | `coviflow.approvalline.service.AutoApprovalLineService` | `setFormAutoApprovalData()` 메서드에서 계열사(V1, V2) 간 조직도 기반 자동결재선 설정 시 Multi-Domain 분기 처리 필요 |
| 양식 관리자 | `coviflow.form.service.FormSvc` / `FormCon.java` | `selectDomainData()`, `selectForm()` 메서드에서 계열사별 양식 분리 조회 로직 확인 필요 |
| 계열사 정보 관리 | `coviflow.common.service.EntInfoSvc` | `getEntinfoListData()` 메서드로 계열사 도메인 목록 조회 기능이 이미 존재함 |
| 계열사 통계 | `coviflow.admin.web.AffiliatesAggregateInformationCon.java` | 계열사별 결재 건수/처리 시간 통계 기능이 이미 구현되어 있어 사간결재 데이터도 통계에 포함되어야 함 |

#### DB 변경점

소스 코드의 SQL 매핑 파일 분석 결과를 기반으로 다음 DB 변경점을 확인하였습니다.

- `form_entInfo.xml`의 `selectEntInfoList` 쿼리에서 `sys_object_domain` 테이블과 `sys_object_group` 테이블을 조인하여 계열사 도메인 정보를 조회하고 있습니다. 계열사 구분을 위한 `DomainCode` 체계가 이미 존재하므로 V1, V2 도메인 코드 등록이 필요합니다.
- `admin_affiliatesAggregateInformation.xml`에서 `jwf_workitem`과 `jwf_process` 테이블을 `CompanyCode`로 그룹핑하여 계열사별 통계를 산출하고 있습니다. 사간결재 문서의 `CompanyCode` 기록 정합성 확인이 필요합니다.
- 사간결재 라우팅 테이블 및 권한 매핑 테이블 구성을 검토해야 합니다.

#### 인프라 변경점

- 베트남 현지 네트워크 환경에서 V1↔V2 간 결재 데이터 전송 시 지연(Latency) 최소화를 위한 캐싱 전략이 필요합니다.
- `FormCon.java`에서 `RedisDataUtil`을 import하여 Redis Cache를 활용하고 있으므로, 결재선 정보 캐싱에 동일한 방식을 적용할 수 있습니다.

### 3-2. 기술적 타당성

#### 구현 난이도

- 소스 분석 결과, `EntInfoSvc.getEntinfoListData()` 및 `AffiliatesAggregateInformationCon.java`에서 이미 계열사 구분 체계(`CompanyCode`, `DomainCode`)가 구현되어 있습니다.
- `AutoApprovalLineService.setFormAutoApprovalData()`에서 자동결재선 설정 시 `strSessionDNCode`(도메인 코드) 파라미터를 이미 받고 있어, 사간결재 확장이 가능한 구조입니다.
- 구현 난이도는 **중간** 수준으로 판단됩니다.

#### 제약 사항

- 계열사 간 결재 데이터의 가시성(Visibility) 및 권한(Authorization) 처리에 대한 명확한 정책 수립이 선행되어야 합니다.
- `form_entInfo.xml`의 `assignedDomain` 파라미터로 도메인 접근 권한이 제어되고 있으므로, V1↔V2 간 상호 도메인 접근 정책 설정이 필요합니다.
- V1, V2 간 조직도 데이터의 실시간 동기화 또는 배치 동기화 방식에 대한 결정이 필요합니다.

#### 대안 제시

- 기존 `EntInfoSvc` 및 `DomainCode` 체계를 활용하여 사간결재 표준 기능을 활성화하고, 계열사별 결재선/양식 분리 전략을 적용하는 것이 코어 수정 최소화 관점에서 가장 효율적입니다.
- 성능 이슈 발생 시 결재선 캐싱(Redis) 및 비동기 알림(`MessagingQueueManager`) 처리를 추가 적용할 수 있습니다.

### 3-5. API 및 서비스 인터페이스 영향도

소스 코드에서 확인된 API 인터페이스 영향 범위는 다음과 같습니다.

- **결재 API** (`coviflow.api.v1.web.ApprovalApiCon.java` / `ApprovalApiSvc.java`): 외부 시스템 연동 API에서 계열사 구분 파라미터 처리가 추가되어야 합니다.
- **결재 프로세스 API** (`ApvProcessCon.java`): `doProcess()`, `doCreateInstance()` 호출 시 사간결재 여부에 따른 분기 처리가 필요합니다.
- **조직 정보 API** (`EntInfoSvc.getEntinfoListData()`): 계열사(V1, V2) 각각의 인사 정보를 조회하는 기존 인터페이스를 활용할 수 있습니다.
- **통합 메시징 서비스**(`sendMessage.do`): 결재 알림 발송 시 계열사 구분이 반영되어야 합니다.

**산출 근거**: 소스코드(`ApprovalApiCon.java`, `ApvProcessSvc.java`, `EntInfoSvc.java`) 분석, 분석 요청서(200AI_C_010.REQ-WFW-001) API 영향도 항목, TECHNICAL_GUIDE_COMMON_SERVICES.md 통합 발송 API 참조

### 3-6. 데이터 정합성 및 트랜잭션 전략

- `ApvProcessSvc.doCreateInstance()` 메서드에서 결재 인스턴스 생성 시 트랜잭션이 관리되고 있으며, 사간결재 시에도 동일한 트랜잭션 경계 내에서 원자성(Atomicity)을 보장해야 합니다.
- `admin_affiliatesAggregateInformation.xml`에서 `jwf_process` 테이블의 `BusinessState`, `ProcessState`로 결재 상태를 관리하고 있으므로, 분산 환경에서 V1↔V2 간 결재 문서 이동 시 상태값의 일관성(Consistency) 유지를 위한 보상 트랜잭션(Compensating Transaction) 전략을 수립해야 합니다.
- 동시 결재 처리 시 낙관적 잠금(Optimistic Lock) 또는 비관적 잠금(Pessimistic Lock) 방식의 적용을 검토해야 합니다.

**산출 근거**: 소스코드(`ApvProcessSvc.java`, `admin_affiliatesAggregateInformation.xml`) 분석, 분석 요청서(200AI_C_010.REQ-WFW-001) 데이터 정합성 항목 기준
