## (3) 분석 계획 범위

### 3-1. 영향도 분석

#### 기존 모듈 영향도

소스 코드 분석 결과, 다국어(I18n) 기능 구현 시 다음 모듈에 직접적인 영향이 있습니다.

| 영향 모듈 | 소스 위치 | 영향 내용 |
|---|---|---|
| 다국어 사전 유틸 | `FormCon.java` → `DicHelper` import | `egovframework.coviframework.util.DicHelper`가 전자결재 모듈 전반에서 다국어 변환에 사용되고 있음. 베트남어(vi-VN) 사전 데이터 추가 필요 |
| 세션 언어 관리 | `FormCon.java` → `SessionHelper` import | `SessionHelper.getSession("lang")`으로 사용자 언어 코드를 세션에서 조회하는 구조. `AutoApprovalLineService.java`에서도 `SessionHelper.getSession("lang")` 필드로 언어 코드 사용 확인 |
| Redis 기반 사전 캐싱 | `FormCon.java` → `RedisDataUtil` import | `RedisDataUtil`을 통해 다국어 사전 데이터를 Redis에서 조회하는 구조가 이미 구현되어 있음 |
| 양식 관리자 | `coviflow.form.web.FormCon.java` / `FormSvc.java` | 양식 포맷(필드명, 안내문구)의 다국어 처리가 필요함 |
| XForm 에디터 | `coviflow.xform.web.XFormEditorCon.java` | 양식 편집기에서 다국어 레이블 적용 확인 필요 |
| 결재선 자동설정 | `coviflow.approvalline.service.AutoApprovalLineService.java` | `final String lang = SessionHelper.getSession("lang");` 코드로 세션 언어를 사용하여 결재선 관련 다국어 처리를 수행하고 있음 |
| 양식 관리 Admin | `coviflow.admin.web.AdminFormCon.java` | 관리자 화면의 메뉴명, 버튼명 등 UI 요소 다국어 대응 필요 |
| 계열사 정보 | `form_entInfo.xml` | `Fn_BaseGetDictionary_S(#{lang}, dn.MultiDisplayName)` SQL 함수로 이미 다국어 조회가 구현되어 있음. 베트남어 데이터 추가 등록 필요 |

#### DB 변경점

소스 코드의 SQL 매핑 파일 분석 결과를 기반으로 다음 DB 변경점을 확인하였습니다.

- `form_entInfo.xml`에서 `covi_smart4j.Fn_BaseGetDictionary_S(#{lang}, dn.MultiDisplayName)` DB 함수를 호출하여 다국어 변환을 수행하고 있습니다. 이 함수가 참조하는 `sys_base_dictionary` 테이블에 베트남어(vi-VN) 언어 코드 및 번역 데이터 등록이 필요합니다.
- `admin_adminForm.xml`의 양식 관련 쿼리에서도 `MultiDisplayName` 컬럼 기반 다국어 조회가 사용될 수 있으므로, 전자결재 모듈 전용 다국어 사전 데이터(양식명, 결재 상태명, 메뉴명 등) 추가 등록이 필요합니다.

#### 인프라 변경점

- TECHNICAL_GUIDE_COMMON_SERVICES.md에 명시된 바와 같이 시스템 구동 시 `sys_base_dictionary` 데이터를 Redis에 캐싱하고 있으므로, 베트남어 사전 데이터 추가 시 Redis Cache 용량 및 갱신 전략 검토가 필요합니다.
- 소스에서 `RedisDataUtil`이 `FormCon.java` 등 전자결재 모듈 전반에서 사용되고 있어, 다국어 사전 데이터 갱신 시 Redis 캐시 무효화(Invalidation) 전략을 수립해야 합니다.

**산출 근거**: 소스코드(`smarts4j.coviapproval/src`) DicHelper/SessionHelper/RedisDataUtil 사용 분석, `form_entInfo.xml` SQL 함수 분석, 요구사항정의서(GAP분석서) REQ-WFW-003 GAP대응방안, TECHNICAL_GUIDE_COMMON_SERVICES.md 다국어 및 설정(I18n & Meta) Redis 기반 사전 로딩 참조

### 3-2. 기술적 타당성

#### 구현 난이도

- 소스 분석 결과, `DicHelper`를 통한 다국어 변환 체계가 전자결재 모듈 전반에 이미 적용되어 있습니다(`FormCon.java`, `ApvProcessCon.java`, `AutoApprovalLineService.java` 등 다수 클래스에서 import 확인).
- `form_entInfo.xml`의 `Fn_BaseGetDictionary_S()` DB 함수가 `#{lang}` 파라미터를 받아 동적으로 다국어 변환을 수행하는 구조이므로, 베트남어 언어 팩 추가 및 사전 데이터 등록 방식으로 구현이 가능합니다.
- 구현 난이도는 **중간** 수준이나, 번역 데이터 확보 및 고객사 검수 기간이 일정에 영향을 미칩니다.

#### 제약 사항

- 요구사항정의서 원본에 "양식 포멧의 언어는 가능하나 본문내용의 언어는 변환 불가함"으로 명시되어 있으므로, 사용자가 작성한 본문 내용의 자동 번역은 불가합니다.
- 베트남어 번역 데이터의 정확성 검증을 위해 고객사(현지) 확인 기간이 필요합니다.
- `AutoApprovalLineService.java`에서 `SessionHelper.getSession("lang")` 값이 클래스 레벨 필드로 선언되어 있어, 세션 언어 변경 시 동적 반영 안정성을 확인해야 합니다.

#### 대안 제시

- 1단계: 기존 `DicHelper` / `Fn_BaseGetDictionary_S()` 체계에 베트남어 언어 코드 추가 및 시스템 공통 영역(메뉴, 버튼, 상태값) 다국어 적용
- 2단계: 전자결재 양식별 필드명/안내문구 다국어 적용 및 고객사 검수
- 본문 내용 번역이 향후 요구될 경우, 외부 번역 API 연동을 별도 요구사항으로 관리하는 것을 권장합니다.

**산출 근거**: 소스코드(`DicHelper` 사용 클래스 분석, `form_entInfo.xml` `Fn_BaseGetDictionary_S` 함수 분석), 분석 요청서(200AI_C_010.REQ-WFW-003) 기술적 타당성 항목, TECHNICAL_GUIDE_COMMON_SERVICES.md 다국어 및 설정 사용자 컨텍스트(LanguageCode) 참조

### 3-5. API 및 서비스 인터페이스 영향도

소스 코드에서 확인된 API 인터페이스 영향 범위는 다음과 같습니다.

- **다국어 사전 조회**: `DicHelper`를 통해 사용자 `LanguageCode` 기반으로 Redis에서 해당 언어 사전 데이터를 반환하는 내부 유틸리티가 전자결재 모듈 전반에서 사용되고 있습니다. 베트남어 데이터 추가 시 기존 인터페이스를 그대로 활용할 수 있습니다.
- **양식 관리자 API** (`XFormEditorCon.java`, `AdminFormCon.java`): 양식 필드명/안내문구의 다국어 데이터를 반환하는 인터페이스 확장이 필요합니다.
- **계열사 정보 API** (`EntInfoSvc`): `form_entInfo.xml`에서 `Fn_BaseGetDictionary_S(#{lang}, dn.MultiDisplayName)`으로 이미 다국어 조회가 구현되어 있어, 베트남어 사전 데이터 등록만으로 기존 인터페이스가 동작합니다.
- **통합 메시징 서비스** (`sendMessage.do`): 알림 발송 시 수신자의 언어 설정에 맞는 알림 내용 발송이 필요합니다.

**산출 근거**: 소스코드(`DicHelper` 사용 패턴, `form_entInfo.xml` 다국어 쿼리) 분석, 분석 요청서(200AI_C_010.REQ-WFW-003) API 영향도 항목, TECHNICAL_GUIDE_COMMON_SERVICES.md 통합 발송 API 및 다국어 설정 참조

### 3-6. 데이터 정합성 및 트랜잭션 전략

- TECHNICAL_GUIDE_COMMON_SERVICES.md에 명시된 바와 같이 `sys_base_dictionary` 데이터가 Redis에 캐싱되고 있으므로, 다국어 사전 데이터 등록/수정 시 DB와 Redis 캐시 간의 데이터 일관성을 보장해야 합니다.
- 소스에서 `RedisDataUtil`이 다수 클래스에서 사용되고 있어, 사전 데이터 갱신 시 캐시 무효화(Cache Invalidation) 후 재로딩하는 전략을 적용해야 합니다.
- 다국어 데이터가 누락된 경우 기본 언어(한국어 또는 영어)로 폴백(Fallback)하는 처리 로직이 필요합니다. `Fn_BaseGetDictionary_S()` 함수의 폴백 동작을 확인해야 합니다.

**산출 근거**: 소스코드(`RedisDataUtil` 사용 분석, `Fn_BaseGetDictionary_S` 함수 동작), 분석 요청서(200AI_C_010.REQ-WFW-003) 데이터 정합성 항목, TECHNICAL_GUIDE_COMMON_SERVICES.md Redis 기반 사전 로딩 메커니즘 참조
